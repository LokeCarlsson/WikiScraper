Generally this refers to either <a href="/wiki/Machine_code" title="Machine code">machine code</a> or <a href="/wiki/Assembly_language" title="Assembly language">assembly language</a>. The word "low" refers to the small or nonexistent amount of <a href="/wiki/Abstraction_(computer_science)" class="mw-redirect" title="Abstraction (computer science)">abstraction</a> between the language and machine language; because of this, low-level languages are sometimes described as being "close to the hardware". Programs written in low-level languages tend to be relatively <a href="/wiki/Software_portability" title="Software portability">non-portable</a>, mainly because of the close relationship between the language and the specific hardware architecture.</p>/n<p>Low-level languages can convert to machine code without a compiler or interpreter— <a href="/wiki/Second-generation_programming_language" title="Second-generation programming language">second-generation programming languages</a> use a simpler processor called an <a href="/wiki/Assembly_language#Assemble" title="Assembly language">assembler</a>— and the resulting code runs directly on the processor. A program written in a low-level language can be made to run very quickly, with a small <a href="/wiki/Memory_footprint" title="Memory footprint">memory footprint</a>. An equivalent program in a <a href="/wiki/High-level_language" class="mw-redirect" title="High-level language">high-level language</a> can be less efficient and use more memory. Low-level languages are simple, but considered difficult to use, due to numerous technical details that the programmer must remember. By comparison, a <a href="/wiki/High-level_programming_language" title="High-level programming language">high-level programming language</a> isolates execution semantics of a computer architecture from the specification of the program, which simplifies development.</p>/n<p>Low-level programming languages are sometimes divided into two categories: <i>first generation</i> and <i>second generation</i>.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. The row of switches at the bottom can be used to toggle in a machine language program</div>/n</div>/n</div>/n<p><a href="/wiki/Machine_code" title="Machine code">Machine code</a> is the only language a computer can process directly without a previous transformation. Currently, programmers almost never write programs directly in machine code, because it requires attention to numerous details that a high-level language handles automatically. Furthermore it requires memorizing or looking up numerical codes for every instruction, and is extremely difficult to modify.</p>/n<p>True <i>machine code</i> is a stream of raw, usually <a href="/wiki/Binary_code" title="Binary code">binary</a>, data. A programmer coding in "machine code" normally codes instructions and data in a more readable form such as <a href="/wiki/Decimal" title="Decimal">decimal</a>, <a href="/wiki/Octal" title="Octal">octal</a>, or <a href="/wiki/Hexadecimal" title="Hexadecimal">hexadecimal</a> which is translated to internal format by a program called a <a href="/wiki/Loader_(computing)" title="Loader (computing)">loader</a> or toggled into the computer's memory from a <a href="/wiki/Front_panel" title="Front panel">front panel</a>.</p>/n<p>Although few programs are written in machine language, programmers often become adept at reading it through working with <a href="/wiki/Core_dump" title="Core dump">core dumps</a> or debugging from the front panel.</p>/n<p>Example: A function in hexadecimal representation of 32-bit <a href="/wiki/X86" title="X86">x86</a> machine code to calculate the <i>n</i>th <a href="/wiki/Fibonacci_number" title="Fibonacci number">Fibonacci number</a>:</p>/n<pre>/n8B542408 83FA0077 06B80000 0000C383/nFA027706 B8010000 00C353BB 01000000/nB9010000 008D0419 83FA0376 078BD989/nC14AEBF1 5BC3/n</pre>/n<h2><span class="mw-headline" id="Assembly">Assembly</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Low-level_programming_language&amp;action=edit&amp;section=2" title="Edit section: Assembly">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Second-generation languages provide one abstraction level on top of the machine code. In the early days of coding on computers like the <a href="/wiki/TX-0" title="TX-0">TX-0</a> and <a href="/wiki/PDP-1" title="PDP-1">PDP-1</a>, the first thing MIT hackers did was write assemblers.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> <a href="/wiki/Assembly_language" title="Assembly language">Assembly language</a> has little <a href="/wiki/Semantics_(computer_science)" title="Semantics (computer science)">semantics</a> or formal specification, being only a mapping of human-readable symbols, including symbolic addresses, to <a href="/wiki/Opcode" title="Opcode">opcodes</a>, <a href="/wiki/Memory_address" title="Memory address">addresses</a>, numeric constants, <a href="/wiki/String_(computer_science)" title="String (computer science)">strings</a> and so on. Typically, one <a href="/wiki/Machine_instruction_(computing)" class="mw-redirect" title="Machine instruction (computing)">machine instruction</a> is represented as one line of assembly code. Assemblers produce <a href="/wiki/Object_file" title="Object file">object files</a> that can <a href="/wiki/Linker_(computing)" title="Linker (computing)">link</a> with other object files or be <a href="/wiki/Loader_(computing)" title="Loader (computing)">loaded</a> on their own.</p>/n<p>Most assemblers provide <a href="/wiki/Macro_(computer_science)" title="Macro (computer science)">macros</a> to generate common sequences of instructions.</p>/n<p>Example: The same <a href="/wiki/Fibonacci_number" title="Fibonacci number">Fibonacci number</a> calculator as above, but in x86 assembly language href="/wiki/Processor_register" title="Processor register">registers</a>) are named and manipulated directly. The function loads its input from a precise location in the <a href="/wiki/Call_stack" title="Call stack">stack</a> (8 bytes higher than the location stored in the <b>ESP</b> stack pointer) and performs its calculation by manipulating values in the <b>EAX</b>, <b>EBX</b>, <b>ECX</b> and <b>EDX</b> registers until it has finished and returns. Note that in this assembly language, there is no concept of returning a value. The result having been stored in the <b>EAX</b> register, the <b>RET</b> command simply moves code processing to the code location stored on the stack (usually the instruction immediately after the one that called this function) and it is up to the author of the calling code to know that this function stores its result in <b>EAX</b> and to retrieve it from there. x86 assembly language imposes no standard for returning values from a function (and so, in fact, has no concept of a function); it is up to the calling code to examine state after the procedure returns if it needs to extract a value.</p>/n<p>Compare this with the same function in C:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="kt">unsigned</span> <span class="kt">int</span> <span class="p">;</span>/n        <span class="n">a</span> <span class="o">=</span> <span class="mi">1</span><span class="p">;</span>/n        <span class="n">b</span> <span class="o">=</span> <span class="mi">1</span><span class="p">;</span>/n        <span class="k">while</span> <span class="p">(</span><span class="mi">1</span><span class="p">)</span> <span class="p">{</span>/n            <span class="n">c</span> <span class="o">=</span> <span class="n">a</span> <span class="o">+</span> <span class="n">b</span><span class="p">;</span>/n            <span class="k">if</span> <span class="p">(</span><span class="n">n</span> <span class="o">&lt;=</span> <span class="mi">3</span><span class="p">)</span> <span class="k">return</span> <span class="n">c</span><span class="p">;</span>/n            <span class="n">a</span> <span class="o">=</span> <span class="n">b</span><span class="p">;</span>/n            <span class="n">b</span> <span class="o">=</span> <span class="n">c</span><span class="p">;</span>/n            <span class="n">n</span><span class="o">--</span><span class="p">;</span>/n        <span class="p">}</span>/n    <span class="p">}</span>/n<span class="p">}</span>/n</pre></div>/n<p>This code is very similar in structure to the assembly language example but there are significant differences in terms of abstraction:</p>/n<ul>/n<li>While the input (parameter <b>n</b>) is loaded from the stack, its precise position on the stack is not specified. The C compiler calculates this based on the calling conventions of the target architecture.</li>/n<li>The assembly language version loads the input parameter from the stack into a register and in each iteration of the loop decrements the value in the register, never altering the value in the memory location on the stack. The C compiler could do the same or could update the value in the stack. Which one it chooses is an implementation decision completely hidden from the code author (and one with no side effects, thanks to C language standards).</li>/n<li>The local variables a, b and c are abstractions that do not specify any specific storage location on the hardware. The C compiler decides how to actually store them for the target architecture.</li>/n<li>The return function specifies the value to return, but does not dictate <i>how</i> it is returned. The C compiler for any specific architecture implements a <b>standard</b> mechanism for returning the value. Compilers for the x86 architecture typically (but not always) use the EAX register to return a value, as in the assembly language example (the author of the assembly language example has <i>chosen</i> to copy the C convention but assembly language does not require this).</li>/n</ul>/n<p>These abstractions make the C code compilable without modification on any architecture for which a C compiler has been written. The x86 assembly language code is specific to the x86 architecture.</p>/n<h2><span class="mw-headline" id="Low-level_programming_in_high-level_languages">Low-level programming in high-level languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Low-level_programming_language&amp;action=edit&amp;section=3" title="Edit section: Low-level programming in high-level languages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>In the late 1960s, high-level languages such as <a href="/wiki/IBM_PL/S" title="IBM PL/S">PL/S</a>, <a href="/wiki/BLISS" title="BLISS">BLISS</a>, <a href="/wiki/BCPL" title="BCPL">BCPL</a>, extended <a href="/wiki/ALGOL" title="ALGOL">ALGOL</a> (for <a href="/wiki/Burroughs_large_systems" title="Burroughs large systems">Burroughs large systems</a>) and <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> included some degree of access to low-level programming functions. One method for this is <a href="/wiki/Inline_assembly" class="mw-redirect" title="Inline assembly">Inline assembly</a>, in which assembly code is embedded in a high-level language that supports this feature. Some of these languages also allow architecture-dependent <a href="/wiki/Optimizing_compiler" title="Optimizing compiler">compiler optimization directives</a> to adjust the way a compiler uses the target processor architecture.</p>/n<h2><span class="mw-headline" id="References">References</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a
