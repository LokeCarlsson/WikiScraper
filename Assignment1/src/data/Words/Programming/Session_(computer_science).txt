Unsourced material may be challenged and removed.</span> <small><i>(July 2014)</i></small> <small class="hide-when-compact"><i>(<a href="/wiki/Help:Maintenance_template_removal" title="Help:Maintenance template removal">Learn how and when to remove this template message</a>)</i></small></div>/n</td>/n</tr>/n</table>/n<p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, in particular <a href="/wiki/Computer_network" title="Computer network">networking</a>, a <b>session</b> is a semi-permanent interactive information interchange, also known as a dialogue, a conversation or a meeting, between two or more communicating devices, or between a computer and user (see <a href="/wiki/Login_session" title="Login session">Login session</a>). A session is set up or established at a certain point in time, and then torn down at some later point. An established communication session may involve more than one message in each direction. A session is typically, but not always, <a href="/wiki/Stateful" class="mw-redirect" title="Stateful">stateful</a>, meaning that at least one of the communicating parts needs to save information about the session history in order to be able to communicate, as opposed to <a href="/wiki/Stateless_server" class="mw-redirect" title="Stateless server">stateless</a> communication, where the communication consists of independent requests with responses.</p>/n<p>An established session is the basic requirement to perform a <a href="/wiki/Connection-oriented_communication" title="Connection-oriented communication">connection-oriented communication</a>. A session also is the basic step to transmit in <a href="/wiki/Connectionless_communication" title="Connectionless communication">connectionless communication</a> modes. However any unidirectional transmission does not define a session.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup></p>/n<p><b>Communication Transport</b> may be implemented as part of protocols and services at the <a href="/wiki/Application_layer" title="Application layer">application layer</a>, at the <a href="/wiki/Session_layer" title="Session layer">session layer</a> or at the <a href="/wiki/Transport_layer" title="Transport layer">transport layer</a> in the <a href="/wiki/OSI_model" title="OSI model">OSI model</a>.</p>/n<ul>/n<li>Application layer examples:/n<ul>/n<li><a href="/wiki/HTTP#HTTP_session" class="mw-redirect" title="HTTP">HTTP sessions</a>, which allow associating information with individual visitors</li>/n<li>A <a href="/wiki/Telnet" title="Telnet">telnet</a> remote login session</li>/n</ul>/n</li>/n<li>Session layer example:/n<ul>/n<li>A <a href="/wiki/Session_Initiation_Protocol" title="Session Initiation Protocol">Session Initiation Protocol</a> (SIP) based <a href="/wiki/Internet_phone" class="mw-redirect" title="Internet phone">Internet phone</a> call</li>/n</ul>/n</li>/n<li>Transport layer example:/n<ul>/n<li>A <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">TCP</a> session, which is synonymous to a TCP <a href="/wiki/Virtual_circuit" title="Virtual circuit">virtual circuit</a>, a TCP connection, or an established TCP <a href="/wiki/Internet_socket" class="mw-redirect" title="Internet socket">socket</a>.</li>/n</ul>/n</li>/n</ul>/n<p>In the case of transport protocols that do not implement a formal session layer (e.g., <a href="/wiki/User_Datagram_Protocol" title="User Datagram Protocol">UDP</a>) or where sessions at the application layer are generally very short-lived (e.g., HTTP), sessions are maintained by a higher level program using a method defined in the data being exchanged. For example, an HTTP exchange between a browser and a remote host may include an <a href="/wiki/HTTP_cookie" title="HTTP cookie">HTTP cookie</a> which identifies state, such as a unique <a href="/wiki/Session_ID" title="Session ID">session ID</a>, information about the user's preferences or authorization level.</p>/n<p><a href="/wiki/HTTP/1.0" class="mw-redirect" title="HTTP/1.0">HTTP/1.0</a> was thought to only allow a single request and response during one Web/HTTP Session. Protocol version <a href="/wiki/HTTP/1.1" class="mw-redirect" title="HTTP/1.1">HTTP/1.1</a> improved this by completing the <a href="/wiki/Common_Gateway_Interface" title="Common Gateway Interface">Common Gateway Interface</a> (CGI), making it easier to maintain the Web Session and supporting <a href="/wiki/HTTP_cookie" title="HTTP cookie">HTTP cookies</a> and file uploads.</p>/n<p>Most client-server sessions are maintained by the transport layer - a single connection for a single session. However each transaction phase of a Web/HTTP session creates a separate connection. Maintaining session continuity between phases requires a <a href="/wiki/Session_ID" title="Session ID">session ID</a>. The <a href="/wiki/Session_ID" title="Session ID">session ID</a> is embedded within the &lt;A HREF&gt; or &lt;FORM&gt; links of <a href="/wiki/Dynamic_web_page" title="Dynamic web page">dynamic web pages</a> so that it is passed back to the CGI. CGI then uses the <a href="/wiki/Session_ID" title="Session ID">session ID</a> to ensure session continuity between transaction phases. One advantage of one connection-per-phase is that it works well over low bandwidth (modem) connections.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toclevel-1 tocsection-12"><a href="#External_links"><span class="tocnumber">8</span> <span class="toctext">External links</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="Software_implementation">Software implementation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Session_(computer_science)&amp;action=edit&amp;section=1" title="Edit section: Software implementation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>TCP sessions are typically implemented in software using <a href="/wiki/Child_process" title="Child process">child processes</a> and/or <a href="/wiki/Thread_(computer_science)" class="mw-redirect" title="Thread (computer science)">multithreading</a>, where a new process or thread is created when the computer establishes or joins a session. HTTP sessions are typically not implemented using one thread per session, but by means of a database with information about the state of each session. The advantage with multiple processes or threads is relaxed complexity of the software, since each thread is an <a href="/wiki/Object_(computer_science)" title="Object (computer science)">instance</a> with its own history and encapsulated variables. The disadvantage is large overhead in terms of system resources, and that the session may be interrupted if the system is restarted.</p>/n<p>When a client may connect to any server in a cluster of servers, a special problem is encountered in maintaining consistency when the servers must maintain session state. The client must either be directed to the same server for the duration of the session, or the servers must transmit server-side session information via a shared file system or database. Otherwise, the client may reconnect to a different server than the one it started the session with, which will cause problems when the new server does not have access to the stored state of the old one.</p>/n<h2><span class="mw-headline" id="Server-side_web_sessions">Server-side web sessions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Session_(computer_science)&amp;action=edit&amp;section=2" title="Edit section: Server-side web sessions">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Server-side sessions are handy and efficient, but can become difficult to handle in conjunction with load-balancing/high-availability systems and are not usable at all in some embedded systems with no storage. The load-balancing problem can be solved by using shared storage or by applying forced peering between each client and a single server in the cluster, although this can compromise system efficiency and load distribution.</p>/n<p>A method of using server-side sessions in systems without mass-storage is to reserve a portion of RAM for storage of session data. This method is applicable for servers with a limited number of clients (e.g. router or access point with infrequent or disallowed access to more than one client at a time).</p>/n<h2><span class="mw-headline" id="Client-side_web_sessions">Client-side web sessions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Session_(computer_science)&amp;action=edit&amp;section=3" title="Edit section: Client-side web sessions">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Client-side sessions use cookies and cryptographic techniques to maintain state without storing as much data on the server. When presenting a dynamic web page, the server sends the current state data to the client (web browser) in the form of a cookie. The client saves the cookie in memory or on disk. With each successive request, the client sends the cookie back to the server, and the server uses the data to "remember" the state of the application for that specific client and generate an appropriate response.</p>/n<p>This mechanism may work well in some contexts; however, data stored on the client is vulnerable to tampering by the user or by software that has access to the client computer. To use client-side sessions where confidentiality and integrity are required, the following must be guaranteed:</p>/n<ol>/n<li>Confidentiality: Nothing apart from the server should be able to interpret session data.</li>/n<li>Data integrity: Nothing apart from the server should manipulate session data (accidentally or maliciously).</li>/n<li>Authenticity: Nothing apart from the server should be able to initiate valid sessions.</li>/n</ol>/n<p>To accomplish this, the server needs to encrypt the session data before sending it to the client, and modification of such information by any other party should be prevented via cryptographic means.</p>/n<p>Transmitting state back and forth with every request is only practical when the size of the cookie is small. In essence, client-side sessions trade server disk space for the extra bandwidth that each web request will require. Moreover, web browsers limit the number and size of cookies that may be stored by a web site. To improve efficiency and allow for more session data, the server may compress the data before creating the cookie, decompressing it later when the cookie is returned by the client.</p>/n<h2><span class="mw-headline" id="HTTP_session_token">HTTP session token</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Session_(computer_science)&amp;action=edit&amp;section=4" title="Edit section: HTTP session token">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>A session token is a unique identifier that is generated and sent from a <a href="/wiki/Server_(computing)" title="Server (computing)">server</a> to a <a href="/wiki/Client_(computing)" title="Client (computing)">client</a> to identify the current interaction session. The client usually stores and sends the token as an <a href="/wiki/HTTP_cookie" title="HTTP cookie">HTTP cookie</a> and/or sends it as a parameter in GET or POST queries. The reason to use session tokens is that the client only has to handle the identifier—all session data is stored on the server (usually in a <a href="/wiki/Database" title="Database">database</a>, to which the client does not have direct access) linked to that identifier. Examples of the names that some programming languages use when naming their HTTP cookie include JSESSIONID (<a href="/wiki/JavaServer_Pages" title="JavaServer Pages">JSP</a>), PHPSESSID (<a href="/wiki/PHP" title="PHP">PHP</a>), CGISESSID (<a href="/wiki/Common_Gateway_Interface" title="Common Gateway Interface">CGI</a>), and ASPSESSIONID (<a href="/wiki/Active_Server_Pages" title="Active Server Pages">ASP</a>).</p>/n<h2><span class="mw-headline" id="Session_management">Session management</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Session_(computer_science)&amp;action=edit&amp;section=5" title="Edit section: Session management">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>In <a href="/wiki/Human%E2%80%93computer_interaction" title="Human–computer interaction">human–computer interaction</a>, <b>session management</b> is the process of keeping track of a user's activity across sessions of interaction with the <a href="/wiki/Computer_system" class="mw-redirect" title="Computer system">computer system</a>.</p>/n<p>Typical session management tasks in a <a href="/wiki/Desktop_environment" title="Desktop environment">desktop environment</a> include keeping track of which applications are open and which documents each application has opened, so that the same state can be restored when the user logs out and logs in later. For a website, session management might involve requiring the user to re-login if the session has expired (i.e., a certain time limit has passed without user activity). It is also used to store information on the server-side between HTTP requests.</p>/n<h3><span class="mw-headline" id="Desktop_session_management">Desktop session management</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Session_(computer_science)&amp;action=edit&amp;section=6" title="Edit section: Desktop session management">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>A desktop session manager is a program that can save and restore desktop sessions. A desktop session is all the windows currently running and their current content. Session management on <a href="/wiki/Linux" title="Linux">Linux</a>-based systems is provided by <a href="/wiki/X_session_manager" title="X session manager">X session manager</a>. On <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> systems, session management is provided by the Session Manager Subsystem (smss.exe); user session functionality can be extended by third-party applications like <a href="/w/index.php?title=Twinsplay&amp;action=edit&amp;redlink=1" class="new" title="Twinsplay (page does not exist)">twinsplay</a>.</p>/n<h3><span class="mw-headline" id="Browser_session_management">Browser session management</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Session_(computer_science)&amp;action=edit&amp;section=7" title="Edit section: Browser session management">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Session management is particularly useful in a <a href="/wiki/Web_browser" title="Web browser">web browser</a> where a user can save all open pages and settings and restore them at a later date. To help recover from a system or application crash, pages and settings can also be restored on next run. <a href="/wiki/Google_Chrome" title="Google Chrome">Google Chrome</a>, <a href="/wiki/Mozilla_Firefox" class="mw-redirect" title="Mozilla Firefox">Mozilla Firefox</a>, <a href="/wiki/Internet_Explorer" title="Internet Explorer">Internet Explorer</a>, <a href="/wiki/OmniWeb" title="OmniWeb">OmniWeb</a> and <a href="/wiki/Opera_(web_browser)" title="Opera (web browser)">Opera</a> are examples of web browsers that support session management. Session management is often managed through the application of <a href="/wiki/HTTP_cookie" title="HTTP cookie">cookies</a>.</p>/n<h3><span class="mw-headline" id="Web_server_session_management">Web server session management</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Session_(computer_science)&amp;action=edit&amp;section=8" title="Edit section: Web server session management">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p><a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">Hypertext Transfer Protocol</a> (HTTP) is stateless: a client computer running a web browser must establish a new <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">Transmission Control Protocol</a> (TCP) network connection to the web server with each new HTTP GET or POST request. The web server, therefore, cannot rely on an established TCP network connection for longer than a single HTTP GET or POST operation. Session management is the technique used by the web developer to make the stateless HTTP protocol support session state. For example, once a user has been authenticated to the web server, the user's next HTTP request (GET or POST) should not cause the web server to ask for the user's account and password again. For a discussion of the methods used to accomplish this see <a href="/wiki/HTTP_cookie" title="HTTP cookie">HTTP cookie</a> and <a href="/wiki/Session_ID" title="Session ID">Session ID</a></p>/n<p>In situations where multiple web servers must share knowledge of session state (as is typical in a <a href="/wiki/Computer_cluster" title="Computer cluster">cluster</a> environment) session information must be shared between the cluster nodes that are running web server software. Methods for sharing session state between nodes in a cluster include: multicasting session information to member nodes (see <a href="/wiki/JGroups" title="JGroups">JGroups</a> for one example of this technique), sharing session information with a partner node using <a href="/wiki/Distributed_shared_memory" title="Distributed shared memory">distributed shared memory</a> or <a href="/wiki/Memory_virtualization" title="Memory virtualization">memory virtualization</a>, sharing session information between nodes using network sockets, storing session information on a shared file system such as a <a href="/wiki/Distributed_file_system" class="mw-redirect" title="Distributed file system">distributed file system</a> or a <a href="/wiki/Global_file_system" title="Global file system">global file system</a>, or storing the session information outside the cluster in a <a href="/wiki/Database" title="Database">database</a>.</p>/n<p>If session information is considered transient, volatile data that is not required for <a href="/wiki/Non-repudiation" title="Non-repudiation">non-repudiation</a> of transactions and does not contain data that is subject to compliance auditing (in the U.S. for example, see the <a href="/wiki/Health_Insurance_Portability_and_Accountability_Act" title="Health Insurance Portability and Accountability Act">Health Insurance Portability and Accountability Act</a> and the <a href="/wiki/Sarbanes-Oxley_Act" class="mw-redirect" title="Sarbanes-Oxley Act">Sarbanes-Oxley Act</a> for examples of two laws that necessitate compliance auditing) then any method of storing session information can be used. However, if session information is subject to audit compliance, consideration should be given to the method used for session storage, replication, and clustering.</p>/n<p>In a <a href="/wiki/Service-oriented_architecture" title="Service-oriented architecture">service-oriented architecture</a>, Simple Object Access Protocol or <a href="/wiki/SOAP" title="SOAP">SOAP</a> messages constructed with Extensible Markup Language (<a href="/wiki/XML" title="XML">XML</a>) messages can be used by consumer applications to cause web servers to create sessions.</p>/n<h3><span class="mw-headline" id="Session_Management_over_SMS">Session Management over SMS</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Session_(computer_science)&amp;action=edit&amp;section=9" title="Edit section: Session Management over SMS">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Just as HTTP is a stateless protocol, so is <a href="/wiki/SMS" title="SMS">SMS</a>. As SMS became interoperable across rival networks in 1999,<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> and text messaging started its ascent towards becoming a ubiquitous global form of communication,<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup> various enterprises became interested in using the SMS channel for commercial purposes. Initial services did not require session management since they were only one-way communications (for example, in 2000, the <a href="/wiki/Mobile_phone#Features" title="Mobile phone">first mobile news service was delivered via SMS in Finland</a>). Today, these applications are referred to as <a href="/wiki/Short_message_service#Application-to-Person_.28A2P.29_SMS" class="mw-redirect" title="Short message service">application-to-peer (A2P) messaging</a> as distinct from <a href="/wiki/Short_Message_Peer-to-Peer" title="Short Message Peer-to-Peer">peer-to-peer (P2P)</a> messaging. The development of interactive enterprise applications required session management, but because SMS is a stateless protocol as defined by the GSM standards,<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> early implementations were controlled <a href="/wiki/Client-side" title="Client-side">client-side</a> by having the end-users enter commands and service identifiers manually.</p>/n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a
