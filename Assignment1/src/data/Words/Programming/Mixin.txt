For the ice cream, see <a href="/wiki/Mix-in" title="Mix-in">Mix-in</a>.</div>/n<p>In <a href="/wiki/Object-oriented_programming_language" class="mw-redirect" title="Object-oriented programming language">object-oriented programming languages</a>, a <b>Mixin</b> is a <a href="/wiki/Class_(computer_science)" class="mw-redirect" title="Class (computer science)">class</a> that contains methods for use by other classes without having to be the parent class of those other classes. How those other classes gain access to the mixin's methods depends on the language. Mixins are sometimes described as being "included" rather than "inherited".</p>/n<p>Mixins encourage <a href="/wiki/Code_reuse" title="Code reuse">code reuse</a> and can be used to avoid the inheritance ambiguity that multiple inheritance can cause<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> (the "<a href="/wiki/Multiple_inheritance#The_diamond_problem" title="Multiple inheritance">diamond problem</a>"), or to work around lack of support for multiple inheritance in a language. A mixin can also be viewed as an <a href="/wiki/Interface_(object-oriented_programming)" class="mw-redirect" title="Interface (object-oriented programming)">interface</a> with implemented <a href="/wiki/Method_(computer_science)" class="mw-redirect" title="Method (computer science)">methods</a>. This pattern is an example of enforcing the <a href="/wiki/Dependency_inversion_principle" title="Dependency inversion principle">dependency inversion principle</a>.</p>/n<p></p>/n<div id="toc" links</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Mixin&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Mixins first appeared in the <a href="/wiki/Symbolics" title="Symbolics">Symbolics</a>'s object-oriented <a href="/wiki/Flavors_(computer_science)" class="mw-redirect" title="Flavors (computer science)">Flavors</a> system (developed by Howard Cannon), which was an approach to object-orientation used in <a href="/wiki/Lisp_Machine_Lisp" title="Lisp Machine Lisp">Lisp Machine Lisp</a>. The name was inspired by <a href="/wiki/Steve%27s_Ice_Cream" title="Steve's Ice Cream">Steve's Ice Cream Parlor</a> in Somerville, Massachusetts:<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> The owner of the ice cream shop offered a basic flavor of ice cream (vanilla, chocolate, etc.) and blended in a combination of extra items (nuts, cookies, fudge, etc.) and called the item a "<a href="/wiki/Mix-in" title="Mix-in">mix-in</a>", his own trademarked term at the time.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup></p>/n<h2><span class="mw-headline" id="Definition">Definition</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Mixin&amp;action=edit&amp;section=2" title="Edit section: Definition">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Mixins are a language concept that allows a programmer to inject some code into a <a href="/wiki/Class_(computer_programming)" title="Class (computer programming)">class</a>. Mixin programming is a style of <a href="/wiki/Software_development" title="Software development">software development</a>, in which units of functionality are created in a class and then mixed in with other classes.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup></p>/n<p>A mixin class acts as the parent class, containing the desired functionality. A <a href="/wiki/Subclass_(computer_science)" class="mw-redirect" title="Subclass (computer science)">subclass</a> can then inherit or simply reuse this functionality, but not as a means of specialization. Typically, the mixin will export the desired functionality to a <a href="/wiki/Subclass_(computer_science)" class="mw-redirect" title="Subclass (computer science)">child class</a>, without creating a rigid, single "is a" relationship. Here lies the important difference between the concepts of mixins and <a href="/wiki/Inheritance_(object-oriented_programming)" title="Inheritance (object-oriented programming)">inheritance</a>, in that the child class can still inherit all the features of the parent class, but, the semantics about the child "being a kind of" the parent need not be necessarily applied.</p>/n<h2><span class="mw-headline" id="Advantages">Advantages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Mixin&amp;action=edit&amp;section=3" title="Edit section: Advantages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<ol>/n<li>It provides a mechanism for <a href="/wiki/Multiple_inheritance" title="Multiple inheritance">multiple inheritance</a> by allowing multiple classes to use the common functionality, but without the complex semantics of multiple inheritance.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup></li>/n<li><a href="/wiki/Code_reuse" title="Code reuse">Code reusability</a>: Mixins are useful when a programmer wants to share functionality between different classes. Instead of repeating the same code over and over again, the common functionality can simply be grouped into a mixin and then included into each class that requires it.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup></li>/n<li>Mixins allow inheritance and use of only the desired features from the parent class, not necessarily all of the features from the parent class.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup></li>/n</ol>/n<h2><span class="mw-headline" id="Implementations">Implementations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Mixin&amp;action=edit&amp;section=4" title="Edit section: Implementations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>In <a href="/wiki/Simula" title="Simula">Simula</a>, classes are defined in a block in which attributes, methods and class initialization are all defined together; thus all the methods that can be invoked on a class are defined together, and the definition of the class is complete.</p>/n<p>In <a href="/wiki/Flavors_(programming_language)" title="Flavors (programming language)">Flavors</a>, a Mixin is a class from which another class can inherit slot definitions and methods. The Mixin usually does not have direct instances. Since a Flavor can inherit from more than one other Flavor, it can inherit from one or more Mixins. Note that the original Flavors did not use generic functions.</p>/n<p>In New Flavors (a successor of Flavors) and <a href="/wiki/CLOS" class="mw-redirect" title="CLOS">CLOS</a>, methods are organized in "<a href="/wiki/Generic_function" title="Generic function">generic functions</a>". These generic functions are functions that are defined in multiple cases (methods) by class dispatch and method combinations.</p>/n<p>CLOS and Flavors allow mixin methods to add behavior to existing methods: <code>:before</code> and <code>:after</code> daemons, whoppers and wrappers in Flavors. CLOS added <code>:around</code> methods and the ability to call shadowed methods via <code class="mw-highlight" id="" style="" dir="ltr">CALL-NEXT-METHOD</code>. So, for example, a stream-lock-mixin can add locking around existing methods of a stream class. In Flavors one would write a wrapper or a whopper and in CLOS one would use an <code>:around</code> method. Both CLOS and Flavors allow the computed reuse via method combinations. <code>:before</code>, <code>:after</code> and <code>:around</code> methods are a feature of the standard method combination. Other method combinations are provided.</p>/n<p>An example is the <code>+</code> method combination, where the resulting values of each of the applicable methods of a generic function are arithmetically added to compute the return value. This is used, for example, with the border-mixin for graphical objects. A graphical object may have a generic width function. The border-mixin would add a border around an object and has a method computing its width. A new class <code>bordered-button</code> (that is both a graphical object and uses the <code>border</code> mixin) would compute its width by calling all applicable width methods—via the <code>+</code> method combination. All return values are added and create the combined width of the object.</p>/n<p>In an OOPSLA 90 paper,<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> Gilad Bracha and William Cook reinterpret different inheritance mechanisms found in Smalltalk, Beta and CLOS as special forms of a mixin inheritance.</p>/n<h2><span class="mw-headline"
