id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="mw-parser-output"><p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, <b>formal specifications</b> are mathematically based techniques whose purpose are to help with the implementation of systems and software. They are used to describe a system, to analyze its behavior, and to aid in its design by verifying key properties of interest through rigorous and effective reasoning tools.<sup id="cite_ref-a9-hierons_1-0" class="reference"><a href="#cite_note-a9-hierons-1">[1]</a></sup><sup id="cite_ref-p223-gaudel_2-0" class="reference"><a href="#cite_note-p223-gaudel-2">[2]</a></sup> These specifications are <i>formal</i> in the sense that they have a syntax, their semantics fall within one domain, and they are able to be used to infer useful information.<sup id="cite_ref-lamsweerde-roadmap_3-0" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup></p>/n<p></p>/n<div id="toc" class="toc">/n<div href="/w/index.php?title=Formal_specification&amp;action=edit&amp;section=1" title="Edit section: Motivation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>In each passing decade computer systems have become increasingly more powerful and as a result they have become more impactful to society. Because of this, better techniques are needed to assist in the design and implementation of reliable software. Established engineering disciplines use mathematical analysis as the foundation of creating and validating product design. Formal specifications are one such way to achieve this in software engineering reliability as once predicted. Other methods such as <a href="/wiki/Software_testing" title="Software testing">testing</a> are more commonly used to enhance code quality.<sup id="cite_ref-a9-hierons_1-1" class="reference"><a href="#cite_note-a9-hierons-1">[1]</a></sup></p>/n<h2><span class="mw-headline" id="Uses">Uses</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Formal_specification&amp;action=edit&amp;section=2" title="Edit section: Uses">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Given such a <a href="/wiki/Specification" class="mw-redirect" title="Specification">specification</a>, it is possible to use <a href="/wiki/Formal_verification" title="Formal verification">formal verification</a> techniques to demonstrate that a system design is correct with respect to its specification. This allows incorrect system designs to be revised before any major investments have been made into an actual implementation. Another approach is to use probably correct <a href="/wiki/Program_refinement" class="mw-redirect" title="Program refinement">refinement</a> steps to transform a specification into a design, which is ultimately transformed into an implementation that is <i>correct by construction</i>.</p>/n<p>It is important to note that a formal specification is <i>not</i> an implementation, but rather it may be used to develop an <a href="/wiki/Implementation" title="Implementation">implementation</a>. Formal specifications describe <i>what</i> a system should do, not <i>how</i> the system should do it.</p>/n<p>A good specification must have some of the following attributes: adequate, internally consistent, unambiguous, complete, satisfied, minimal<sup id="cite_ref-lamsweerde-roadmap_3-1" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup></p>/n<p>A good specification will have:<sup id="cite_ref-lamsweerde-roadmap_3-2" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup></p>/n<ul>/n<li>Constructability, manageability and evolvability</li>/n<li>Usability</li>/n<li>Communicability</li>/n<li>Powerful and efficient analysis</li>/n</ul>/n<p>One of the main reasons there is interest in formal specifications is that they will provide an ability to perform proofs on software implementations.<sup id="cite_ref-p223-gaudel_2-1" class="reference"><a href="#cite_note-p223-gaudel-2">[2]</a></sup> These proofs may be used to validate a specification, verify correctness of design, or to prove that a program satisfies a specification.<sup id="cite_ref-p223-gaudel_2-2" class="reference"><a href="#cite_note-p223-gaudel-2">[2]</a></sup></p>/n<h2><span class="mw-headline" id="Limitations">Limitations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Formal_specification&amp;action=edit&amp;section=3" title="Edit section: Limitations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>A design (or implementation) cannot ever be declared “correct” on its own. It can only ever be “correct with respect to a given specification”. Whether the formal specification correctly describes the problem to be solved is a separate issue. It is also a difficult issue to address, since it ultimately concerns the problem constructing abstracted formal representations of an informal concrete <a href="/wiki/Problem_domain" title="Problem domain">problem domain</a>, and such an abstraction step is not amenable to formal proof. However, it is possible to <a href="/wiki/Verification_and_validation" title="Verification and validation">validate</a> a specification by proving “challenge” <a href="/wiki/Theorem" title="Theorem">theorems</a> concerning properties that the specification is expected to exhibit. If correct, these theorems reinforce the specifier's understanding of the specification and its relationship with the underlying problem domain. If not, the specification probably needs to be changed to better reflect the domain understanding of those involved with producing (and implementing) the specification.</p>/n<p>Formal methods of software development are not widely used in industry. Most companies do not consider it cost-effective to apply them in their software development processes.<sup id="cite_ref-Ch_27_Formal_spec_4-0" class="reference"><a href="#cite_note-Ch_27_Formal_spec-4">[4]</a></sup> This may be for a variety of reasons, some of which are:</p>/n<ul>/n<li>Time/n<ul>/n<li>High initial start up cost with low measurable returns</li>/n</ul>/n</li>/n<li>Flexibility/n<ul>/n<li>A lot of software companies use <a href="/wiki/Agile_software_development" title="Agile software development">agile methodologies</a> that focus on flexibility. Doing a formal specification of the whole system up front is often perceived as being the opposite of flexible. However, there is some research into the benefits of using formal specifications with "agile" development<sup id="cite_ref-p14-nummenmaasen_5-0" class="reference"><a href="#cite_note-p14-nummenmaasen-5">[5]</a></sup></li>/n</ul>/n</li>/n<li>Complexity/n<ul>/n<li>They require a high level of mathematical expertise and the analytical skills to understand and apply them effectively<sup id="cite_ref-p14-nummenmaasen_5-1" class="reference"><a href="#cite_note-p14-nummenmaasen-5">[5]</a></sup></li>/n<li>A solution to this would be to develop tools and models that allow for these techniques to be implemented but hide the underlying mathematics<sup id="cite_ref-p223-gaudel_2-3" class="reference"><a href="#cite_note-p223-gaudel-2">[2]</a></sup><sup id="cite_ref-p14-nummenmaasen_5-2" class="reference"><a href="#cite_note-p14-nummenmaasen-5">[5]</a></sup></li>/n</ul>/n</li>/n<li>Limited scope<sup id="cite_ref-lamsweerde-roadmap_3-3" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup>/n<ul>/n<li>They do not capture properties of interest for all <a href="/wiki/Requirements_analysis#Stakeholder_identification" title="Requirements analysis">stakeholders</a> in the project<sup id="cite_ref-lamsweerde-roadmap_3-4" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup></li>/n<li>They do not do a good job of specifying user interfaces and user interaction<sup id="cite_ref-Ch_27_Formal_spec_4-1" class="reference"><a href="#cite_note-Ch_27_Formal_spec-4">[4]</a></sup></li>/n</ul>/n</li>/n<li>Not cost-effective/n<ul>/n<li>This is not entirely true, by limiting their use to only core parts of critical systems they have shown to be cost-effective<sup id="cite_ref-Ch_27_Formal_spec_4-2" class="reference"><a href="#cite_note-Ch_27_Formal_spec-4">[4]</a></sup></li>/n</ul>/n</li>/n</ul>/n<p>Other limitations:<sup id="cite_ref-lamsweerde-roadmap_3-5" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup></p>/n<ul>/n<li>Isolation</li>/n<li>Low-level ontologies</li>/n<li>Poor guidance</li>/n<li>Poor separation of concerns</li>/n<li>Poor tool feedback</li>/n</ul>/n<h2><span class="mw-headline" id="Paradigms">Paradigms</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Formal_specification&amp;action=edit&amp;section=4" title="Edit section: Paradigms">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Formal specification techniques have existed in various domains and on various scales for quite some time.<sup id="cite_ref-p179-vanderpoll_6-0" class="reference"><a href="#cite_note-p179-vanderpoll-6">[6]</a></sup> Implementations of formal specifications will differ depending on what kind of system they are attempting to model, how they are applied and at what point in the software life cycle they have been introduced.<sup id="cite_ref-p223-gaudel_2-4" class="reference"><a href="#cite_note-p223-gaudel-2">[2]</a></sup> These types of models can be categorized into the following specification paradigms:</p>/n<ul>/n<li>History-based specification<sup id="cite_ref-lamsweerde-roadmap_3-6" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup>/n<ul>/n<li>behavior based on system histories</li>/n<li>assertions are interpreted over time</li>/n</ul>/n</li>/n<li>State-based Specification<sup id="cite_ref-lamsweerde-roadmap_3-7" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup>/n<ul>/n<li>behavior based on system states</li>/n<li>series of sequential steps, (e.g. a financial transaction)</li>/n<li>languages such as <a href="/wiki/Z_notation" title="Z notation">Z</a>, <a href="/wiki/Vienna_Development_Method" title="Vienna Development Method">VDM</a> or <a href="/wiki/B-Method" title="B-Method">B</a> rely on this paradigm<sup id="cite_ref-lamsweerde-roadmap_3-8" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup></li>/n</ul>/n</li>/n<li>Transition-based specification<sup id="cite_ref-lamsweerde-roadmap_3-9" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup>/n<ul>/n<li>behavior based on transitions from state-to-state of the system</li>/n<li>best used with a reactive system</li>/n<li>languages such as Statecharts, PROMELA, STeP-SPL, RSML or SCR rely on this paradigm<sup id="cite_ref-lamsweerde-roadmap_3-10" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup></li>/n</ul>/n</li>/n<li>Functional specification<sup id="cite_ref-lamsweerde-roadmap_3-11" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup>/n<ul>/n<li>specify a system as a structure of mathematical functions</li>/n<li>OBJ, ASL, PLUSS, LARCH, HOL or PVS rely on this paradigm<sup id="cite_ref-lamsweerde-roadmap_3-12" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup></li>/n</ul>/n</li>/n<li>Operational Specification<sup id="cite_ref-lamsweerde-roadmap_3-13" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup>/n<ul>/n<li>early languages such as <a href="/w/index.php?title=Paisley_(programming_language)&amp;action=edit&amp;redlink=1" class="new" title="Paisley (programming language) (page does not exist)">Paisley</a>, GIST, Petri nets or process algebras rely on this paradigm<sup id="cite_ref-lamsweerde-roadmap_3-14" class="reference"><a href="#cite_note-lamsweerde-roadmap-3">[3]</a></sup></li>/n</ul>/n</li>/n</ul>/n<p>In addition to the above paradigms there are ways to apply certain heuristics to help improve the creation of these specifications. The paper referenced here best discusses heuristics to use when designing a specification.<sup id="cite_ref-p179-vanderpoll_6-1" class="reference"><a href="#cite_note-p179-vanderpoll-6">[6]</a></sup> They do so by applying a <a href="/wiki/Divide_and_conquer_algorithm" title="Divide and conquer algorithm">divide-and-conquer</a> approach.</p>/n<h2><span class="mw-headline" id="Software_tools">Software tools</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Formal_specification&amp;action=edit&amp;section=5" title="Edit section: Software tools">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The <a href="/wiki/Z_notation" title="Z notation">Z notation</a> is an example of a leading formal <a href="/wiki/Specification_language" title="Specification language">specification language</a>. Others include the Specification Language(VDM-SL) of the <a href="/wiki/Vienna_Development_Method" title="Vienna Development Method">Vienna Development Method</a> and the <a href="/w/index.php?title=Abstract_Machine_Notation&amp;action=edit&amp;redlink=1" class="new" title="Abstract Machine Notation (page does not exist)">Abstract Machine Notation</a> (AMN) of the <a href="/wiki/B-Method" title="B-Method">B-Method</a>. class="reference"><a href="#cite_note-7">[7]</a></sup> (Web services <a href="/wiki/Quality_of_Service" class="mw-redirect" title="Quality of Service">Quality of Service</a>).</p>/n<p>Some tools are:<sup id="cite_ref-Ch_27_Formal_spec_4-3" class="reference"><a href="#cite_note-Ch_27_Formal_spec-4">[4]</a></sup></p>/n<ul>/n<li>Algebraic/n<ol>/n<li><a href="/wiki/Larch_family" title="Larch family">Larch</a></li>/n<li><a href="/wiki/OBJ_(programming_language)" title="OBJ (programming language)">OBJ</a></li>/n<li><a href="/wiki/Language_Of_Temporal_Ordering_Specification" title="Language Of Temporal Ordering Specification">Lotos</a></li>/n</ol>/n</li>/n<li>Model-based/n<ol>/n<li><a href="/wiki/Z_notation" title="Z notation">Z</a></li>/n<li><a href="/wiki/B-Method"
