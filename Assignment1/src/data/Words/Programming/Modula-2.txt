title="Seed7">Seed7</a>, <a href="/wiki/Zonnon" title="Zonnon">Zonnon</a>, Modula-GM</td>/n</tr>/n</table>/n<p><b>Modula-2</b> is a computer <a href="/wiki/Programming_language" title="Programming language">programming language</a> designed and developed between 1977 and 1985 by <a href="/wiki/Niklaus_Wirth" title="Niklaus Wirth">Niklaus Wirth</a> at the Swiss Federal Institute of Technology in Zurich (<a href="/wiki/ETH_Zurich" title="ETH Zurich">ETH Zurich</a>) as a revision of <a href="/wiki/Pascal_(programming_language)" title="Pascal (programming language)">Pascal</a> to serve as the sole programming language for the operating system and application software for the personal workstation <a href="/wiki/Lilith_(computer)" title="Lilith (computer)">Lilith</a>.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> The principal concepts were:</p>/n<ul>/n<li>The module as a compilation unit for separate compilation</li>/n<li>The coroutine as the basic building block for concurrent processes</li>/n<li>Types and procedures that allow access to machine-specific data.</li>/n</ul>/n<p>Modula-2 was viewed by Niklaus Wirth as a successor to his earlier programming languages <a href="/wiki/Pascal_(programming_language)" title="Pascal (programming language)">Pascal</a> and <a href="/wiki/Modula" title="Modula">Modula</a>.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup><sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup> The language design was also influenced by the <a title="Xerox Alto">Xerox Alto</a>, both from Xerox, that Wirth saw during his 1976 <a href="/wiki/Sabbatical" title="Sabbatical">sabbatical</a> year at <a href="/wiki/Xerox_PARC" class="mw-redirect" title="Xerox PARC">Xerox PARC</a>.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> The computer magazine <a href="/wiki/BYTE" class="mw-redirect" title="BYTE">BYTE</a> devoted the August 1984 issue to the language and its surrounding environment.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup></p>/n<p></p>/n<div id="toc" class="toc">/n<div id="Description">Description</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Modula-2&amp;action=edit&amp;section=1" title="Edit section: Description">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Modula-2 is a general purpose <a href="/wiki/Procedural_language" class="mw-redirect" title="Procedural language">procedural language</a>, sufficiently flexible to do systems programming, but with much broader application. In particular, it was designed to support separate compilation and data abstraction in a straightforward way. Much of the syntax is based on Wirth's earlier and better-known language, <a href="/wiki/Pascal_(programming_language)" title="Pascal (programming language)">Pascal</a>. Modula-2 was designed to be broadly similar to Pascal, with some elements and syntactic ambiguities removed and the important addition of the <i>module</i> concept, and direct language support for <a href="/wiki/Multiprogramming" class="mw-redirect" title="Multiprogramming">multiprogramming</a>.</p>/n<p>Here is an example of the source code for the "Hello world" program:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="kr">MODULE</span> <span class="n">Hello</span><span class="p">;</span>/n<span class="kr">FROM</span> <span class="n">STextIO</span> <span class="kr">IMPORT</span> <span class="n">WriteString</span><span class="p">;</span>/n<span class="kr">BEGIN</span>/n  <span class="n">WriteString</span><span class="p">(</span><span class="s">"Hello World!"</span><span class="p">);</span>/n<span class="kr">END</span> <span class="n">Hello</span><span class="p">.</span>/n</pre></div>/n<p>The Modula-2 <i>module</i> may be used to encapsulate a set of related subprograms and data structures, and restrict their visibility from other portions of the program. The module design implemented the data abstraction feature of Modula-2 in a very clean way. Modula-2 programs are composed of modules, each of which is made up of two parts: a <b>definition module</b>, the interface portion, which contains only those parts of the subsystem that are <i>exported</i> (visible to other modules), and an <b>implementation module</b>, which contains the working code that is internal to the module.</p>/n<p>The language has strict scope control. In particular the scope of a module can be considered as an impenetrable wall: Except for standard identifiers no object from the outer world is visible inside a module unless explicitly imported; no internal module object is visible from the outside unless explicitly exported.</p>/n<p>Suppose module M1 exports objects a, b, c, and P by enumerating its identifiers in an explicit export list</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n  <span class="kr">DEFINITION</span> <span class="kr">MODULE</span> <span class="n">M1</span><span class="p">;</span>/n    <span class="kr">EXPORT</span> <span class="kr">QUALIFIED</span> <span class="n">a</span><span class="p">,</span> <span class="n">b</span><span class="p">,</span> <span class="n">c</span><span class="p">,</span> <span class="n">P</span><span class="p">;</span>/n    <span class="p">...</span>/n</pre></div>/n<p>Then the objects a, b,c, and P from module M1 become now known outside module M1 as M1.a, M1.b, M1.c, and M1.P. They are exported in a <i>qualified</i> manner to the universe (assumed module M1 is global). The exporting module's name, i.e. M1, is used as a qualifier followed by the object's name.</p>/n<p>Suppose module M2 contains the following IMPORT declaration</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n  <span class="kr">MODULE</span> <span class="n">M2</span><span class="p">;</span>/n    <span class="kr">IMPORT</span> <span class="n">M1</span><span class="p">;</span>/n    <span class="p">...</span>/n</pre></div>/n<p>Then this means that the objects exported by module M1 to the universe of its enclosing program can now be used inside module M2. They are referenced in a <i>qualified</i> manner like this: M1.a, M1.b, M1.c, and M1.P. Example:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n    <span class="p">...</span>/n    <span class="n">M1</span><span class="p">.</span><span class="n">a</span> <span class="o">:=</span> <span class="mi">0</span><span class="p">;</span>/n    <span class="n">M1</span><span class="p">.</span><span class="n">c</span> <span class="o">:=</span> <span class="n">M1</span><span class="p">.</span><span class="n">P</span><span class="p">(</span><span class="n">M1</span><span class="p">.</span><span class="n">a</span> <span class="o">+</span> <span class="n">M1</span><span class="p">.</span><span class="n">b</span><span class="p">);</span>/n    <span class="p">...</span>/n</pre></div>/n<p>Qualified export avoids name clashes: For instance, if another module M3 would also export an object called P, then we can still distinguish the two objects, since M1.P differs from M3.P. Thanks to the qualified export it does not matter that both objects are called P inside their exporting modules M1 and M3.</p>/n<p>There is an alternative technique available, which is in wide use by Modula-2 programmers. Suppose module M4 is formulated as this</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n  <span class="kr">MODULE</span> <span class="n">M4</span><span class="p">;</span>/n    <span class="kr">FROM</span> <span class="n">M1</span> <span class="kr">IMPORT</span> <span class="n">a</span><span class="p">,</span> <span class="n">b</span><span class="p">,</span> <span class="n">c</span><span class="p">,</span> <span class="n">P</span><span class="p">;</span>/n</pre></div>/n<p>Then this means that objects exported by module M1 to the universe can again be used inside module M4, but now by mere references to the exported identifiers in an "unqualified" manner like this: a, b, c, and P. Example:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n    <span class="p">...</span>/n    <span class="n">a</span> <span class="o">:=</span> <span class="mi">0</span><span class="p">;</span>/n    <span class="n">c</span> <span class="o">:=</span> <span class="n">P</span><span class="p">(</span><span class="n">a</span> <span class="o">+</span> <span class="n">b</span><span class="p">);</span>/n    <span class="p">...</span>/n</pre></div>/n<p>This technique of unqualifying import allows use of variables and other objects outside their exporting module in exactly the same simple, i.e. <i>unqualified</i>, manner as inside the exporting module. The walls surrounding all modules have now become irrelevant for all those objects for which this has been explicitly allowed. Of course unqualifying import is only usable if there are no name clashes.</p>/n<p>These export and import rules may seem unnecessarily restrictive and verbose. But they do not only safeguard objects against unwanted access, but also have the pleasant side-effect of providing automatic cross-referencing of the definition of every identifier in a program: if the identifier is qualified by a module name, then the definition comes from that module. Otherwise if it occurs unqualified, simply search backwards, and you will either encounter a declaration of that identifier, or its occurrence in an IMPORT statement which names the module it comes from. This property becomes very useful when trying to understand large programs containing many modules.</p>/n<p>The language provides for (limited) single-processor concurrency (<a href="/wiki/Monitor_(synchronization)" title="Monitor (synchronization)">monitors</a>, <a href="/wiki/Coroutine" title="Coroutine">coroutines</a> and explicit transfer of control) and for hardware access (absolute addresses, bit manipulation, and <a href="/wiki/Interrupt" title="Interrupt">interrupts</a>). It uses a <a href="/wiki/Nominal_type_system" title="Nominal type system">nominal type system</a>.</p>/n<h2><span class="mw-headline" id="Dialects">Dialects</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Modula-2&amp;action=edit&amp;section=2" title="Edit section: Dialects">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>There are two major dialects of Modula-2. The first is <b>PIM</b>, named after the book "Programming in Modula-2" by Niklaus Wirth. There were three major editions of PIM, the second, third (corrected) and fourth editions, each describing slight variants of the language. The second major dialect is <b>ISO</b>, from the standardization effort by the <a href="/wiki/International_Organization_for_Standardization" title="International Organization for Standardization">International Organization for Standardization</a>. Here are a few of the differences amongst them.</p>/n<ul>/n<li><b>PIM2</b> (1983)/n<ul>/n<li>Required explicit <b>EXPORT</b> clause in definition modules.</li>/n<li>Function <b>SIZE</b> needs to be imported from module <b>SYSTEM</b></li>/n</ul>/n</li>/n<li><b>PIM3</b> (1985)/n<ul>/n<li>Removed the <b>EXPORT</b> clause from definition modules following the observation that everything within a definition module defines the interface to that module, hence the <b>EXPORT</b> clause was redundant.</li>/n<li>Function <b>SIZE</b> is pervasive (visible in any scope without import)</li>/n</ul>/n</li>/n<li><b>PIM4</b> (1988)/n<ul>/n<li>Specified the behaviour of the <b>MOD</b> operator when the operands are negative.</li>/n<li>Required all <b>ARRAY OF CHAR</b> strings to be terminated by ASCII NUL, even if the string fits exactly into its array.</li>/n</ul>/n</li>/n<li><b>ISO</b> (1996 and 1998)/n<ul>/n<li>ISO Modula-2 resolved most of the ambiguities in PIM Modula-2. It added the data types <b>COMPLEX</b> and <b>LONGCOMPLEX</b>, exceptions, module termination (<b>FINALLY</b> clause) and a complete standard I/O library. There are numerous minor differences and clarifications.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup></li>/n</ul>/n</li>/n</ul>/n<h2><span class="mw-headline" The Mod51 compiler generates standalone code for 80C51 based microcontrollers.</p>/n<h3><span class="mw-headline" id="Modula-GM">Modula-GM</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Modula-2&amp;action=edit&amp;section=11" title="Edit section: Modula-GM">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p><a href="/wiki/Delco_Electronics" title="Delco Electronics">Delco Electronics</a>, then a subsidiary of GM <a href="/wiki/Hughes_Electronics" title="Hughes Electronics">Hughes Electronics</a>, developed a version of Modula-2 for embedded control systems starting in 1985. Delco named it Modula-GM. It was the first high level language used to replace machine language code for embedded systems in Delco's engine control units (ECUs). This was significant because Delco was producing over 28,000 ECUs per day in 1988 for GM; this was then the world's largest producer of ECUs.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">[16]</a></sup> The first experimental use of Modula-GM in an embedded controller was in the 1985 Antilock Braking System Controller which was based on the Motorola 68xxx microprocessor, and in 1993 Gen-4 ECU used by the <a href="/wiki/Champ_Car" title="Champ Car">CART</a> (Championship Auto Racing Teams) and <a href="/wiki/Indy_Racing_League" class="mw-redirect" title="Indy Racing League">IRL</a> (Indy Racing League) teams.<sup id="cite_ref-17" class="reference"><a href="#cite_note-17">[17]</a></sup> The first production use of Modula-GM was its use in GM trucks starting with the 1990 model year VCM (Vehicle Control Module) used to manage GM Powertrain's <a href="/wiki/Vortec" class="mw-redirect" title="Vortec">Vortec</a> engines. Modula-GM was also used on all ECUs for GM's <a href="/wiki/Buick_V6_engine" title="Buick V6 engine">90° Buick V6 family</a> 3800 Series II used in the 1997-2005 model year <a href="/wiki/Buick_Park_Avenue" title="Buick Park Avenue">Buick Park Avenue</a>. The Modula-GM compilers and associated software management tools were sourced by Delco from <a href="/wiki/Intermetrics" title="Intermetrics">Intermetrics</a>.</p>/n<p>Modula-2 was selected as the basis for Delco's high level language because of its many strengths over other alternative language choices in 1986. After Delco Electronics was spun off from GM (with other component divisions) to form <a href="/wiki/Delphi_(auto_parts)" class="mw-redirect" title="Delphi (auto parts)">Delphi</a> in 1997, global sourcing required that a non-proprietary high-level software language be used.
