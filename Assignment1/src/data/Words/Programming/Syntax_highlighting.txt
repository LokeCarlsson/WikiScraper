data-file-height="250" /></a>/n<div class="thumbcaption"><a href="/wiki/HTML" title="HTML">HTML</a> syntax highlighting</div>/n</div>/n</div>/n<p><b>Syntax highlighting</b> is a feature of <a href="/wiki/Text_editor" title="Text editor">text editors</a> that are used for <a href="/wiki/Software_development" title="Software development">programming</a>, <a href="/wiki/Scripting_language" title="Scripting language">scripting</a>, or <a href="/wiki/Markup_language" title="Markup language">markup</a> languages, such as <a href="/wiki/HTML" title="HTML">HTML</a>. The feature displays text, especially <a href="/wiki/Source_code" title="Source code">source code</a>, in different colors and <a href="/wiki/Typeface" title="Typeface">fonts</a> according to the category of terms.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> This feature facilitates writing in a structured language such as a <a href="/wiki/Programming_language" title="Programming language">programming language</a> or a <a href="/wiki/Markup_language" title="Markup language">markup language</a> as both structures and <a href="/wiki/Syntax_error" title="Syntax error">syntax errors</a> are visually distinct. Highlighting does not affect the meaning of the text itself; it is intended only for human readers.</p>/n<p>Syntax highlighting is a form of <a href="/wiki/Secondary_notation" title="Secondary notation">secondary notation</a>, since the highlights are not part of the text meaning, but serve to reinforce it. Some editors also integrate syntax highlighting with other features, such as <a href="/wiki/Spell_checking" class="mw-redirect" title="Spell checking">spell checking</a> or <a href="/wiki/Code_folding" title="Code folding">code folding</a>, as aids to editing which are external to the language.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Syntax_highlighting&amp;action=edit&amp;section=1" title="Edit section: Practical benefits">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div class="thumb tright">/n<div class="thumbinner" style="width:314px;"><a href="/wiki/File:Syntax-highlighting-javascript.gif" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/6/60/Syntax-highlighting-javascript.gif/312px-Syntax-highlighting-javascript.gif" width="312" height="84" class="thumbimage" href="/wiki/File:Syntax-highlighting-javascript.gif" class="internal" title="Enlarge"></a></div>/nHighlighting the effect of missing delimiter (after <i>watch='false</i>) in <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a></div>/n</div>/n</div>/n<p>Syntax highlighting is one strategy to improve the readability and context of the text; especially for code that spans several pages. The reader can easily ignore large sections of comments or code, depending on what they are looking for. Syntax highlighting also helps programmers find errors in their program. For example, most editors highlight <a href="/wiki/String_literal" title="String literal">string literals</a> in a different color. Consequently, spotting a missing <a href="/wiki/String_literal#Bracketed_delimiters" title="String literal">delimiter</a> is much easier because of the contrasting color of the text. <a href="/wiki/Brace_matching" title="Brace matching">Brace matching</a> is another important feature with many popular editors. This makes it simple to see if a brace has been left out or locate the match of the brace the cursor is on by highlighting the pair in a different color.</p>/n<p>A study published in the conference <a href="/wiki/Psychology_of_programming" title="Psychology of programming">PPIG</a> evaluated the effects of syntax highlighting on the comprehension of short programs, finding that the presence of syntax highlighting significantly reduces the time taken for a programmer to internalise the semantics of a program.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> Additionally, data gathered from an eye-tracker during the study suggested that syntax highlighting enables programmers to pay less attention to standard syntactic components such as keywords.</p>/n<h2><span class="mw-headline" id="Support_in_text_editors">Support in text editors</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Syntax_highlighting&amp;action=edit&amp;section=2" title="Edit section: Support in text editors">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Some text editors can also export the coloured markup in a format that is suitable for printing or for importing into word-processing and other kinds of text-formatting software; for instance as a HTML, colorized LaTeX, <a href="/wiki/PostScript" title="PostScript">PostScript</a> or <a href="/wiki/Rich_Text_Format" title="Rich Text Format">RTF</a> version of its syntax highlighting. There are several syntax highlighting <a href="/wiki/Library_(computing)" title="Library (computing)">libraries</a> or "engines" that can be used in other applications, but are not complete programs in themselves, for example the Generic Syntax Highlighter (<a href="/wiki/GeSHi" title="GeSHi">GeSHi</a>) extension for <a href="/wiki/PHP" title="PHP">PHP</a>.</p>/n<p>For editors that support more than one language, the user can usually specify the language of the text, such as <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a>, <a href="/wiki/LaTeX" title="LaTeX">LaTeX</a>, <a href="/wiki/HTML" title="HTML">HTML</a>, or the text editor can automatically recognize it based on the file extension or by scanning contents of the file. This automatic language detection presents potential problems. For example, a user may want to edit a document containing:</p>/n<ul>/n<li>more than one language (for example when editing an <a href="/wiki/HTML" title="HTML">HTML</a> file that contains embedded <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> code),</li>/n<li>a language that is not recognized (for example when editing source code for an obscure or relatively new programming language),</li>/n<li>a language that differs from the file type (for example when editing source code in an extension-less file in an editor that uses file extensions to detect the language).</li>/n</ul>/n<p>In these cases, it is not clear what language to use, and a document may not be highlighted or be highlighted incorrectly.</p>/n<h2><span class="mw-headline" id="Syntax_elements">Syntax elements</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Syntax_highlighting&amp;action=edit&amp;section=3" title="Edit section: Syntax elements">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Most editors with syntax highlighting allow different colors and text styles to be given to dozens of different lexical sub-elements of syntax. These include keywords, comments, control-flow statements, variables, and other elements. Programmers often heavily customize their settings in an attempt to show as much useful information as possible without making the code difficult to read.</p>/n<h3><span class="mw-headline" The <a href="/wiki/Comment_(computer_programming)" title="Comment (computer programming)">comment</a> at the beginning is also highlighted in a specific manner to distinguish it from working code.</p>/n<h2><span class="mw-headline" id="History_and_limitations">History and limitations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Syntax_highlighting&amp;action=edit&amp;section=5" title="Edit section: History and limitations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The ideas of syntax highlighting overlap significantly with those of <a href="/wiki/Structure_editor" title="Structure editor">syntax-directed editors</a>. One of the first such editor for code was Wilfred Hansen's 1969 code editor, Emily.<sup id="cite_ref-hansen_3-0" class="reference"><a href="#cite_note-hansen-3">[3]</a></sup><sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> It provided advanced language-independent <a href="/wiki/Autocomplete" title="Autocomplete">code completion</a> facilities, and unlike modern editors with syntax highlighting, actually made it impossible to create syntactically incorrect programs.</p>/n<p>In 1985 <a href="/wiki/Ben_Shneiderman" title="Ben Shneiderman">Ben Shneiderman</a> suggested "color coding of text strings to suggest meaning".<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> The Live Parsing Editor (<a href="/wiki/LEXX_(text_editor)" title="LEXX (text editor)">LEXX</a>) written for the <a href="/wiki/VM_(operating_system)" title="VM (operating system)">VM</a> operating system for the computerization of the <i><a href="/wiki/Oxford_English_Dictionary" title="Oxford English Dictionary">Oxford English Dictionary</a></i> in 1985 was one of the first to use color syntax highlighting. Its <i>live parsing</i> capability allowed user-supplied parsers to be added to the editor, for text, programs, data file, etc.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> On <a href="/wiki/Microcomputers" class="mw-redirect" title="Microcomputers">microcomputers</a>, <i>MacPascal 1.0</i> (October 10, 1985) recognized Pascal syntax as it was typed and used font changes (e.g., bold for keywords) to highlight syntax on the monochrome <a href="/wiki/Compact_Macintosh" title="Compact Macintosh">compact Macintosh</a> and automatically indented code to match its structure.<sup id="cite_ref-macpascal_7-0" class="reference"><a href="#cite_note-macpascal-7">[7]</a></sup></p>/n<p>Some text editors and code formatting tools perform syntax highlighting using <a href="/wiki/Pattern_matching" title="Pattern matching">pattern matching</a> <a href="/wiki/Heuristics" class="mw-redirect" title="Heuristics">heuristics</a> (e.g. <a href="/wiki/Regular_expressions" class="mw-redirect" title="Regular expressions">Regular expressions</a>) rather than implementing a <a href="/wiki/Parser" class="mw-redirect" title="Parser">parser</a> for each possible language.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> This can result in a text rendering system displaying somewhat inaccurate syntax highlighting and in some cases performing slowly. A solution used by text editors to overcome this problem is not always parsing the whole file but rather just the visible area, sometimes scanning backwards in the text up to a limited number of lines for "syncing".</p>/n<p>On the other hand, the editor often displays code during its creating, while it is incomplete or incorrect, and the strict parsers (like ones used in compiles) would fail to parse the code most of the time.</p>/n<p>Some modern, language-specific <a href="/wiki/Integrated_development_environment" title="Integrated development environment">IDEs</a> (in contrast to text editors) perform full language parsing which results in very accurate understanding of code. An extension of syntax highlighting was called <i>"semantic highlighting"</i> in 2009 by David Nolden <sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> for the open-source C++ IDE <a href="/wiki/KDevelop" title="KDevelop">KDevelop</a>. For example, semantic highlighting may give local variables unique distinct colors to improve the comprehensibility of code. In 2014 the idea of colored local variables was further popularized due to a blog post by Evan Brooks,<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup> and after that, the idea was transferred to other popular IDEs like <a href="/wiki/Visual_Studio" class="mw-redirect" title="Visual Studio">Visual Studio</a>,<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup> <a href="/wiki/Xcode" title="Xcode">Xcode</a>,<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[12]</a></sup> and others.</p>/n<h2><span
