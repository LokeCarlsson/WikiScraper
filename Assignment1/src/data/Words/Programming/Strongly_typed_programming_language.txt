typing</h1>/n									<div id="bodyContent" class="mw-body-content">/n									<div id="siteSub" class="noprint">From Wikipedia, the free encyclopedia</div>/n								<div id="contentSub"><span class="mw-redirectedfrom">  (Redirected from <a href="/w/index.php?title=Strongly_typed_programming_language&amp;redirect=no" class="mw-redirect" title="Strongly typed programming language">Strongly typed programming language</a>)</span></div>/n												<div id="jump-to-nav" class="mw-jump">/n					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>/n				</div>/n				<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="mw-parser-output"><p>In <a href="/wiki/Computer_programming" title="Computer programming">computer programming</a>, <a href="/wiki/Programming_language" title="Programming language">programming languages</a> are often colloquially classified as <b>strongly typed</b> or <b>weakly typed</b> (<b>loosely typed</b>). These terms do not have a precise definition, but in general, a strongly typed language is more likely to generate an error or refuse to compile if the argument passed to a function does not closely match the expected type. On the other hand, a weakly typed language may produce unpredictable results or may perform implicit type conversion.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> A different but related concept is <a href="/wiki/Latent_typing" title="Latent typing">latent typing</a>.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toctext">References</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Strong_and_weak_typing&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>In 1974, <a href="/wiki/Barbara_Liskov" title="Barbara Liskov">Liskov</a> and Zilles defined a strongly-typed language as one in which "whenever an object is passed from a calling function to a called function, its type must be compatible with the type declared in the called function."<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> In 1977, Jackson wrote, "In a strongly typed language each data area will have a distinct type and each process will state its communication requirements in terms of these types."<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup></p>/n<h2><span id="Definitions_of_&quot;strong&quot;_or_&quot;weak&quot;"></span><span class="mw-headline" id="Definitions_of_.22strong.22_or_.22weak.22">Definitions of "strong" or "weak"</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Strong_and_weak_typing&amp;action=edit&amp;section=2" title="Edit section: Definitions of &quot;strong&quot; or &quot;weak&quot;">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>A number of different language design decisions have been referred to as evidence of "strong" or "weak" typing. In fact, many of these are more accurately understood as the presence or absence of <a href="/wiki/Type_safety" title="Type safety">type safety</a>, <a href="/wiki/Memory_safety" title="Memory safety">memory safety</a>, <a href="/wiki/Static_type-checking" class="mw-redirect" title="Static type-checking">static type-checking</a>, or <a href="/wiki/Type_system#Dynamic_type-checking_and_runtime_type_information" title="Type system">dynamic type-checking</a>.</p>/n<p>"Strong typing" generally refers to use of programming language types in order to both capture invariants of the code, and ensure its correctness, and definitely exclude certain classes of programming errors. Thus there are many "strong typing" disciplines used to achieve these goals.</p>/n<h3><span id="Implicit_type_conversions_and_&quot;type_punning&quot;"></span><span class="mw-headline" id="Implicit_type_conversions_and_.22type_punning.22">Implicit type conversions and "type punning"</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Strong_and_weak_typing&amp;action=edit&amp;section=3" title="Edit section: Implicit type conversions and &quot;type punning&quot;">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Some programming languages make it easy to use a value of one type as if it were a value of another type. This is sometimes described as "weak typing".</p>/n<p>For example, Aahz Maruch opines that "<a href="/wiki/Type_conversion" title="Type conversion">Coercion</a> occurs when you have a <a href="/wiki/Statically_typed" class="mw-redirect" title="Statically typed">statically typed</a> language and you use the syntactic features of the language to force the usage of one type as if it were a different type (consider the common use of void* in C). Coercion is usually a symptom of weak typing. Conversion, on the other hand, creates a brand-new object of the appropriate type."<sup id="cite_ref-artima_4-0" class="reference"><a href="#cite_note-artima-4">[4]</a></sup></p>/n<p>As another example, <a href="/wiki/GNU_Compiler_Collection" title="GNU Compiler Collection">GCC</a> describes this as <a href="/wiki/Type_punning" title="Type punning">type-punning</a> and warns that it will break strict <a href="/wiki/Aliasing_(computing)" title="Aliasing (computing)">aliasing</a>. <a href="/w/index.php?title=Thiago_Macieira&amp;action=edit&amp;redlink=1" class="new" title="Thiago Macieira (page does not exist)">Thiago Macieira</a> discusses several problems that can arise when type-punning causes the <a href="/wiki/Compiler" title="Compiler">compiler</a> to make inappropriate <a href="/wiki/Optimization" class="mw-redirect" title="Optimization">optimizations</a>.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup></p>/n<p>There are many examples of languages that allow <a href="/wiki/Implicit_type_conversion" class="mw-redirect" title="Implicit type conversion">implicit type conversions</a>, but in a type-safe manner. For example, both C++ and C# allow programs to define operators to convert a value from one type to another in a semantically meaningful way. When a C++ compiler encounters such a conversion, it treats the operation just like a function call. In contrast, converting a value to the C type <span style="font-family: monospace, monospace;">void*</span> is an unsafe operation that is invisible to the compiler.</p>/n<h3><span class="mw-headline" id="Pointers">Pointers</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Strong_and_weak_typing&amp;action=edit&amp;section=4" title="Edit section: Pointers">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Some programming languages expose pointers as if they were numeric values, and allow users to perform arithmetic on them. These languages are sometimes referred to as "weakly typed", since pointer arithmetic can be used to bypass the language's type system.</p>/n<h3><span class="mw-headline" id="Untagged_unions">Untagged unions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Strong_and_weak_typing&amp;action=edit&amp;section=5" title="Edit section: Untagged unions">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Some programming languages support <a href="/wiki/Union_type#C.2B.2B" title="Union type">untagged unions</a>, which allow a value of one type to be viewed as if it were a value of another type.</p>/n<h3><span class="mw-headline" id="Static_type-checking">Static type-checking</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Strong_and_weak_typing&amp;action=edit&amp;section=6" title="Edit section: Static type-checking">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>In <a href="/wiki/Luca_Cardelli" title="Luca Cardelli">Luca Cardelli</a>'s article <a href="/w/index.php?title=Typeful_programming&amp;action=edit&amp;redlink=1" class="new" title="Typeful programming (page does not exist)">Typeful Programming</a>,<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> a "strong type system" is described as one in which there is no possibility of an unchecked runtime type error. In other writing, the absence of unchecked run-time errors is referred to as <i>safety</i> or <i>type safety</i>; <a href="/wiki/C._A._R._Hoare" class="mw-redirect" title="C. A. R. Hoare">Tony Hoare</a>'s early papers call this property <i>security</i><sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (April 2017)">citation needed</span></a></i>]</sup>.</p>/n<h3><span class="mw-headline" id="Dynamic_type-checking">Dynamic type-checking</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Strong_and_weak_typing&amp;action=edit&amp;section=7" title="Edit section: Dynamic type-checking">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Some programming languages do not have static type-checking. In many such languages, it is easy to write programs that would be rejected by most static type-checkers. For example, a variable might store either a number or the Boolean value "false".</p>/n<h2><span class="mw-headline" id="Variation_across_programming_languages">Variation across programming languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Strong_and_weak_typing&amp;action=edit&amp;section=8" title="Edit section: Variation across programming languages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Note that some of these definitions are contradictory, others are merely conceptually independent, and still others are special cases (with additional constraints) of other, more "liberal" (less strong) definitions. Because of the wide divergence among these definitions, it is possible to defend claims about most programming languages that they are either strongly or weakly typed. For instance:</p>/n<ul>/n<li><a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, <a href="/wiki/Pascal_(programming_language)" title="Pascal (programming language)">Pascal</a>, <a href="/wiki/Ada_(programming_language)" title="Ada (programming language)">Ada</a> and <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> require all <a href="/wiki/Variable_(programming)" class="mw-redirect" title="Variable (programming)">variables</a> to have a declared type, and support the use of explicit casts of arithmetic values to other arithmetic types. Java, C#, Ada and Pascal are sometimes said to be more strongly typed than C, a claim that is probably based on the fact that C supports more kinds of implicit conversions, and C also allows <a href="/wiki/Pointer_(computer_programming)" title="Pointer (computer programming)">pointer</a> values to be explicitly cast while Java and Pascal do not. Java itself may be considered more strongly typed than Pascal as manners of evading the static type system in Java are controlled by the <a href="/wiki/Java_virtual_machine" title="Java virtual machine">Java virtual machine</a>'s type system. C# and VB.NET are similar to Java in that respect, though they allow disabling of dynamic type checking by explicitly putting code segments in an "unsafe context". Pascal's type system has been described as "too strong", because the size of an array or string is part of its type, making some programming tasks very difficult.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup></li>/n<li><a href="/wiki/Smalltalk" title="Smalltalk">Smalltalk</a>, <a href="/wiki/Perl" title="Perl">Perl</a>, <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a>, <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, and <a href="/wiki/Self_(programming_language)" title="Self (programming language)">Self</a> are all "strongly typed" in the sense that typing errors are prevented at runtime and they do little implicit <a href="/wiki/Type_conversion" title="Type conversion">type conversion</a>, but these languages make no use of static type checking: the compiler does not check or enforce type constraint rules. The term <a href="/wiki/Duck_typing" title="Duck typing">duck typing</a> is now used to describe the <a href="/wiki/Dynamic_typing" class="mw-redirect" title="Dynamic typing">dynamic typing</a> paradigm used by the languages in this group.</li>/n<li>The <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">Lisp</a> family of languages are all "strongly typed" in the sense that typing errors are prevented at runtime. Some Lisp dialects like <a href="/wiki/Common_Lisp" title="Common Lisp">Common Lisp</a> or <a href="/wiki/Clojure" title="Clojure">Clojure</a> do support various forms of type declarations<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> and some compilers (<a href="/wiki/CMUCL" class="mw-redirect" title="CMUCL">CMUCL</a><sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> and related) use these declarations together with <a href="/wiki/Type_inference" title="Type inference">type inference</a> to enable various optimizations and also limited forms of compile time type checks.</li>/n<li><a href="/wiki/Standard_ML" title="Standard ML">Standard ML</a>, <a href="/wiki/F_Sharp_(programming_language)" title="F Sharp (programming language)">F#</a>, <a href="/wiki/OCaml" title="OCaml">OCaml</a>, <a href="/wiki/Haskell_(programming_language)" title="Haskell (programming language)">Haskell</a>, and <a href="/wiki/Rust_(programming_language)" title="Rust (programming language)">Rust</a> are statically type-checked, but the compiler automatically infers a precise type for most values.</li>/n<li><a href="/wiki/Visual_Basic" title="Visual Basic">Visual Basic</a> is a hybrid language. In addition to variables with declared types, it is also possible to declare a variable of "Variant" data type that can store data of any type. Its implicit casts are fairly liberal where, for example, one can sum string variants and pass the result into an integer variable. Visual Basic is weakly typed.</li>/n<li><a href="/wiki/Assembly_language" title="Assembly language">Assembly language</a> and <a href="/wiki/Forth_(programming_language)" title="Forth (programming language)">Forth</a> have been said to be <i>untyped</i>. There is no type checking; it is up to the programmer to ensure that data given to functions is of the appropriate type. Any type conversion required is explicit.</li>/n</ul>/n<p>For this reason, writers who wish to write unambiguously about type systems often eschew the term "strong typing" in favor of specific expressions such as "<a href="/wiki/Type_safety" title="Type safety">type safety</a>".</p>/n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Strong_and_weak_typing&amp;action=edit&amp;section=9" title="Edit section: See also">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<ul>/n<li><a href="/wiki/Data_type" title="Data type">Data type</a> includes a more thorough discussion of typing issues</li>/n<li><a href="/wiki/Comparison_of_programming_languages" title="Comparison of programming languages">Comparison of programming languages</a></li>/n<li><a href="/wiki/Latent_typing" title="Latent typing">Latent typing</a></li>/n<li><a href="/wiki/Memory_safety" title="Memory safety">Memory safety</a></li>/n<li><a href="/wiki/Type_safety" title="Type safety">Type safety</a></li>/n<li><a href="/wiki/Type_system" title="Type system">Type system</a></li>/n</ul>/n<h2><span class="mw-headline" id="References">References</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Strong_and_weak_typing&amp;action=edit&amp;section=10" title="Edit section: References">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div class="reflist columns references-column-width" style="-moz-column-width: 30em; -webkit-column-width: 30em; column-width: 30em; list-style-type: decimal;">/n<ol class="references">/n<li id="cite_note-1"><span class="mw-cite-backlink"><b><a href="#cite_ref-1">^</a></b></span> <span class="reference-text"><cite class="citation web"><a rel="nofollow" class="external text" href="http://www.cs.cornell.edu/courses/CS1130/2012sp/1130selfpaced/module1/module1part4/strongtyping.htm">"CS1130.
