class="mw-parser-output"><p><b>Autocode</b> is the name of a family of "simplified coding systems", later called <a href="/wiki/High-level_programming_language" title="High-level programming language">programming languages</a>, devised in the 1950s and 1960s for a series of <a href="/wiki/Digital_computer" class="mw-redirect" title="Digital computer">digital computers</a> at the Universities of <a href="/wiki/University_of_Manchester" title="University of Manchester">Manchester</a>, <a href="/wiki/University_of_Cambridge" title="University of Cambridge">Cambridge</a> and <a href="/wiki/University_of_London" title="University of London">London</a>. Autocode was a generic term; the autocodes for different machines were not necessarily closely related as are, for example, the different versions of the single language <a href="/wiki/FORTRAN" class="mw-redirect" title="FORTRAN">FORTRAN</a>.</p>/n<p>Today the term is used to refer to the family of early languages descended from the <a href="/wiki/Manchester_Mark_1" title="Manchester Mark 1">Manchester Mark 1</a> autocoder systems, which were generally similar. In the 1960s, the term <b>autocoders</b> was used more generically as to refer to any <a href="/wiki/High-level_programming_language" title="High-level programming language">high-level programming language</a> using a <a href="/wiki/Compiler" title="Compiler">compiler</a>.<sup id="cite_ref-kleith_1-0" class="reference"><a href="#cite_note-kleith-1">[1]</a></sup> Examples of languages referred to as autocodes are <a href="/wiki/COBOL" title="COBOL">COBOL</a> and <a href="/wiki/Fortran" title="Fortran">Fortran</a>.<sup id="cite_ref-kleith2_2-0" class="reference"><a href="#cite_note-kleith2-2">[2]</a></sup></p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toctitle">/n<h2>Contents</h2>/n</div>/n<ul>/n<li class="toclevel-1 tocsection-1"><a id="Glennie.27s_Autocode">Glennie's Autocode</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Autocode&amp;action=edit&amp;section=1" title="Edit section: Glennie's Autocode">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The first autocode and its compiler were developed by <a href="/wiki/Alick_Glennie" title="Alick Glennie">Alick Glennie</a> in 1952 for the Mark 1 computer at the University of Manchester and is considered by some to be the first <a href="/wiki/Compiler" title="Compiler">compiled</a> programming language. His main goal was increased comprehensibility in the programming of Mark 1 machines, which were known for their particularly abstruse machine code. Although the resulting language was much clearer than the machine code, it was still very machine dependent.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup></p>/n<p>Below is an example of Glennie's Autocode function which calculates the formula: <span class="mwe-math-element"><span class="mwe-math-mathml-inline mwe-math-mathml-a11y" style="display: none;"><math xmlns="http://www.w3.org/1998/Math/MathML" >/n  <semantics>/n    <mrow class="MJX-TeXAtom-ORD">/n      <mstyle displaystyle="true" scriptlevel="0">/n        <mi>f</mi>/n        <mo stretchy="false">(</mo>/n        <mi>t</mi>/n        <mo stretchy="false">)</mo>/n        <mo>=</mo>/n        <mrow class="MJX-TeXAtom-ORD">/n          <msqrt>/n            <mrow class="MJX-TeXAtom-ORD">/n              <mo stretchy="false">|</mo>/n            </mrow>/n            <mi>t</mi>/n            <mrow class="MJX-TeXAtom-ORD">/n              <mo stretchy="false">|</mo>/n            </mrow>/n          </msqrt>/n        </mrow>/n        <mo>+</mo>/n        <mn>5</mn>/n        <msup>/n          <mi>t</mi>/n          <mrow class="MJX-TeXAtom-ORD">/n            <mn>3</mn>/n          </mrow>/n        </msup>/n      </mstyle>/n    </mrow>/n    <annotation encoding="application/x-tex">{\displaystyle f(t)={\sqrt {|t|}}+5t^{3}}</annotation>/n  </semantics>/n</math></span><img src="https://wikimedia.org/api/rest_v1/media/math/render/svg/1728f3c549413e39ffdb1cf55e47490b1136bde1" class="mwe-math-fallback-image-inline" aria-hidden="true" style="vertical-align: -1.838ex; width:17.513ex; height:4.843ex;" alt="f(t)={\sqrt  {|t|}}+5t^{3}" /></span>. The example omits necessary scaling instruction needed to place integers into variables and assumes that results of multiplication fit into lower accumulator.</p>/n<pre>/nc@VA t@IC x@½C y@RC z@NC/nINTEGERS +5 →c           # Put 5 into c/n      →t                 # Load argument from lower accumulator to variable t/n   +t     TESTA Z        # Put |t| into lower accumulator/n   -t/n          ENTRY Z/nSUBROUTINE 6 →z          # Run square root subroutine on lower accumulator/n                         # value and put the result into z/n  +tt →y →x              # Calculate t^3 and put it into x /n  +tx →y →x/n+z+cx   CLOSE WRITE 1    # Put z + (c * x) into lower accumulator and return/n</pre>/n<p>User's manual of Glennie's Autocode Compiler mentioned that "the loss of efficiency is no more than 10%".<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup></p>/n<p>Autocode's impact on other Manchester users' programming habits was negligible. It wasn't even mentioned in Brooker's 1958 paper called "The Autocode Programs developed for the Manchester University Computers".</p>/n<h2><span class="mw-headline" id="Mark_1_Autocode">Mark 1 Autocode</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Autocode&amp;action=edit&amp;section=2" title="Edit section: Mark 1 Autocode">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The second autocode for the Mark 1 was planned in 1954 and developed by <a href="/wiki/Tony_Brooker" title="Tony Brooker">R. A. Brooker</a> in 1955 and was called the "Mark 1 Autocode". The language was nearly machine-independent and had floating-point arithmetic, unlike the first one. On the other hand it allowed only one operation per line, offered few mnemonic names and had no way to define user subroutines.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup></p>/n<p>An example code which loads array of size 11 of floating-point numbers from the input would look like this</p>/n<pre>/n1     n1 = 1         /n     vn1 = I         reads input into v[n[1]]/n      n1 = n1 + 1/n   j1,11 ≥ n1        jumps to 1 if n[1] ≤ 11/n</pre>/n<p>Brooker's Autocode removed two main difficulties of Mark 1's programmer: scaling and management of two-level storage. Unlike its predecessor it was heavily used.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup></p>/n<h2><span class="mw-headline" id="Later_Autocodes">Later Autocodes</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Autocode&amp;action=edit&amp;section=3" title="Edit section: Later Autocodes">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Brooker also developed an autocode for the <a href="/wiki/Ferranti_Mercury" title="Ferranti Mercury">Ferranti Mercury</a> in the 1950s in conjunction with the University of Manchester. <i>Mercury Autocode</i> had a limited repertoire of <a href="/wiki/Variable_(programming)" class="mw-redirect" title="Variable (programming)">variables</a> a-z and a'-z' and, in some ways resembled early versions of the later <a href="/wiki/Dartmouth_BASIC" title="Dartmouth BASIC">Dartmouth BASIC</a> language. It pre-dated <a href="/wiki/ALGOL" title="ALGOL">ALGOL</a>, having no concept of <a href="/wiki/Call_stack" title="Call stack">stacks</a> and hence no <a href="/wiki/Recursion" title="Recursion">recursion</a> or <a href="/wiki/Dynamic_memory_allocation" class="mw-redirect" title="Dynamic memory allocation">dynamically-allocated</a> <a href="/wiki/Array_data_structure" title="Array data structure">arrays</a>. In order to overcome the relatively small store size available on Mercury, large programs were written as distinct "chapters", each of which constituted an <a href="/wiki/Overlay_(programming)" title="Overlay (programming)">overlay</a>. Some skill was required to minimise time-consuming transfers of control between chapters. This concept of overlays from <a href="/wiki/Drum_memory" title="Drum memory">drum</a> under user control became common until <a href="/wiki/Virtual_memory" title="Virtual memory">virtual memory</a> became available in later machines. Slightly different dialects of Mercury Autocode were implemented for the <a href="/wiki/Atlas_Computer_(Manchester)" class="mw-redirect" title="Atlas Computer (Manchester)">Ferranti Atlas</a> (distinct from the later <a href="/wiki/Atlas_Autocode" title="Atlas Autocode">Atlas Autocode</a>) and the <a href="/wiki/ICT_1301" title="ICT 1301">ICT 1300</a> and <a href="/wiki/ICT_1900_series" title="ICT 1900 series">1900</a> range.</p>/n<p>The version for the <a href="/wiki/EDSAC" class="mw-redirect" title="EDSAC">EDSAC 2</a> was devised by <a href="/wiki/David_Hartley_(computer_scientist)" title="David Hartley (computer scientist)">David Hartley</a> of <a href="/wiki/University_of_Cambridge_Mathematical_Laboratory" class="mw-redirect" title="University of Cambridge Mathematical Laboratory">University of Cambridge Mathematical Laboratory</a> in 1961. Known as EDSAC 2 Autocode, it was a straight development from Mercury Autocode adapted for local circumstances, and was noted for its object code optimisation and source-language diagnostics which were advanced for the time. A version was developed for the successor <a href="/wiki/Titan_(computer)" title="Titan (computer)">Titan</a> (the prototype Atlas 2 computer) as a temporary stop-gap while a more substantially advanced language known as <a href="/wiki/CPL_(programming_language)" title="CPL (programming language)">CPL</a> was being developed. CPL was never completed but did give rise to <a href="/wiki/BCPL" title="BCPL">BCPL</a> (developed by M. Richards), which in turn led to <a href="/wiki/B_(programming_language)" title="B (programming language)">B</a> and ultimately <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a>. A contemporary but separate thread of development, <a href="/wiki/Atlas_Autocode" title="Atlas Autocode">Atlas Autocode</a> was developed for the University of Manchester <a href="/wiki/Atlas_Computer_(Manchester)" class="mw-redirect" title="Atlas Computer (Manchester)">Atlas 1</a> machine.</p>/n<h2><span class="mw-headline" id="References">References</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Autocode&amp;action=edit&amp;section=4" title="Edit section: References">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div class="reflist" style="list-style-type: decimal;">/n<div class="mw-references-wrap">/n<ol class="references">/n<li id="cite_note-kleith-1"><span class="mw-cite-backlink"><b><a href="#cite_ref-kleith_1-0">^</a></b></span> <span class="reference-text"><cite class="citation book">London, Keith (1968).
