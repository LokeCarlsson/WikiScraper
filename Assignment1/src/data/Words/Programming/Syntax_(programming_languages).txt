Color coded highlighting is used in this piece of code written in <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>.</div>/n</div>/n</div>/n<p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, the <b>syntax</b> of a <a href="/wiki/Computer_language" title="Computer language">computer language</a> is the set of rules that defines the combinations of symbols that are considered to be a correctly structured document or fragment in that language. This applies both to <a href="/wiki/Programming_language" title="Programming language">programming languages</a>, where the document represents <a href="/wiki/Source_code" title="Source code">source code</a>, and <a href="/wiki/Markup_language" title="Markup language">markup languages</a>, where the document represents data. The syntax of a language defines its surface form.<sup id="cite_ref-eopl_1-0" class="reference"><a href="#cite_note-eopl-1">[1]</a></sup> Text-based computer languages are based on sequences of characters, while <a href="/wiki/Visual_programming_languages" class="mw-redirect" title="Visual programming languages">visual programming languages</a> are based on the spatial layout and connections between symbols (which may be textual or graphical). Documents that are syntactically invalid are said to have a <a href="/wiki/Syntax_error" title="Syntax error">syntax error</a>.</p>/n<p>Syntax – the form – is contrasted with <a href="/wiki/Semantics_(computer_science)" title="Semantics (computer science)">semantics</a> – the meaning. In processing computer languages, semantic processing generally comes after syntactic processing, but in some cases semantic processing is necessary for complete syntactic analysis, and these are done together or <a href="/wiki/Concurrency_(computer_science)" title="Concurrency (computer science)">concurrently</a>. In a <a href="/wiki/Compiler" title="Compiler">compiler</a>, the syntactic analysis comprises the <a href="/wiki/Compiler_frontend" class="mw-redirect" title="Compiler frontend">frontend</a>, while semantic analysis comprises the <a href="/wiki/Compiler_backend" class="mw-redirect" title="Compiler backend">backend</a> (and middle end, if this phase is distinguished).</p>/n<p></p>/n<div id="toc" href="/w/index.php?title=Syntax_(programming_languages)&amp;action=edit&amp;section=1" title="Edit section: Levels of syntax">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Computer language syntax is generally distinguished into three levels:</p>/n<ul>/n<li>Words – the lexical level, determining how characters form tokens;</li>/n<li>Phrases – the grammar level, narrowly speaking, determining how tokens form phrases;</li>/n<li>Context – determining what objects or variables names refer to, if types are valid, etc.</li>/n</ul>/n<p>Distinguishing in this way yields modularity, allowing each level to be described and processed separately, and often independently. First a lexer turns the linear sequence of characters into a linear sequence of <a href="/wiki/Token_(parser)" class="mw-redirect" title="Token (parser)">tokens</a>; this is known as "<a href="/wiki/Lexical_analysis" title="Lexical analysis">lexical analysis</a>" or "lexing". Second the parser turns the linear sequence of tokens into a hierarchical syntax tree; this is known as "<a href="/wiki/Parsing" title="Parsing">parsing</a>" narrowly speaking. Thirdly the contextual analysis resolves names and checks types. This modularity is sometimes possible, but in many real-world languages an earlier step depends on a later step – for example, <a href="/wiki/The_lexer_hack" title="The lexer hack">the lexer hack</a> in C is because tokenization depends on context. Even in these cases, syntactical analysis is often seen as approximating this ideal model.</p>/n<p>The parsing stage itself can be divided into two parts: the <a href="/wiki/Parse_tree" title="Parse tree">parse tree</a> or "concrete syntax tree" which is determined by the grammar, but is generally far too detailed for practical use, and the <a href="/wiki/Abstract_syntax_tree" title="Abstract syntax tree">abstract syntax tree</a> (AST), which simplifies this into a usable form. The AST and contextual analysis steps can be considered a form of semantic analysis, as they are adding meaning and interpretation to the syntax, or alternatively as informal, manual implementations of syntactical rules that would be difficult or awkward to describe or implement formally.</p>/n<p>The levels generally correspond to levels in the <a href="/wiki/Chomsky_hierarchy" title="Chomsky hierarchy">Chomsky hierarchy</a>. Words are in a <a href="/wiki/Regular_language" title="Regular language">regular language</a>, specified in the <a href="/wiki/Lexical_grammar" title="Lexical grammar">lexical grammar</a>, which is a Type-3 grammar, generally given as <a href="/wiki/Regular_expression" title="Regular expression">regular expressions</a>. Phrases are in a <a href="/wiki/Context-free_language" title="Context-free language">context-free language</a> (CFL), generally a <a href="/wiki/Deterministic_context-free_language" title="Deterministic context-free language">deterministic context-free language</a> (DCFL), specified in a <a href="/wiki/Phrase_structure_grammar" title="Phrase structure grammar">phrase structure grammar</a>, which is a Type-2 grammar, generally given as <a href="/wiki/Production_(computer_science)" title="Production (computer science)">production rules</a> in <a href="/wiki/Backus%E2%80%93Naur_form" title="Backus–Naur form">Backus–Naur form</a> (BNF). Phrase grammars are often specified in much more constrained grammars than full <a href="/wiki/Context-free_grammar" title="Context-free grammar">context-free grammars</a>, in order to make them easier to parse; while the <a href="/wiki/LR_parser" title="LR parser">LR parser</a> can parse any DCFL in linear time, the simple <a href="/wiki/LALR_parser" title="LALR parser">LALR parser</a> and even simpler <a href="/wiki/LL_parser" title="LL parser">LL parser</a> are more efficient, but can only parse grammars whose production rules are constrained. Contextual structure can in principle be described by a <a href="/wiki/Context-sensitive_grammar" title="Context-sensitive grammar">context-sensitive grammar</a>, and automatically analyzed by means such as <a href="/wiki/Attribute_grammar" title="Attribute grammar">attribute grammars</a>, though in general this step is done manually, via <a href="/wiki/Name_resolution_(programming_languages)" title="Name resolution (programming languages)">name resolution</a> rules and <a href="/wiki/Type_checking" class="mw-redirect" title="Type checking">type checking</a>, and implemented via a <a href="/wiki/Symbol_table" title="Symbol table">symbol table</a> which stores names and types for each scope.</p>/n<p>Tools have been written that automatically generate a lexer from a lexical specification written in regular expressions and a parser from the phrase grammar written in BNF: this allows one to use <a href="/wiki/Declarative_programming" title="Declarative programming">declarative programming</a>, rather than need to have procedural or functional programming. A notable example is the <a href="/wiki/Lex_(software)" title="Lex (software)">lex</a>-<a href="/wiki/Yacc" title="Yacc">yacc</a> pair. These automatically produce a <i>concrete</i> syntax tree; the parser writer must then manually write code describing how this is converted to an <i>abstract</i> syntax tree. Contextual analysis is also generally implemented manually. Despite the existence of these automatic tools, parsing is often implemented manually, for various reasons – perhaps the phrase structure is not context-free, or an alternative implementation improves performance or error-reporting, or allows the grammar to be changed more easily. Parsers are often written in functional languages, such as Haskell, in scripting languages, such as Python or Perl, or in C or C++.</p>/n<h3><span class="mw-headline" id="Examples_of_errors">Examples of errors</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Syntax_(programming_languages)&amp;action=edit&amp;section=2" title="Edit section: Examples of errors">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Syntax_error" title="Syntax error">Syntax error</a></div>/n<p>As an example, <code>(add 1 1)</code> is a syntactically valid Lisp program (assuming the 'add' function exists, else name resolution fails), adding 1 and 1. However, the following are invalid:</p>/n<pre>/n(_ 1 1)    lexical error: '_' is not valid/n(add 1 1   parsing error: missing closing ')'/n</pre>/n<p>Note that the lexer is unable to identify the first error – all it knows is that, after producing the token LEFT_PAREN, '(' the remainder of the program is invalid, since no word rule begins with '_'. The second error is detected at the parsing stage: The parser has identified the "list" production rule due to the '(' token (as the only match), and thus can give an error message; in general it may be ambiguous.</p>/n<p>Type errors and undeclared variable errors are sometimes considered to be syntax errors when they are detected at compile-time (which is usually the case when compiling strongly-typed languages), though it is common to classify these kinds of error as <a href="/wiki/Programming_language#Semantics" title="Programming language">semantic</a> errors instead.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup><sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup><sup id="cite_ref-uninitialized_var_4-0" class="reference"><a href="#cite_note-uninitialized_var-4">[4]</a></sup></p>/n<p>As an example, the Python code</p>/n<pre>/n'a' + 1/n</pre>/n<p>contains a type error because it adds a string literal to an integer literal. Type errors of this kind can be detected at compile-time: They can be detected during parsing (phrase analysis) if the compiler uses separate rules that allow "integerLiteral + integerLiteral" but not "stringLiteral + integerLiteral", though it is more likely that the compiler will use a parsing rule that allows all expressions of the form "LiteralOrIdentifier + LiteralOrIdentifier" and then the error will be detected during contextual analysis (when type checking occurs). In some cases this validation is not done by the compiler, and these errors are only detected at runtime.</p>/n<p>In a dynamically typed language, where type can only be determined at runtime, many type errors can only be detected at runtime. For example, the Python code</p>/n<pre>/na + b/n</pre>/n<p>is syntactically valid at the phrase level, but the correctness of the types of a and b can only be determined at runtime, as variables do not have types in Python, only values do. Whereas there is disagreement about whether a type error detected by the compiler should be called a syntax error (rather than a <a href="/wiki/Programming_language#Static_semantics" title="Programming language">static semantic</a> error), type errors which can only be detected at program execution time are always regarded as semantic rather than syntax errors.</p>/n<h2><span class="mw-headline" id="Syntax_definition">Syntax definition</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Syntax_(programming_languages)&amp;action=edit&amp;section=3" title="Edit section: Syntax definition">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div class="thumb tright">/n<div class="thumbinner" style="width:398px;"><a href="/wiki/File:Python_add5_parse.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/en/thumb/c/cb/Python_add5_parse.svg/396px-Python_add5_parse.svg.png" width="396" height="304" class="thumbimage" class="magnify"><a href="/wiki/File:Python_add5_parse.svg" class="internal" title="Enlarge"></a></div>/n<a href="/wiki/Parse_tree" title="Parse tree">Parse tree</a> of Python code with inset tokenization</div>/n</div>/n</div>/n<p>The syntax of textual programming languages is usually defined using a combination of <a href="/wiki/Regular_expression" title="Regular expression">regular expressions</a> (for <a href="/wiki/Lexical_analysis" title="Lexical analysis">lexical</a> structure) and <a href="/wiki/Backus%E2%80%93Naur_form" title="Backus–Naur form">Backus–Naur form</a> (for <a href="/wiki/Context-free_grammar" title="Context-free grammar">grammatical</a> structure) to inductively specify <a href="/wiki/Syntactic_category" title="Syntactic category">syntactic categories</a> (nonterminals) and <i>terminal</i> symbols. Syntactic categories are defined by rules called <i>productions</i>, which specify the values that belong to a particular syntactic category.<sup id="cite_ref-eopl_1-1" class="reference"><a href="#cite_note-eopl-1">[1]</a></sup> Terminal symbols are the concrete characters or strings of characters (for example <a href="/wiki/Keyword_(computer_programming)" class="mw-redirect" title="Keyword (computer programming)">keywords</a> such as <i>define</i>, <i>if</i>, <i>let</i>, or <i>void</i>) from which syntactically valid programs are constructed.</p>/n<p>A language can have different equivalent grammars, such as equivalent regular expressions (at the lexical levels), or different phrase rules which generate the same language. Using a broader category of grammars, such as LR grammars, can allow shorter or simpler grammars compared with more restricted categories, such as LL grammar, which may require longer grammars with more rules. Different but equivalent phrase grammars yield different parse trees, though the underlying language (set of valid documents) is the same.</p>/n<h3><span class="mw-headline" id="Example:_Lisp">Example: Lisp</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Syntax_(programming_languages)&amp;action=edit&amp;section=4" title="Edit section: Example: Lisp">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Below is a simple grammar, defined using the notation of regular expressions and <a href="/wiki/Extended_Backus%E2%80%93Naur_form" title="Extended Backus–Naur form">Extended Backus–Naur form</a>. It describes the syntax of <a href="/wiki/Lisp_programming_language" class="mw-redirect" title="Lisp programming language">Lisp</a>, which defines productions for the syntactic categories <i>expression</i>, <i>atom</i>, <i>number</i>, <i>symbol</i>, and <i>list</i>:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="k">expression </span><span class="o">=</span> <span class="k">atom   </span><span class="p">|</span> <span class="k">list</span>/n<span class="k">atom       </span><span class="err">=</span> <span class="k">number </span><span class="p">|</span> <span class="k">symbol    </span>/n<span class="k">number     </span><span class="err">=</span> <span class="p">[</span><span class="err">+</span><span class="o">-</span><span class="p">]</span><span class="err">?</span><span class="p">[</span><span class="s1">'0'</span><span class="o">-</span><span class="s1">'9'</span><span class="p">]</span><span class="err">+</span>/n<span class="k">symbol     </span><span class="err">=</span> <span class="p">[</span><span class="s1">'A'</span><span class="o">-</span><span class="s1">'Z''a'</span><span class="o">-</span><span class="s1">'z'</span><span class="p">].</span><span class="err">*</span>/n<span class="k">list       </span><span class="o">=</span> <span class="s1">'('</span><span class="p">,</span> <span class="k">expression</span><span class="err">*</span><span class="p">,</span> <span class="s1">')'</span>/n</pre></div>/n<p>This grammar specifies the following:</p>/n<ul>/n<li>an <i>expression</i> is either an <i>atom</i> or a <i>list</i>;</li>/n<li>an <i>atom</i> is either a <i>number</i> or a <i>symbol</i>;</li>/n<li>a <i>number</i> is an unbroken sequence of one or more decimal digits, optionally preceded by a plus or minus sign;</li>/n<li>a <i>symbol</i> is a letter followed by zero or more of any characters (excluding whitespace); and</li>/n<li>a <i>list</i> is a matched pair of parentheses, with zero or more <i>expressions</i> inside it.</li>/n</ul>/n<p>Here the decimal digits, upper- and lower-case characters, and parentheses are terminal symbols.</p>/n<p>The following are examples of well-formed token sequences in this grammar: '<code>12345</code>', '<code>()</code>', '<code>(a b c232 (1))</code>'</p>/n<h3><span class="mw-headline" id="Complex_grammars">Complex grammars</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Syntax_(programming_languages)&amp;action=edit&amp;section=5" title="Edit section: Complex grammars">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>The grammar needed to specify a programming language can be classified by its position in the <a href="/wiki/Chomsky_hierarchy" title="Chomsky hierarchy">Chomsky hierarchy</a>. The phrase grammar of most programming languages can be specified using a Type-2 grammar, i.e., they are <a href="/wiki/Context-free_grammar" title="Context-free grammar">context-free grammars</a>,<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> though the overall syntax is context-sensitive (due to variable declarations and nested scopes), hence Type-1. However, there are exceptions, and for some languages the phrase grammar is Type-0 (Turing-complete).</p>/n<p>In some languages like Perl and Lisp the specification (or implementation) of the language allows constructs that execute during the parsing phase. Furthermore, these languages have constructs that allow the programmer to alter the behavior of the parser. This combination effectively blurs the distinction between parsing and execution, and makes syntax analysis an <a href="/wiki/Undecidable_problem" title="Undecidable problem">undecidable problem</a> in these languages, meaning that the parsing phase may not finish. For example, in Perl it is possible to execute code during parsing using a <code>BEGIN</code> statement, and Perl function prototypes may alter the syntactic interpretation, and possibly even the syntactic validity of the remaining code.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> Colloquially this is referred to as "only Perl can parse Perl" (because code must be executed during parsing, and can modify the grammar), or more strongly "even Perl cannot parse Perl" (because it is undecidable). Similarly, Lisp <a href="/wiki/Macro_instruction" title="Macro instruction">macros</a> introduced by the <code>defmacro</code> syntax also execute during parsing, meaning that a Lisp compiler must have an entire Lisp run-time system present. In contrast C macros are merely string replacements, and do not require code execution.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup><sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup></p>/n<h2><span class="mw-headline" id="Syntax_versus_semantics">Syntax versus semantics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Syntax_(programming_languages)&amp;action=edit&amp;section=6" title="Edit section: Syntax versus semantics">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The syntax of a language describes the form of a valid program, but does not provide any information about the meaning of the program or the results of executing that program. The meaning given to a combination of symbols is handled by semantics (either <a href="/wiki/Formal_semantics_of_programming_languages" class="mw-redirect" title="Formal semantics of programming languages">formal</a> or hard-coded in a <a href="/wiki/Reference_implementation_(computing)" class="mw-redirect" title="Reference implementation (computing)">reference implementation</a>). Not all syntactically correct programs are semantically correct. Many syntactically correct programs are nonetheless ill-formed, per the language's rules; and may (depending on the language specification and the soundness of the implementation) result in an error on translation or execution. In some cases, such programs may exhibit <a href="/wiki/Undefined_behavior" title="Undefined behavior">undefined behavior</a>. Even when a program is well-defined within a language, it may still have a meaning that is not intended by the person who wrote it.</p>/n<p>Using <a href="/wiki/Natural_language" title="Natural language">natural language</a> as an example, it may not be possible to assign a meaning to a grammatically correct sentence or the sentence may be false:</p>/n<ul>/n<li>"<a href="/wiki/Colorless_green_ideas_sleep_furiously" title="Colorless green ideas sleep furiously">Colorless green ideas sleep furiously</a>." is grammatically well formed but has no generally accepted meaning.</li>/n<li>"John is a married bachelor." is grammatically well formed but expresses a meaning that cannot be true.</li>/n</ul>/n<p>The following C language fragment is syntactically correct, but performs an operation that is not semantically defined (because <tt>p</tt> is a <a href="/wiki/Null_pointer" title="Null pointer">null pointer</a>, the operations <tt>p-&gt;real</tt> and <tt>p-&gt;im</tt> have no meaning):</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n <span class="n">complex</span> <span class="o">*</span><span class="n">p</span> <span class="o">=</span> <span class="nb">NULL</span><span class="p">;</span>/n <span class="n">complex</span> <span class="n">abs_p</span> <span class="o">=</span> <span class="n">sqrt</span> <span class="p">(</span><span class="n">p</span><span class="o">-&gt;</span><span class="n">real</span> <span class="o">*</span> <span class="n">p</span><span class="o">-&gt;</span><span class="n">real</span> <span class="o">+</span> <span class="n">p</span><span class="o">-&gt;</span><span class="n">im</span> <span class="o">*</span> <span class="n">p</span><span class="o">-&gt;</span><span class="n">im</span><span class="p">);</span>/n</pre></div>/n<p>As a simpler example,</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n <span class="kt">int</span> <span class="n">x</span><span class="p">;</span>/n <span class="n">printf</span><span class="p">(</span><span class="s">"%d"</span><span class="p">,</span> <span class="n">x</span><span class="p">);</span>/n</pre></div>/n<p>is syntactically valid, but not semantically defined, as it uses an <a href="/wiki/Uninitialized_variable" title="Uninitialized variable">uninitialized variable</a>. Even though compilers for some programming languages (e.g., Java and C#) would detect uninitialized variable errors of this kind, they should be regarded as <a href="/wiki/Programming_language#Semantics" title="Programming language">semantic</a> errors rather than syntax errors.<sup id="cite_ref-uninitialized_var_4-1" class="reference"><a href="#cite_note-uninitialized_var-4">[4]</a></sup><sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup></p>/n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Syntax_(programming_languages)&amp;action=edit&amp;section=7" title="Edit section: See also">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>To quickly compare syntax of various programming languages, take a look at the list of <a href="/wiki/%22Hello,_World!%22_program" title="&quot;Hello, World!&quot;
