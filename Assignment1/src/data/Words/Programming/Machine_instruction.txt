href="/wiki/Instruction_set" class="mw-redirect" title="Instruction set">instructions</a> executed directly by a <a href="/wiki/Computer" title="Computer">computer</a>'s <a href="/wiki/Central_processing_unit" title="Central processing unit">central processing unit</a> (CPU). Each instruction performs a very specific task, such as a load, a <a href="/wiki/Jump_instruction" class="mw-redirect" title="Jump instruction">jump</a>, or an <a href="/wiki/Arithmetic_logic_unit" title="Arithmetic logic unit">ALU</a> operation on a unit of data in a <a href="/wiki/Processor_register" title="Processor register">CPU register</a> or memory. Every program directly executed by a CPU is made up of a series of such instructions. (The phrase 'directly executed' needs some qualification: machine code is by definition the lowest level of programming detail visible to the programmer, but internally many processors use <a href="/wiki/Microcode" title="Microcode">microcode</a> or optimise and transform machine code instructions into sequences of <a href="/wiki/Micro-operation" title="Micro-operation">micro-ops</a> in a sophisticated way.)</p>/n<p>Numerical machine code (i.e., not <a href="/wiki/Assembly_code" class="mw-redirect" title="Assembly code">assembly code</a>) may be regarded as the lowest-level representation of a <a href="/wiki/Compiler" title="Compiler">compiled</a> or <a href="/wiki/Assembly_language" title="Assembly language">assembled</a> <a href="/wiki/Computer_program" title="Computer program">computer program</a> or as a primitive and <a href="/wiki/Computer_hardware" title="Computer hardware">hardware</a>-dependent <a href="/wiki/Programming_language" title="Programming language">programming language</a>. While it is possible to write programs directly in numerical machine code, it is tedious and error prone to manage individual bits and calculate numerical addresses and constants manually. For this reason, programs are almost never written directly in machine code in modern contexts. Three exceptions are: for low level <a href="/wiki/Debugging" title="Debugging">debugging</a>, program <a href="/wiki/Patch_(computing)" title="Patch (computing)">patching</a>, and <a href="/wiki/Assembly_language" title="Assembly language">assembly language</a> <a href="/wiki/Disassembly" class="mw-redirect" title="Disassembly">disassembly</a> (which would be called for in the absence of assembly <a href="/wiki/Source_code" title="Source code">source code</a>, when only the machine language <a href="/wiki/Object_code" title="Object code">object code</a> is available).</p>/n<p>All practical programs today are written in <a href="/wiki/High-level_programming_language" title="High-level programming language">higher-level languages</a> or assembly language. The source code is then translated to executable machine code by utilities such as <a href="/wiki/Compiler" title="Compiler">compilers</a>, <a href="/wiki/Assembler_(computing)" class="mw-redirect" title="Assembler (computing)">assemblers</a>, and <a href="/wiki/Linker_(computing)" title="Linker (computing)">linkers</a>, with the important exception of <a href="/wiki/Interpreted_language" title="Interpreted language">interpreted</a> programs,<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> which are not translated into machine code. However, the <i><a href="/wiki/Interpreter_(computing)" title="Interpreter (computing)">interpreter</a></i> itself, which may be seen as an executor or processor, performing the instructions of the source code, typically consists of directly executable machine code (generated from assembly or <a href="/wiki/High-level_programming_language" title="High-level programming language">high-level language</a> source code).</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toctitle">/n<h2>Contents</h2>/n</div>/n<ul>/n<li class="toclevel-1 tocsection-1"><a reading</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="Machine_code_instructions">Machine code instructions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Machine_code&amp;action=edit&amp;section=1" title="Edit section: Machine code instructions">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Instruction_set" class="mw-redirect" title="Instruction set">Instruction set</a></div>/n<p>Every processor or processor family has its own machine code <a href="/wiki/Instruction_set" class="mw-redirect" title="Instruction set">instruction set</a>. Instructions are patterns of <a href="/wiki/Bit" title="Bit">bits</a> that by physical design correspond to different commands to the machine. Thus, the instruction set is specific to a class of processors using (mostly) the same architecture. Successor or derivative processor designs often include all the instructions of a predecessor and may add additional instructions. Occasionally, a successor design will discontinue or alter the meaning of some instruction code (typically because it is needed for new purposes), affecting code compatibility to some extent; even nearly completely compatible processors may show slightly different behavior for some instructions, but this is rarely a problem. Systems may also differ in other details, such as memory arrangement, operating systems, or <a href="/wiki/Peripheral" title="Peripheral">peripheral devices</a>. Because a program normally relies on such factors, different systems will typically not run the same machine code, even when the same type of processor is used.</p>/n<p>A machine code instruction set may have all instructions of the same length, or it may have variable-length instructions. How the patterns are organized varies strongly with the particular architecture and often also with the type of instruction. Most instructions have one or more <a href="/wiki/Opcode" title="Opcode">opcode</a> fields which specifies the basic instruction type (such as arithmetic, logical, <a href="/wiki/Branch_(computer_science)" title="Branch (computer science)">jump</a>, etc.) and the actual operation (such as add or compare) and other fields that may give the type of the <a href="/wiki/Operand" title="Operand">operand</a>(s), the <a href="/wiki/Addressing_mode" title="Addressing mode">addressing mode</a>(s), the addressing offset(s) or index, or the actual value itself (such constant operands contained in an instruction are called <i>immediates</i>).<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup></p>/n<p>Not all machines or individual instructions have explicit operands. An <a href="/wiki/Accumulator_machine" class="mw-redirect" title="Accumulator machine">accumulator machine</a> has a combined left operand and result in an implicit accumulator for most arithmetic instructions. Other architectures (such as 8086 and the x86-family) have accumulator versions of common instructions, with the accumulator regarded as one of the general registers by longer instructions. A <a href="/wiki/Stack_machine" title="Stack machine">stack machine</a> has most or all of its operands on an implicit stack. Special purpose instructions also often lack explicit operands (CPUID in the x86 architecture writes values into four implicit destination registers, for instance). This distinction between explicit and implicit operands is important in code generators, especially in the register allocation and live range tracking parts. A good code optimizer can track implicit as well as explicit operands which may allow more frequent <a href="/wiki/Constant_propagation" class="mw-redirect" title="Constant propagation">constant propagation</a>, <a href="/wiki/Constant_folding" title="Constant folding">constant folding</a> of registers (a register assigned the result of a constant expression freed up by replacing it by that constant) and other code enhancements.</p>/n<h2><span class="mw-headline" id="Programs">Programs</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Machine_code&amp;action=edit&amp;section=2" title="Edit section: Programs">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>A computer program is a sequence of instructions that are executed by a CPU. While simple processors execute instructions one after another, <a href="/wiki/Superscalar" class="mw-redirect" title="Superscalar">superscalar</a> processors are capable of executing several instructions at once.</p>/n<p><a href="/wiki/Program_flow" class="mw-redirect" title="Program flow">Program flow</a> may be influenced by special 'jump' instructions that transfer execution to an instruction other than the numerically following one. <a href="/wiki/Conditional_branch" class="mw-redirect" title="Conditional branch">Conditional jumps</a> are taken (execution continues at another address) or not (execution continues at the next instruction) depending on some condition.</p>/n<h2><span class="mw-headline" id="Assembly_languages">Assembly languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Machine_code&amp;action=edit&amp;section=3" title="Edit section: Assembly languages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Assembly_language" title="Assembly language">Assembly language</a></div>/n<p>A much more readable rendition of machine language, called <a href="/wiki/Assembly_language" title="Assembly language">assembly language</a>, uses <a href="/wiki/Assembly_language#Opcode_mnemonics_and_extended_mnemonics" title="Assembly language">mnemonic codes</a> to refer to machine code instructions, rather than using the instructions' numeric values directly. For example, on the <a href="/wiki/Zilog_Z80" title="Zilog Z80">Zilog Z80</a> processor, the machine code <code>00000101</code>, which causes the CPU to decrement the <code>B</code> <a href="/wiki/Processor_register" title="Processor register">processor register</a>, would be represented in assembly language as <code>DEC B</code>.</p>/n<h2><span class="mw-headline" id="Example">Example</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Machine_code&amp;action=edit&amp;section=4" title="Edit section: Example">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The <a href="/wiki/MIPS_architecture" title="MIPS architecture">MIPS architecture</a> provides a specific example for a machine code whose instructions are always 32 bits long. The general type of instruction is given by the <i>op</i> (operation) field, the highest 6 bits. J-type (jump) and I-type (immediate) instructions are fully specified by <i>op</i>. R-type (register) instructions include an additional field <i>funct</i> to determine the exact operation. The fields used in these types are:</p>/n<pre>/n   6      5     5     5     5      6 bits/n[  op  |  rs |  rt |  rd |shamt| funct]  R-type/n[  op  |  rs |  rt | address/immediate]  I-type/n[  op  |        target address        ]  J-type/n</pre>/n<p><i>rs</i>, <i>rt</i>, and <i>rd</i> indicate register operands; <i>shamt</i> gives a shift amount; and the <i>address</i> or <i>immediate</i> fields contain an operand directly.</p>/n<p>For example, adding the registers 1 and 2 and placing the result in register 6 is encoded:</p>/n<pre>/n[  op  |  rs |  rt |  rd |shamt| funct]/n    0     1     2     6     0     32     decimal/n 000000 00001 00010 00110 00000 100000   binary/n</pre>/n<p>Load a value into register 8, taken from the memory cell 68 cells after the location listed in register 3:</p>/n<pre>/n[  op  |  rs |  rt | address/immediate]/n   35     3     8           68           decimal/n 100011 00011 01000 00000 00001 000100   binary/n</pre>/n<p>Jumping to the address 1024:</p>/n<pre>/n[  op  |        target address        ]/n    2                 1024               decimal/n 000010 00000 00000 00000 10000 000000   binary/n</pre>/n<h2><span class="mw-headline" id="Relationship_to_microcode">Relationship to microcode</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Machine_code&amp;action=edit&amp;section=5" title="Edit section: Relationship to microcode">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>In some <a href="/wiki/Computer_architecture" title="Computer architecture">computer architectures</a>, the machine code is implemented by an even more fundamental underlying layer called <a href="/wiki/Microcode" title="Microcode">microcode</a>, providing a common machine language interface across a line or family of different models of computer with widely different underlying <a href="/wiki/Dataflow" title="Dataflow">dataflows</a>. This is done to facilitate <a href="/wiki/Porting" title="Porting">porting</a> of machine language programs between different models. An example of this use is the IBM <a href="/wiki/System/360" class="mw-redirect" title="System/360">System/360</a> family of computers and their successors. With dataflow path widths of 8 bits to 64 bits and beyond, they nevertheless present a common architecture at the machine language level across the entire line.</p>/n<p>Using microcode to implement an <a href="/wiki/Emulator" title="Emulator">emulator</a> enables the computer to present the architecture of an entirely different computer. The System/360 line used this to allow porting programs from earlier IBM machines to the new family of computers, e.g. an <a href="/wiki/IBM_1400_series" title="IBM 1400 series">IBM 1401/1440/1460</a> emulator on the IBM S/360 model 40.</p>/n<h2><span class="mw-headline" id="Relationship_to_bytecode">Relationship to bytecode</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Machine_code&amp;action=edit&amp;section=6" title="Edit section: Relationship to bytecode">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Machine code is generally different than <a href="/wiki/Bytecode" title="Bytecode">bytecode</a> (also known as p-code), which is either executed by an interpreter or itself compiled into machine code for faster (direct) execution. An exception is when a processor is designed to use a particular bytecode directly as its machine code, such as is the case with <a href="/wiki/Java_processor" title="Java processor">Java processors</a>.</p>/n<p>Machine code and assembly code are sometimes called <i><a href="/wiki/Native_(computing)" title="Native (computing)">native</a> code</i> when referring to platform-dependent parts of language features or libraries.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup></p>/n<h2><span class="mw-headline" id="Storing_in_memory">Storing in memory</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Machine_code&amp;action=edit&amp;section=7" title="Edit section: Storing in memory">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The <a href="/wiki/Harvard_architecture" title="Harvard architecture">Harvard architecture</a> is a computer architecture with physically separate storage and signal pathways for the code (instructions) and <a href="/wiki/Data_(computing)" title="Data (computing)">data</a>. Today, most processors implement such separate signal pathways for performance reasons but actually implement a <a href="/wiki/Modified_Harvard_architecture" title="Modified Harvard architecture">Modified Harvard architecture</a>,<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (March 2014)">citation needed</span></a></i>]</sup> so they can support tasks like loading an <a href="/wiki/Executable" title="Executable">executable</a> program from <a href="/wiki/Data_storage_device" class="mw-redirect" title="Data storage device">disk storage</a> as data and then executing it. Harvard architecture is contrasted to the <a href="/wiki/Von_Neumann_architecture" title="Von Neumann architecture">Von Neumann architecture</a>, where data and code are stored in the same memory which is read by the processor allowing the computer to execute commands.</p>/n<p>From the point of view of a <a href="/wiki/Process_(computing)" title="Process (computing)">process</a>, the <i>code space</i> is the part of its <a href="/wiki/Virtual_address_space" title="Virtual address space">address space</a> where the code in execution is stored. In <a href="/wiki/Computer_multitasking" title="Computer multitasking">multitasking</a> systems this comprises the program's <a href="/wiki/Code_segment" title="Code segment">code segment</a> and usually <a href="/wiki/Shared_libraries" class="mw-redirect" title="Shared libraries">shared libraries</a>. In <a href="/wiki/Thread_(computing)" title="Thread (computing)">multi-threading</a> environment, different threads of one process share code space along with data space, which reduces the overhead of <a href="/wiki/Context_switching" class="mw-redirect" title="Context switching">context switching</a> considerably as compared to process switching.</p>/n<h2><span class="mw-headline" id="Readability_by_humans">Readability by humans</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Machine_code&amp;action=edit&amp;section=8" title="Edit section: Readability by humans">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Pamela Samuelson wrote that machine code is so unreadable that the <a href="/wiki/United_States_Copyright_Office" title="United States Copyright Office">United States Copyright Office</a> cannot identify whether a particular encoded program is an original work of authorship;<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> however, the US Copyright Office <i>does</i> allow for copyright registration of computer programs<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> and a program's machine code can sometimes be <a href="/wiki/Decompiler" title="Decompiler">decompiled</a> in order to make its functioning more easily understandable to humans.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup></p>/n<p>Cognitive science professor <a href="/wiki/Douglas_Hofstadter" title="Douglas Hofstadter">Douglas Hofstadter</a> has compared machine code to <a href="/wiki/Genetic_code" title="Genetic code">genetic code</a>, saying that "Looking at a program written in machine language is vaguely comparable to looking at a <a href="/wiki/DNA" title="DNA">DNA</a> molecule atom by atom."<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup></p>/n<h2><span
