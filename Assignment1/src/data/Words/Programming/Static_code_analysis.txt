analysis (computer science)">analysis of computer software</a> that is performed without actually executing programs (analysis performed on executing programs is known as <a href="/wiki/Dynamic_program_analysis" title="Dynamic program analysis">dynamic analysis</a>).<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> In most cases the analysis is performed on some version of the <a href="/wiki/Source_code" title="Source code">source code</a>, and in the other cases, some form of the <a href="/wiki/Object_code" title="Object code">object code</a>.</p>/n<p>The term is usually applied to the analysis performed by an <a href="/wiki/List_of_tools_for_static_code_analysis" title="List of tools for static code analysis">automated tool</a>, with human analysis being called program understanding, <a href="/wiki/Program_comprehension" title="Program comprehension">program comprehension</a>, or <a href="/wiki/Code_review" title="Code review">code review</a>. <a href="/wiki/Software_inspection" title="Software inspection">Software inspections</a> and <a href="/wiki/Software_walkthrough" title="Software walkthrough">software walkthroughs</a> are also used in the latter case.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toctitle">/n<h2>Contents</h2>/n</div>/n<ul>/n<li class="toclevel-1 tocsection-1"><a class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Static_program_analysis&amp;action=edit&amp;section=1" title="Edit section: Rationale">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The sophistication of the analysis performed by tools varies from those that only consider the behaviour of individual statements and declarations, to those that include the complete <a href="/wiki/Source_code" title="Source code">source code</a> of a program in their analysis. The uses of the information obtained from the analysis vary from highlighting possible coding errors (e.g., the <a href="/wiki/Lint_programming_tool" class="mw-redirect" title="Lint programming tool">lint</a> tool) to <a href="/wiki/Formal_methods" title="Formal methods">formal methods</a> that mathematically prove properties about a given program (e.g., its behaviour matches that of its specification).</p>/n<p><a href="/wiki/Software_metric" title="Software metric">Software metrics</a> and <a href="/wiki/Reverse_engineering" title="Reverse engineering">reverse engineering</a> can be described as forms of static analysis. Deriving software metrics and static analysis are increasingly deployed together, especially in creation of embedded systems, by defining so-called <i>software quality objectives</i>.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup></p>/n<p>A growing commercial use of static analysis is in the verification of properties of software used in <a href="/wiki/Safety-critical" class="mw-redirect" title="Safety-critical">safety-critical</a> computer systems and locating potentially <a href="/wiki/Vulnerability_(computing)" title="Vulnerability (computing)">vulnerable</a> code.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup> For example, the following industries have identified the use of static code analysis as a means of improving the quality of increasingly sophisticated and complex software:</p>/n<ol>/n<li><a href="/wiki/Medical_software" title="Medical software">Medical software</a>: The U.S. <a href="/wiki/Food_and_Drug_Administration" title="Food and Drug Administration">Food and Drug Administration</a> (FDA) has identified the use of static analysis for medical devices.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup></li>/n<li>Nuclear software: In the UK the Office for Nuclear Regulation (ONR) recommends the use of static analysis on <a href="/wiki/Reactor_protection_system" title="Reactor protection system">reactor protection systems</a>.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup></li>/n<li>Aviation software (in combination with <a href="/wiki/Dynamic_program_analysis" title="Dynamic program analysis">dynamic analysis</a>)<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup></li>/n</ol>/n<p>A study in 2012 by VDC Research reports that 28.7% of the embedded software engineers surveyed currently use static analysis tools and 39.7% expect to use them within 2 years.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup> A study from 2010 found that 60% of the interviewed developers in European research projects made at least use of their basic IDE built-in static analyzers. However, only about 10% employed an additional other (and perhaps more advanced) analysis tool.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup></p>/n<p>In the application security industry the name <i>Static Application Security Testing</i> (SAST) is also used. Actually, SAST is an important part of Security Development Lifecycles (SDLs) such as the SDL defined by Microsoft <sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> and a common practice in software companies.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup></p>/n<h2><span class="mw-headline" id="Tool_types">Tool types</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Static_program_analysis&amp;action=edit&amp;section=2" title="Edit section: Tool types">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The OMG (<a href="/wiki/Object_Management_Group" title="Object Management Group">Object Management Group</a>) published a study regarding the types of software analysis required for <a href="/wiki/Software_quality" title="Software quality">software quality</a> measurement and assessment. This document on "How to Deliver Resilient, Secure, Efficient, and Easily Changed IT Systems in Line with CISQ Recommendations" describes three levels of software analysis.<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup></p>/n<dl>/n<dt>Unit Level</dt>/n<dd>Analysis that takes place within a specific program or subroutine, without connecting to the context of that program.</dd>/n<dt>Technology Level</dt>/n<dd>Analysis that takes into account interactions between unit programs to get a more holistic and semantic view of the overall program in order to find issues and avoid obvious false positives.</dd>/n<dt>System Level</dt>/n<dd>Analysis that takes into account the interactions between unit programs, but without being limited to one specific technology or programming language.</dd>/n</dl>/n<p>A further level of software analysis can be defined.</p>/n<dl>/n<dt>Mission/Business Level</dt>/n<dd>Analysis that takes into account the business/mission layer terms, rules and processes that are implemented within the software system for its operation as part of enterprise or program/mission layer activities. These elements are implemented without being limited to one specific technology or programming language and in many cases are distributed across multiple languages, but are statically extracted and analyzed for system understanding for mission assurance.</dd>/n</dl>/n<h2><span class="mw-headline" id="Formal_methods">Formal methods</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Static_program_analysis&amp;action=edit&amp;section=3" title="Edit section: Formal methods">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Formal_methods" title="Formal methods">Formal methods</a></div>/n<p>Formal methods is the term applied to the analysis of <a href="/wiki/Software" title="Software">software</a> (and <a href="/wiki/Computer_hardware" title="Computer hardware">computer hardware</a>) whose results are obtained purely through the use of rigorous mathematical methods. The mathematical techniques used include <a href="/wiki/Denotational_semantics" title="Denotational semantics">denotational semantics</a>, <a href="/wiki/Axiomatic_semantics" title="Axiomatic semantics">axiomatic semantics</a>, <a href="/wiki/Operational_semantics" title="Operational semantics">operational semantics</a>, and <a href="/wiki/Abstract_interpretation" title="Abstract interpretation">abstract interpretation</a>.</p>/n<p>By a straightforward reduction to the <a href="/wiki/Halting_problem" title="Halting problem">halting problem</a>, it is possible to prove that (for any <a href="/wiki/Turing_complete" class="mw-redirect" title="Turing complete">Turing complete</a> language), finding all possible run-time errors in an arbitrary program (or more generally any kind of violation of a specification on the final result of a program) is <a href="/wiki/Decision_problem" title="Decision problem">undecidable</a>: there is no mechanical method that can always answer truthfully whether an arbitrary program may or may not exhibit runtime errors. This result dates from the works of <a href="/wiki/Alonzo_Church" title="Alonzo Church">Church</a>, <a href="/wiki/Kurt_G%C3%B6del" title="Kurt Gödel">Gödel</a> and <a href="/wiki/Alan_Turing" title="Alan Turing">Turing</a> in the 1930s (see: <a href="/wiki/Halting_problem" title="Halting problem">Halting problem</a> and <a href="/wiki/Rice%27s_theorem" title="Rice's theorem">Rice's theorem</a>). As with many undecidable questions, one can still attempt to give useful approximate solutions.</p>/n<p>Some of the implementation techniques of formal static analysis include:<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[12]</a></sup></p>/n<ul>/n<li><a href="/wiki/Abstract_interpretation" title="Abstract interpretation">Abstract interpretation</a>, to model the effect that every statement has on the state of an abstract machine (i.e., it 'executes' the software based on the mathematical properties of each statement and declaration). This abstract machine over-approximates the behaviours of the system: the abstract system is thus made simpler to analyze, at the expense of <i>incompleteness</i> (not every property true of the original system is true of the abstract system). If properly done, though, abstract interpretation is <i>sound</i> (every property true of the abstract system can be mapped to a true property of the original system).<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">[13]</a></sup> The <a href="/wiki/Frama-C" title="Frama-C">Frama-C</a> value analysis plugin and <a href="/wiki/Polyspace" title="Polyspace">Polyspace</a> heavily rely on abstract interpretation.</li>/n<li><a href="/wiki/Data_flow_analysis" class="mw-redirect" title="Data flow analysis">Data-flow analysis</a>, a lattice-based technique for gathering information about the possible set of values;</li>/n<li><a href="/wiki/Hoare_logic" title="Hoare logic">Hoare logic</a>, a <a href="/wiki/Formal_system" title="Formal system">formal system</a> with a set of logical rules for reasoning rigorously about the <a href="/wiki/Correctness_of_computer_programs" class="mw-redirect" title="Correctness of computer programs">correctness of computer programs</a>. There is tool support for some programming languages (e.g., the <a href="/wiki/SPARK_programming_language" class="mw-redirect" title="SPARK programming language">SPARK programming language</a> (a subset of <a href="/wiki/Ada_(programming_language)" title="Ada (programming language)">Ada</a>) and the <a href="/wiki/Java_Modeling_Language" title="Java Modeling Language">Java Modeling Language</a>—JML—using <a href="/wiki/ESC/Java" title="ESC/Java">ESC/Java</a> and <a href="/wiki/ESC/Java2" class="mw-redirect" title="ESC/Java2">ESC/Java2</a>, Frama-C WP (<a href="/wiki/Weakest_precondition" class="mw-redirect" title="Weakest precondition">weakest precondition</a>) plugin for the C language extended with ACSL (<a href="/wiki/ANSI/ISO_C_Specification_Language" title="ANSI/ISO C Specification Language">ANSI/ISO C Specification Language</a>) ).</li>/n<li><a href="/wiki/Model_checking" title="Model checking">Model checking</a>, considers systems that have <a href="/wiki/Finite-state_machine" title="Finite-state machine">finite state</a> or may be reduced to finite state by <a href="/wiki/Abstraction_(computer_science)" class="mw-redirect" title="Abstraction (computer science)">abstraction</a>;</li>/n<li><a href="/wiki/Symbolic_execution" title="Symbolic execution">Symbolic execution</a>, as used to derive mathematical expressions representing the value of mutated variables at particular points in the code.</li>/n</ul>/n<h2><span class="mw-headline" id="See_also">See
