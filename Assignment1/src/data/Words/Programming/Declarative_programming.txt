template">e</abbr></a></li>/n</ul>/n</div>/n</td>/n</tr>/n</table>/n<p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, <b>declarative programming</b> is a <a href="/wiki/Programming_paradigm" title="Programming paradigm">programming paradigm</a>—a style of building the structure and elements of computer programs—that expresses the logic of a <a href="/wiki/Computation" title="Computation">computation</a> without describing its <a href="/wiki/Control_flow" title="Control flow">control flow</a>.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup></p>/n<p>Many languages that apply this style attempt to minimize or eliminate <a href="/wiki/Side_effect_(computer_science)" title="Side effect (computer science)">side effects</a> by describing <i>what</i> the program must accomplish in terms of the <a href="/wiki/Problem_domain" title="Problem domain">problem domain</a>, rather than describe <i>how</i> to accomplish it as a sequence of the programming language <a href="/wiki/Language_primitive" title="Language primitive">primitives</a><sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> (the <i>how</i> being left up to the language's <a href="/wiki/Programming_language_implementation" title="Programming language implementation">implementation</a>). This is in contrast with <a href="/wiki/Imperative_programming" title="Imperative programming">imperative programming</a>, which implements <a href="/wiki/Algorithm" title="Algorithm">algorithms</a> in explicit steps.</p>/n<p>Declarative programming often considers <a href="/wiki/Program_(machine)" title="Program (machine)">programs</a> as theories of a <a href="/wiki/Formal_logic" class="mw-redirect" title="Formal logic">formal logic</a>, and computations as deductions in that logic space. Declarative programming may greatly simplify writing <a href="/wiki/Parallel_computing" title="Parallel computing">parallel programs</a>.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup></p>/n<p>Common declarative languages include those of <a href="/wiki/Database_query_language" class="mw-redirect" title="Database query language">database query languages</a> (e.g., <a href="/wiki/SQL" title="SQL">SQL</a>, <a href="/wiki/XQuery" title="XQuery">XQuery</a>), <a href="/wiki/Regular_expression" title="Regular expression">regular expressions</a>, <a href="/wiki/Logic_programming" title="Logic programming">logic programming</a>, <a href="/wiki/Functional_programming" title="Functional programming">functional programming</a>, and <a href="/wiki/Configuration_management" title="Configuration management">configuration management</a> systems.</p>/n<p></p>/n<div id="toc" class="toc">/n<div id="Definition">Definition</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Declarative_programming&amp;action=edit&amp;section=1" title="Edit section: Definition">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Declarative programming is often defined as any style of programming that is not <a href="/wiki/Imperative_programming" title="Imperative programming">imperative</a>. A number of other common definitions exist that attempt to give the term a definition other than simply contrasting it with imperative programming. For example:</p>/n<ul>/n<li>A high-level program that describes what a computation should perform.</li>/n<li>Any programming language that lacks <a href="/wiki/Side_effect_(computer_science)" title="Side effect (computer science)">side effects</a> (or more specifically, is <a href="/wiki/Referential_transparency_(computer_science)" class="mw-redirect" title="Referential transparency (computer science)">referentially transparent</a>)</li>/n<li>A language with a clear correspondence to <a href="/wiki/Mathematical_logic" title="Mathematical logic">mathematical logic</a>.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup></li>/n</ul>/n<p>These definitions overlap substantially.</p>/n<p>Declarative programming contrasts with imperative and <a href="/wiki/Procedural_programming" title="Procedural programming">procedural programming</a>. Declarative programming is a non-imperative style of programming in which programs describe their desired results without explicitly listing commands or steps that must be performed. <a href="/wiki/Functional_programming" title="Functional programming">Functional</a> and <a href="/wiki/Logical_programming" class="mw-redirect" title="Logical programming">logical programming</a> languages are characterized by a declarative programming style. In <a href="/wiki/Logic_programming" title="Logic programming">logical programming languages</a>, programs consist of logical statements, and the program executes by searching for proofs of the statements.</p>/n<p>In a <a href="/wiki/Pure_functional_language" class="mw-redirect" title="Pure functional language">pure functional language</a>, such as <a href="/wiki/Haskell_(programming_language)" title="Haskell (programming language)">Haskell</a>, all functions are <a href="/wiki/Pure_function" title="Pure function">without side effects</a>, and state changes are only represented as functions that transform the state, which is explicitly represented as a <a href="/wiki/First-class_citizen" title="First-class citizen">first class</a> object in the program. Although pure functional languages are non-imperative, they often provide a facility for describing the effect of a function as a series of steps. Other functional languages, such as <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">Lisp</a>, <a href="/wiki/OCaml" title="OCaml">OCaml</a> and <a href="/wiki/Erlang_(programming_language)" title="Erlang (programming language)">Erlang</a>, support a mixture of procedural and functional programming.</p>/n<p>Some <a href="/wiki/Logic_programming" title="Logic programming">logical programming languages</a>, such as <a href="/wiki/Prolog" title="Prolog">Prolog</a>, and <a href="/wiki/Database_query_language" class="mw-redirect" title="Database query language">database query languages</a>, such as <a href="/wiki/SQL" title="SQL">SQL</a>, while declarative in principle, also support a procedural style of programming.</p>/n<h2><span class="mw-headline" id="Subparadigms">Subparadigms</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Declarative_programming&amp;action=edit&amp;section=2" title="Edit section: Subparadigms">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Declarative programming is an <a href="/wiki/Umbrella_term" title="Umbrella term">umbrella term</a> that includes a number of better-known <a href="/wiki/Programming_paradigm" title="Programming paradigm">programming paradigms</a>.</p>/n<h3><span class="mw-headline" id="Constraint_programming">Constraint programming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Declarative_programming&amp;action=edit&amp;section=3" title="Edit section: Constraint programming">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p><a href="/wiki/Constraint_programming" title="Constraint programming">Constraint programming</a> states relations between variables in the form of constraints that specify the properties of the target solution. The set of constraints is <a href="/wiki/Solver_(computer_science)" class="mw-redirect" title="Solver (computer science)">solved</a> by giving a value to each variable so that the solution is consistent with the maximum number of constraints. Constraint programming often complements other paradigms: functional, logical, or even imperative programming.</p>/n<h3><span class="mw-headline" id="Domain-specific_languages">Domain-specific languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Declarative_programming&amp;action=edit&amp;section=4" title="Edit section: Domain-specific languages">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Well-known examples of declarative <a href="/wiki/Domain-specific_language" title="Domain-specific language">domain-specific languages</a> (DSLs) include the <a href="/wiki/Yacc" title="Yacc">yacc</a> parser generator input language, <a href="/wiki/QML" title="QML">QML</a>, the <a href="/wiki/Make_(software)" title="Make (software)">Make</a> build specification language, <a href="/wiki/Puppet_(software)" title="Puppet (software)">Puppet</a>'s configuration management language, <a href="/wiki/Regular_expression" title="Regular expression">regular expressions</a>, and a subset of <a href="/wiki/SQL" title="SQL">SQL</a> (SELECT queries, for example). DSLs have the advantage of being useful while not necessarily needing to be <a href="/wiki/Turing-complete" class="mw-redirect" title="Turing-complete">Turing-complete</a>, which makes it easier for a language to be purely declarative.</p>/n<p>Many markup languages such as <a href="/wiki/HTML" title="HTML">HTML</a>, <a href="/wiki/MXML" title="MXML">MXML</a>, <a href="/wiki/XAML" class="mw-redirect" title="XAML">XAML</a>, <a href="/wiki/XSLT" title="XSLT">XSLT</a> or other <a href="/wiki/User-interface_markup_language" class="mw-redirect" title="User-interface markup language">user-interface markup languages</a> are often declarative. HTML, for example, only describes what should appear on a webpage - it specifies neither <a href="/wiki/Control_flow" title="Control flow">control flow</a> for rendering a page nor the page's possible <a href="/wiki/Interaction" title="Interaction">interactions</a> with a user.</p>/n<p>As of 2013<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=Declarative_programming&amp;action=edit">[update]</a></sup> some software systems combine traditional user-interface markup languages (such as HTML) with declarative markup that defines what (but not how) the back-end server systems should do to support the declared interface. Such systems, typically using a domain-specific <a href="/wiki/XML" title="XML">XML</a> <a href="/wiki/XML_namespace" title="XML namespace">namespace</a>, may include abstractions of <a href="/wiki/SQL" title="SQL">SQL</a> database syntax or parameterised calls to web services using <a href="/wiki/Representational_state_transfer" title="Representational state transfer">representational state transfer</a> (REST) and <a href="/wiki/SOAP" title="SOAP">SOAP</a>.</p>/n<h3><span class="mw-headline" id="Hybrid_languages">Hybrid languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Declarative_programming&amp;action=edit&amp;section=5" title="Edit section: Hybrid languages">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/Multi-paradigm_programming_language" class="mw-redirect" title="Multi-paradigm programming language">Multi-paradigm programming language</a></div>/n<p><a href="/wiki/Make_(software)" title="Make (software)">Makefiles</a>, for example, specify dependencies in a declarative fashion,<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> but include an imperative list of actions to take as well. Similarly, <a href="/wiki/Yacc" title="Yacc">yacc</a> specifies a context free grammar declaratively, but includes code snippets from a host language, which is usually imperative (such as <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a>).</p>/n<h3><span class="mw-headline" id="Logic_programming">Logic programming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Declarative_programming&amp;action=edit&amp;section=6" title="Edit section: Logic programming">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p><a href="/wiki/Logic_programming" title="Logic programming">Logic programming</a> languages such as <a href="/wiki/Prolog" title="Prolog">Prolog</a> state and query relations. The specifics of <i>how</i> these queries are answered is up to the implementation and its theorem prover, but typically take the form of some sort of <a href="/wiki/Unification_(computing)" class="mw-redirect" title="Unification (computing)">unification</a>. Like functional programming, many logic programming languages permit side effects, and as a result are not strictly declarative.</p>/n<h3><span class="mw-headline" id="Modeling">Modeling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Declarative_programming&amp;action=edit&amp;section=7" title="Edit section: Modeling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Mathematical_model" title="Mathematical model">Mathematical model</a></div>/n<p>Models, or mathematical representations, of physical systems may be implemented in computer code that is declarative. The code contains a number of equations, not imperative assignments, that describe ("declare") the behavioral relationships. When a model is expressed in this formalism, a computer is able to perform algebraic manipulations to best formulate the solution algorithm. The mathematical causality is typically imposed at the boundaries of the physical system, while the behavioral description of the system itself is declarative or acausal. Declarative <a href="/wiki/Modeling_language" title="Modeling language">modeling languages</a> and environments include <a href="/wiki/Analytica_(software)" title="Analytica (software)">Analytica</a>, <a href="/wiki/Modelica" title="Modelica">Modelica</a> and <a href="/w/index.php?title=Simile_(computing)&amp;action=edit&amp;redlink=1" class="new" title="Simile (computing) (page does not exist)">Simile</a>.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup></p>/n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span
