class="mw-parser-output"><p>In <a href="/wiki/Mathematics" title="Mathematics">mathematics</a>, <a href="/wiki/Logic" title="Logic">logic</a>, and <a href="/wiki/Computer_science" title="Computer science">computer science</a>, a <b>type theory</b> is any of a class of <a href="/wiki/Formal_system" title="Formal system">formal systems</a>, some of which can serve as alternatives to <a href="/wiki/Set_theory" title="Set theory">set theory</a> as a <a href="/wiki/Foundations_of_mathematics" title="Foundations of mathematics">foundation for all mathematics</a>. In type theory, every "term" has a "type" and operations are restricted to terms of a certain type.</p>/n<p>Type theory is closely related to (and in some cases overlaps with) <a href="/wiki/Type_system" title="Type system">type systems</a>, which are a <a href="/wiki/Programming_language" title="Programming language">programming language</a> feature used to reduce <a href="/wiki/Bug_(computer_programming)" class="mw-redirect" title="Bug (computer programming)">bugs</a>. Type theory was created to avoid paradoxes in a variety of formal <a href="/wiki/Mathematical_logic" title="Mathematical logic">logics</a> and <a href="/wiki/Rewrite_system" class="mw-redirect" title="Rewrite system">rewrite systems</a>.</p>/n<p>Two well-known type theories that can serve as mathematical foundations are <a href="/wiki/Alonzo_Church" title="Alonzo Church">Alonzo Church</a>'s <a href="/wiki/Typed_lambda_calculus" title="Typed lambda calculus">typed λ-calculus</a> and <a href="/wiki/Per_Martin-L%C3%B6f" title="Per Martin-Löf">Per Martin-Löf</a>'s <a href="/wiki/Intuitionistic_type_theory" title="Intuitionistic type theory">intuitionistic type theory</a>.</p>/n<p></p>/n<div id="toc" class="toc">/n<div links</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_theory&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/History_of_type_theory" title="History of type theory">History of type theory</a></div>/n<p>Between 1902 and 1908 <a href="/wiki/Bertrand_Russell" title="Bertrand Russell">Bertrand Russell</a> proposed various "theories of type" in response to his discovery that <a href="/wiki/Gottlob_Frege" title="Gottlob Frege">Gottlob Frege</a>'s version of <a href="/wiki/Naive_set_theory" title="Naive set theory">naive set theory</a> was afflicted with <a href="/wiki/Russell%27s_paradox" title="Russell's paradox">Russell's paradox</a>. By 1908 Russell arrived at a "ramified" theory of types together with an "<a href="/wiki/Axiom_of_reducibility" title="Axiom of reducibility">axiom of reducibility</a>" both of which featured prominently in <a href="/wiki/Alfred_North_Whitehead" title="Alfred North Whitehead">Whitehead</a> and <a href="/wiki/Bertrand_Russell" title="Bertrand Russell">Russell</a>'s <i><a href="/wiki/Principia_Mathematica" title="Principia Mathematica">Principia Mathematica</a></i> published between 1910 and 1913. They attempt to avoid Russell's paradox by first creating a hierarchy of types, then assigning each mathematical (and possibly other) entity to a type. Objects of a given type are built exclusively from objects of preceding types (those lower in the hierarchy), thus preventing loops. In the 1920s, <a href="/wiki/Leon_Chwistek" title="Leon Chwistek">Leon Chwistek</a> and <a href="/wiki/Frank_P._Ramsey" title="Frank P. Ramsey">Frank P. Ramsey</a> proposed a simpler theory, now known as the "theory of simple types" or "simple type theory", that collapsed the complicated hierarchy of the "ramified theory" and did not require the axiom of reducibility.</p>/n<p>The common usage of "type theory" is when those types are used with a <a href="/wiki/Term_rewrite_system" class="mw-redirect" title="Term rewrite system">term rewrite system</a>. The most famous early example is <a href="/wiki/Alonzo_Church" title="Alonzo Church">Alonzo Church</a>'s <a href="/wiki/Simply_typed_lambda_calculus" title="Simply typed lambda calculus">simply typed lambda calculus</a>. Church's theory of types<sup id="cite_ref-church_1-0" class="reference"><a href="#cite_note-church-1">[1]</a></sup> helped the formal system avoid the <a href="/wiki/Kleene%E2%80%93Rosser_paradox" title="Kleene–Rosser paradox">Kleene–Rosser paradox</a> that afflicted the original untyped lambda calculus. Church demonstrated that it could serve as a foundation of mathematics and it was referred to as a <a href="/wiki/Higher-order_logic" title="Higher-order logic">higher-order logic</a>.</p>/n<p>Some other type theories include <a href="/wiki/Per_Martin-L%C3%B6f" title="Per Martin-Löf">Per Martin-Löf</a>'s <a href="/wiki/Intuitionistic_type_theory" title="Intuitionistic type theory">intuitionistic type theory</a>, which has been the foundation used in some areas of <a href="/wiki/Constructivism_(mathematics)" title="Constructivism (mathematics)">constructive mathematics</a> and for the proof assistant <a href="/wiki/Agda_(programming_language)" title="Agda (programming language)">Agda</a>. <a href="/wiki/Thierry_Coquand" title="Thierry Coquand">Thierry Coquand</a>'s <a href="/wiki/Calculus_of_constructions" title="Calculus of constructions">calculus of constructions</a> and its derivatives are the foundation used by <a href="/wiki/Coq" title="Coq">Coq</a> and others. The field is an area of active research, as demonstrated by <a href="/wiki/Homotopy_type_theory" title="Homotopy type theory">homotopy type theory</a>.</p>/n<h2><span class="mw-headline" id="Basic_concepts">Basic concepts</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_theory&amp;action=edit&amp;section=2" title="Edit section: Basic concepts">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>In a system of type theory, each <b><a href="/wiki/Term_(logic)" title="Term (logic)">term</a></b> has a <b>type</b> and operations are restricted to terms of a certain type. It requires a separate system like <a href="/wiki/Predicate_logic" class="mw-redirect" title="Predicate logic">predicate logic</a> underneath it. In type theory, concepts like "and" and "or" can be encoded as types in the type theory itself.</li>/n<li>In set theory, an element can belong to multiple sets, either to a subset or to a superset. In type theory, terms (generally) belong to only one type. (Where a subset would be used, type theory creates a new type, called a <a href="/wiki/Intuitionistic_type_theory#.CE.A3-types" title="Intuitionistic type theory">dependent sum type</a>, with new terms. Union is similarly achieved by a new <a href="/wiki/Tagged_union" title="Tagged union">sum type</a> and new terms.)</li>/n<li>In set theory, sets can contain unrelated elements, e.g., apples and real numbers. In type theory, types that combine unrelated types do so by creating new terms.</li>/n<li>Set theory usually encodes numbers as sets. (0 is the empty set, 1 is a set containing the empty set, etc.) Type theory can encode numbers as functions using <a href="/wiki/Church_encoding" title="Church encoding">Church encoding</a> or more naturally as <a href="/wiki/Intuitionistic_type_theory#Inductive_types" title="Intuitionistic type theory">inductive types</a>, which are types with well-behaved constant terms.</li>/n<li>Set theory allows <a href="/wiki/Set-builder_notation" title="Set-builder notation">set builder notation</a>.</li>/n<li>Type theory has a simple connection to constructive mathematics through the <a href="/wiki/BHK_interpretation" class="mw-redirect" title="BHK interpretation">BHK interpretation</a>.</li>/n</ul>/n<h2><span class="mw-headline" id="Optional_features">Optional features</span><span Some systems build them out of functions using <a href="/wiki/Church_encoding" title="Church encoding">Church encoding</a>. Other systems have <b>inductive types</b>: a set of base types and a set of <a href="/wiki/Type_constructor" title="Type constructor">type constructors</a> that generate types with well-behaved properties. For example, <a href="/wiki/Structural_induction" title="Structural induction">certain recursive functions</a> called on inductive types are guaranteed to terminate.</p>/n<p><b>Coinductive type</b> are infinite data types created by giving a function that generates the next element(s). See <a href="/wiki/Coinduction" title="Coinduction">Coinduction</a> and <a href="/wiki/Corecursion" title="Corecursion">Corecursion</a>.</p>/n<p><b><a href="/w/index.php?title=Induction-Induction_(type_theory)&amp;action=edit&amp;redlink=1" class="new" title="Induction-Induction (type theory) (page does not exist)">Induction induction</a></b> is a feature for declaring an inductive type and a family of types that depends on the inductive type.</p>/n<p><b><a href="/wiki/Induction-recursion_(type_theory)" class="mw-redirect" title="Induction-recursion (type theory)">Induction recursion</a></b> allows a wider range of well-behaved types, allowing the type and recursive functions operating on it to be defined at the same time.</p>/n<h3><span class="mw-headline" id="Universe_types">Universe types</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_theory&amp;action=edit&amp;section=9" title="Edit section: Universe types">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Types were created to prevent paradoxes, such as <a href="/wiki/Russell%27s_paradox" title="Russell's paradox">Russell's paradox</a>. However, the motives that lead to those paradoxes—being able to say things about all types—still exist. So, many type theories have a "universe type", which contains all <i>other</i> types (and not itself).</p>/n<p>In systems where you might want to say something about universe types, there is a hierarchy of universe types, each containing the one below it in the hierarchy. The hierarchy is defined as being infinite, but statements must only refer to a finite number of universe levels.</p>/n<p>Type universes are particularly tricky in type theory. The initial proposal of intuitionistic type theory suffered from <a href="/wiki/Girard%27s_paradox" class="mw-redirect" title="Girard's paradox">Girard's paradox</a>.</p>/n<h3><span class="mw-headline" id="Computational_component">Computational component</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_theory&amp;action=edit&amp;section=10" title="Edit section: Computational component">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Many systems of type theory, such as the <a href="/wiki/Simply-typed_lambda_calculus" class="mw-redirect" title="Simply-typed lambda calculus">simply-typed lambda calculus</a>, <a href="/wiki/Intuitionistic_type_theory" title="Intuitionistic type theory">intuitionistic type theory</a>, and the <a href="/wiki/Calculus_of_constructions" title="Calculus of constructions">calculus of constructions</a>, are also programming languages. That is, they are said to have a "computational component". The computation is the reduction of terms of the language using <a href="/wiki/Rewriting" title="Rewriting">rewriting rules</a>.</p>/n<p>A system of type theory that has a well-behaved computational component also has a simple connection to constructive mathematics through the <a href="/wiki/BHK_interpretation" class="mw-redirect" title="BHK interpretation">BHK interpretation</a>.</p>/n<p>Non-constructive mathematics in these systems is possible by adding operators on continuations such as <a href="/wiki/Call/cc#Relation_to_non-constructive_logic" class="mw-redirect" title="Call/cc">call with current continuation</a>. Researchers are exploring connections between dependent types (especially the identity type) and <a href="/wiki/Algebraic_topology" title="Algebraic topology">algebraic topology</a> (specifically <a href="/wiki/Homotopy" title="Homotopy">homotopy</a>).</p>/n<h3><span class="mw-headline" id="Proof_assistants">Proof assistants</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_theory&amp;action=edit&amp;section=18" title="Edit section: Proof assistants">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Proof_assistant" title="Proof assistant">Proof assistant</a></div>/n<p>Much of the current research into type theory is driven by <a href="/wiki/Automated_proof_checking" title="Automated proof checking">proof checkers</a>, interactive <a href="/wiki/Proof_assistant" title="Proof assistant">proof assistants</a>, and <a href="/wiki/Automated_theorem_proving" title="Automated theorem proving">automated theorem provers</a>. Most of these systems use a type theory as the mathematical foundation for encoding proofs, which is not surprising, given the close connection between type theory and programming languages:</p>/n<ul>/n<li><a href="/wiki/Logical_framework" title="Logical framework">LF</a> is used by <a href="/wiki/Twelf" title="Twelf">Twelf</a>, often to define other type theories;</li>/n<li>multiple type theories falling under <a href="/wiki/Higher-order_logic" title="Higher-order logic">higher-order logic</a> are used by the <a href="/wiki/HOL_(proof_assistant)" title="HOL (proof assistant)">HOL family of provers</a> and <a href="/wiki/Prototype_Verification_System" title="Prototype Verification System">PVS</a>;</li>/n<li><a href="/wiki/Intuitionistic_type_theory" title="Intuitionistic type theory">intuitionistic type theory</a> is used by <a href="/wiki/Agda_(programming_language)" title="Agda (programming language)">Agda</a> which is both a programming language and proof assistant;</li>/n<li>computational type theory is used by <a href="/wiki/NuPRL" class="mw-redirect" title="NuPRL">NuPRL</a>;</li>/n<li><a href="/wiki/Calculus_of_constructions" title="Calculus of constructions">calculus of constructions</a> and its derivatives are used by <a href="/wiki/Coq" title="Coq">Coq</a> and <a href="/wiki/Matita" title="Matita">Matita</a>.</li>/n</ul>/n<p>Multiple type theories are supported by <a href="/wiki/LEGO_(proof_assistant)" title="LEGO (proof assistant)">LEGO</a> and <a href="/wiki/Isabelle_(proof_assistant)" title="Isabelle (proof assistant)">Isabelle</a>. Isabelle also supports foundations besides type theories, such as <a href="/wiki/Zermelo%E2%80%93Fraenkel_set_theory" title="Zermelo–Fraenkel set theory">ZFC</a>. <a href="/wiki/Mizar_system" title="Mizar system">Mizar</a> is an example of a proof system that only supports set theory.</p>/n<h3><span class="mw-headline" id="Linguistics">Linguistics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_theory&amp;action=edit&amp;section=19" title="Edit section: Linguistics">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Type theory is also widely in use in <a href="/wiki/Formal_semantics_(linguistics)" title="Formal semantics (linguistics)">formal theories of semantics</a> of <a href="/wiki/Natural_language" title="Natural language">natural languages</a>, especially <a href="/wiki/Montague_grammar" title="Montague grammar">Montague grammar</a> and its descendants. As <a href="/wiki/John_Lane_Bell" title="John Lane Bell">John Lane Bell</a> writes: "In fact categories can <i>themselves</i> be viewed as type theories of a certain kind; this fact alone indicates that type theory is much more closely related to category theory than it is to set theory." In brief, a category can be viewed as a type theory by regarding its objects as types (or sorts), i.e. "Roughly speaking, a category may be thought of as a type theory shorn of its syntax." A number of significant results follow in this way:<sup id="cite_ref-Sets_and_Extensions_in_the_Twentieth_Century_3-0" class="reference"><a href="#cite_note-Sets_and_Extensions_in_the_Twentieth_Century-3">[3]</a></sup></p>/n<ul>/n<li><a href="/wiki/Cartesian_closed_category" title="Cartesian closed category">cartesian closed categories</a> correspond to the typed λ-calculus (<a href="/wiki/Lambek" class="mw-redirect" title="Lambek">Lambek</a>, 1970);</li>/n<li><a href="/w/index.php?title=C-monoid&amp;action=edit&amp;redlink=1" class="new" title="C-monoid (page does not exist)">C-monoids</a> (categories with products and exponentials and a single, nonterminal object) correspond to the untyped λ-calculus (observed independently by Lambek and <a href="/wiki/Dana_Scott" title="Dana Scott">Dana Scott</a> around 1980);</li>/n<li><a href="/wiki/Locally_cartesian_closed_category" class="mw-redirect" title="Locally cartesian closed category">locally cartesian closed categories</a> correspond to <a href="/wiki/Martin-L%C3%B6f_type_theory" class="mw-redirect" title="Martin-Löf type theory">Martin-Löf type theories</a> (Seely, 1984).</li>/n</ul>/n<p>The interplay, known as <a href="/wiki/Categorical_logic" title="Categorical logic">categorical logic</a>, has been a subject of active research since then; see the monograph of Jacobs (1999) for instance.</p>/n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_theory&amp;action=edit&amp;section=22" title="Edit section: See also">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<ul>/n<li><a href="/wiki/Data_type" title="Data type">Data type</a> for concrete types of data in programming</li>/n<li><a href="/wiki/Domain_theory" title="Domain theory">Domain theory</a></li>/n<li><a href="/wiki/Type_(model_theory)" title="Type (model theory)">Type (model theory)</a></li>/n<li><a href="/wiki/Type_system" title="Type system">Type system</a> for a more practical discussion of type systems for programming languages</li>/n</ul>/n<h2><span class="mw-headline" id="Notes">Notes</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_theory&amp;action=edit&amp;section=23" title="Edit section: Notes">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div class="reflist columns references-column-width" style="-moz-column-width: 30em;
