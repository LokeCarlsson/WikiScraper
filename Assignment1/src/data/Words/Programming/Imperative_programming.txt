href="/wiki/Programming_paradigm" title="Programming paradigm">programming paradigm</a> that uses <a href="/wiki/Statement_(computer_science)" title="Statement (computer science)">statements</a> that change a program's <a href="/wiki/State_(computer_science)" title="State (computer science)">state</a>. In much the same way that the <a href="/wiki/Imperative_mood" title="Imperative mood">imperative mood</a> in <a href="/wiki/Natural_language" title="Natural language">natural languages</a> expresses commands, an imperative program consists of <a href="/wiki/Command_(computing)" title="Command (computing)">commands</a> for the <a href="/wiki/Computer" title="Computer">computer</a> to perform. Imperative programming focuses on describing <i>how</i> a program operates.</p>/n<p>The term is often used in contrast to <a href="/wiki/Declarative_programming" title="Declarative programming">declarative programming</a>, which focuses on <i>what</i> the program should accomplish without specifying <i>how</i> the program should achieve the result.</p>/n<p></p>/n<div id="toc" class="toctext">References</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="Imperative_and_procedural_programming">Imperative and procedural programming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Imperative_programming&amp;action=edit&amp;section=1" title="Edit section: Imperative and procedural programming">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p><a href="/wiki/Procedural_programming" title="Procedural programming">Procedural programming</a> is a type of imperative programming in which the program is built from one or more procedures (also termed <a href="/wiki/Subroutine" title="Subroutine">subroutines</a> or functions). The terms are often used as synonyms, but the use of procedures has a dramatic effect on how imperative programs appear and how they are constructed. Heavily-procedural programming, in which <a href="/wiki/State_(computer_science)" title="State (computer science)">state</a> changes are localized to procedures or restricted to explicit arguments and returns from procedures, is a form of <a href="/wiki/Structured_programming" title="Structured programming">structured programming</a>. From the 1960s onwards, structured programming and <a href="/wiki/Modular_programming" title="Modular programming">modular programming</a> in general have been promoted as techniques to improve the <a href="/wiki/Maintainability" title="Maintainability">maintainability</a> and overall quality of imperative programs. The concepts behind <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> attempt to extend this approach.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup></p>/n<p>Procedural programming could be considered a step towards declarative programming. A programmer can often tell, simply by looking at the names, arguments, and return types of procedures (and related comments), what a particular procedure is supposed to do, without necessarily looking at the details of how it achieves its result. At the same time, a complete program is still imperative since it <i>fixes</i> the statements to be executed and their order of execution to a large extent.</p>/n<h2><span class="mw-headline" id="Rationale_and_foundations_of_imperative_programming">Rationale and foundations of imperative programming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Imperative_programming&amp;action=edit&amp;section=2" title="Edit section: Rationale and foundations of imperative programming">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The hardware implementation of almost all computers is imperative.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[note 1]</a></sup> Nearly all computer hardware is designed to execute <a href="/wiki/Machine_code" title="Machine code">machine code</a>, which is native to the computer, written in the imperative style. From this low-level perspective, the program state is defined by the contents of memory, and the statements are instructions in the native machine language of the computer. Higher-level imperative languages use <a href="/wiki/Variable_(programming)" class="mw-redirect" title="Variable (programming)">variables</a> and more complex statements, but still follow the same paradigm. <a href="/wiki/Recipe" title="Recipe">Recipes</a> and process <a href="/wiki/Checklist" title="Checklist">checklists</a>, while not <a href="/wiki/Computer_program" title="Computer program">computer programs</a>, are also familiar concepts that are similar in style to imperative programming; each step is an instruction, and the physical world holds the state. Since the basic ideas of imperative programming are both conceptually familiar and directly embodied in the hardware, most computer languages are in the imperative style.</p>/n<p><a href="/wiki/Destructive_assignment" class="mw-redirect" title="Destructive assignment">Assignment statements</a>, in imperative paradigm, perform an operation on information located in memory and store the results in memory for later use. High-level imperative languages, in addition, permit the <a href="/wiki/Evaluation_(disambiguation)" class="mw-disambig" title="Evaluation (disambiguation)">evaluation</a> of complex <a href="/wiki/Expression_(programming)" class="mw-redirect" title="Expression (programming)">expressions</a>, which may consist of a combination of <a href="/wiki/Arithmetic#Arithmetic_operations" title="Arithmetic">arithmetic operations</a> and <a href="/wiki/Function_(mathematics)" title="Function (mathematics)">function</a> evaluations, and the assignment of the resulting value to memory. Looping statements (as in <a href="/wiki/While_loop" title="While loop">while loops</a>, <a href="/wiki/Do_while_loop" title="Do while loop">do while loops</a>, and <a href="/wiki/For_loop" title="For loop">for loops</a>) allow a sequence of statements to be executed multiple times. Loops can either execute the statements they contain a predefined number of times, or they can execute them repeatedly until some condition changes. <a href="/wiki/Conditional_(programming)" class="mw-redirect" title="Conditional (programming)">Conditional</a> <a href="/wiki/Branch_(computer_science)" title="Branch (computer science)">branching</a> statements allow a sequence of statements to be executed only if some condition is met. Otherwise, the statements are skipped and the execution sequence continues from the statement following them. Unconditional branching statements allow an execution sequence to be transferred to another part of a program. These include the jump (called <i><a href="/wiki/Goto" title="Goto">goto</a></i> in many languages), <a href="/wiki/Switch_statement" title="Switch statement">switch</a>, and the subprogram, <a href="/wiki/Subroutine" title="Subroutine">subroutine</a>, or procedure call (which usually returns to the next statement after the call).</p>/n<p>Early in the development of <a href="/wiki/High-level_programming_language" title="High-level programming language">high-level programming languages</a>, the introduction of the <a href="/wiki/Block_(programming)" title="Block (programming)">block</a> enabled the construction of programs in which a group of statements and declarations could be treated as if they were one statement. This, alongside the introduction of <a href="/wiki/Subroutine" title="Subroutine">subroutines</a>, enabled complex structures to be expressed by hierarchical decomposition into simpler procedural structures.</p>/n<p>Many imperative programming languages (such as <a href="/wiki/Fortran" title="Fortran">Fortran</a>, <a href="/wiki/BASIC" title="BASIC">BASIC</a>, and <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a>) are <a href="/wiki/Abstraction_(computer_science)" class="mw-redirect" title="Abstraction (computer science)">abstractions</a> of <a href="/wiki/Assembly_language" title="Assembly language">assembly language</a>.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[2]</a></sup></p>/n<h2><span class="mw-headline" id="History_of_imperative_and_object-oriented_languages">History of imperative and object-oriented languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Imperative_programming&amp;action=edit&amp;section=3" title="Edit section: History of imperative and object-oriented languages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The earliest imperative languages were the machine languages of the original computers. In these languages, instructions were very simple, which made hardware implementation easier, but hindered the creation of complex programs. FORTRAN, developed by <a href="/wiki/John_Backus" title="John Backus">John Backus</a> at <a href="/wiki/International_Business_Machines" class="mw-redirect" title="International Business Machines">International Business Machines</a> (IBM) starting in 1954, was the first major programming language to remove the obstacles presented by machine code in the creation of complex programs. FORTRAN was a <a href="/wiki/Compiled_language" title="Compiled language">compiled language</a> that allowed named variables, complex expressions, subprograms, and many other features now common in imperative languages. The next two decades saw the development of many other major high-level imperative programming languages. In the late 1950s and 1960s, <a href="/wiki/ALGOL" title="ALGOL">ALGOL</a> was developed in order to allow mathematical algorithms to be more easily expressed, and even served as the <a href="/wiki/Operating_system" title="Operating system">operating system</a>'s target language for some computers. <a href="/wiki/MUMPS" title="MUMPS">MUMPS</a> (1966) carried the imperative paradigm to a logical extreme, by not having any statements at all, relying purely on commands, even to the extent of making the IF and ELSE commands independent of each other, connected only by an intrinsic variable named $TEST. <a href="/wiki/COBOL" title="COBOL">COBOL</a> (1960) and <a href="/wiki/BASIC" title="BASIC">BASIC</a> (1964) were both attempts to make programming syntax look more like English. In the 1970s, <a href="/wiki/Pascal_(programming_language)" title="Pascal (programming language)">Pascal</a> was developed by <a href="/wiki/Niklaus_Wirth" title="Niklaus Wirth">Niklaus Wirth</a>, and <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> was created by <a href="/wiki/Dennis_Ritchie" title="Dennis Ritchie">Dennis Ritchie</a> while he was working at <a href="/wiki/Bell_Laboratories" class="mw-redirect" title="Bell Laboratories">Bell Laboratories</a>. Wirth went on to design <a href="/wiki/Modula-2" title="Modula-2">Modula-2</a> and <a href="/wiki/Oberon_(programming_language)" title="Oberon (programming language)">Oberon</a>. For the needs of the <a href="/wiki/United_States_Department_of_Defense" title="United States Department of Defense">United States Department of Defense</a>, <a href="/wiki/Jean_Ichbiah" title="Jean Ichbiah">Jean Ichbiah</a> and a team at <a href="/wiki/Honeywell" title="Honeywell">Honeywell</a> began designing <a href="/wiki/Ada_(programming_language)" title="Ada (programming language)">Ada</a> in 1978, after a 4-year project to define the requirements for the language. The specification was first published in 1983, with revisions in 1995, 2005 and 2012.</p>/n<p>The 1980s saw a rapid growth in interest in <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a>. These languages were imperative in style, but added features to support <a href="/wiki/Object_(computing)" class="mw-redirect" title="Object (computing)">objects</a>. The last two decades of the 20th century saw the development of many such languages. <a href="/wiki/Smalltalk" title="Smalltalk">Smalltalk</a>-80, originally conceived by <a href="/wiki/Alan_Kay" title="Alan Kay">Alan Kay</a> in 1969, was released in 1980, by the Xerox Palo Alto Research Center (<a href="/wiki/PARC_(company)" title="PARC (company)">PARC</a>). Drawing from concepts in another object-oriented language—<a href="/wiki/Simula" title="Simula">Simula</a> (which is considered the world's first <a href="/wiki/Object-oriented_programming_language" class="mw-redirect" title="Object-oriented programming language">object-oriented programming language</a>, developed in the 1960s)—<a href="/wiki/Bjarne_Stroustrup" title="Bjarne Stroustrup">Bjarne Stroustrup</a> designed <a href="/wiki/C%2B%2B" title="C++">C++</a>, an object-oriented language based on <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a>. Design of <a href="/wiki/C%2B%2B" title="C++">C++</a> began in 1979 and the first implementation was completed in 1983. In the late 1980s and 1990s, the notable imperative languages drawing on object-oriented concepts were <a href="/wiki/Perl" title="Perl">Perl</a>, released by <a href="/wiki/Larry_Wall" title="Larry Wall">Larry Wall</a> in 1987; <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, released by <a href="/wiki/Guido_van_Rossum" title="Guido van Rossum">Guido van Rossum</a> in 1990; <a href="/wiki/Visual_Basic" title="Visual Basic">Visual Basic</a> and <a href="/wiki/Visual_C%2B%2B" class="mw-redirect" title="Visual C++">Visual C++</a> (which included <a href="/wiki/Microsoft_Foundation_Class_Library" title="Microsoft Foundation Class Library">Microsoft Foundation Class Library</a> (MFC) 2.0), released by <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> in 1991 and 1993 respectively; <a href="/wiki/PHP" title="PHP">PHP</a>, released by <a href="/wiki/Rasmus_Lerdorf" title="Rasmus Lerdorf">Rasmus Lerdorf</a> in 1994; <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, released by <a href="/wiki/Sun_Microsystems" title="Sun Microsystems">Sun Microsystems</a> in 1994, <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a>, by <a href="/wiki/Brendan_Eich" title="Brendan Eich">Brendan Eich</a> (<a href="/wiki/Netscape" title="Netscape">Netscape</a>), and <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a>, by Yukihiro "Matz" Matsumoto, both released in 1995. Microsoft's <a href="/wiki/.NET_Framework" title=".NET Framework">.NET Framework</a> (2002) is imperative at its core, as are its main target languages, <a href="/wiki/VB.NET" class="mw-redirect" title="VB.NET">VB.NET</a> and <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a> that run on it; however Microsoft's <a href="/wiki/F_Sharp_(programming_language)" title="F Sharp (programming language)">F#</a>, a functional language, also runs on it.</p>/n<h2><span class="mw-headline" id="See_also">See also</span><span
