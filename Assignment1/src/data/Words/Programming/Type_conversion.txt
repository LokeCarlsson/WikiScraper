Unsourced material may be challenged and removed.</span> <small><i>(May 2011)</i></small> <small class="hide-when-compact"><i>(<a href="/wiki/Help:Maintenance_template_removal" title="Help:Maintenance template removal">Learn how and when to remove this template message</a>)</i></small></div>/n</td>/n</tr>/n</table>/n<p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, <b>type conversion</b>, <b>type casting</b>, and <b>type coercion</b> are different ways of changing an entity of one <a href="/wiki/Data_type" title="Data type">data type</a> into another. An example would be the conversion of an <a href="/wiki/Integer_(computer_science)" title="Integer (computer science)">integer</a> value into a <a href="/wiki/Floating_point" class="mw-redirect" title="Floating point">floating point</a> value or its textual representation as a <a href="/wiki/String_(computer_science)" title="String (computer science)">string</a>, and vice versa. Type conversions can take advantage of certain features of <a href="/wiki/Type_hierarchy" class="mw-redirect" title="Type hierarchy">type hierarchies</a> or <a href="/wiki/Data_representation" class="mw-redirect" title="Data representation">data representations</a>. Two important aspects of a type conversion is whether it happens <i>implicitly</i> or <i>explicitly</i>, and whether the underlying data representation is converted from one representation into another, or a given representation is merely <i>reinterpreted</i> as the representation of another data type. In general, both <a href="/wiki/Primitive_data_type" title="Primitive data type">primitive</a> and <a href="/wiki/Compound_data_type" class="mw-redirect" title="Compound data type">compound data types</a> can be converted.</p>/n<p>Each <a href="/wiki/Programming_language" title="Programming language">programming language</a> has its own rules on how types can be converted. Languages with <a href="/wiki/Strong_typing" class="mw-redirect" title="Strong typing">strong typing</a> typically do little implicit conversion and discourage the reinterpretation of representations, while languages with <a href="/wiki/Weak_typing" class="mw-redirect" title="Weak typing">weak typing</a> perform many implicit conversions between data types. Weak typing language often allow forcing the <a href="/wiki/Compiler" title="Compiler">compiler</a> to arbitrarily interpret a data item as having different representations—this can be a non-obvious programming error, or a technical method to directly deal with underlying hardware.</p>/n<p>In most languages, the word <i>coercion</i> is used to denote an <i>implicit</i> conversion, either during compilation or during <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run time</a>. For example, in an expression mixing integer and floating point numbers (like 5 + 0.1), the compiler will automatically convert integer representation into floating point representation so fractions are not lost. Explicit type conversions are either indicated by writing additional code (e.g. adding type identifiers or calling built-in <a href="/wiki/Subroutine" title="Subroutine">routines</a>) or by coding conversion routines for the compiler to use when it otherwise would halt with a type mismatch.</p>/n<p>In most <a href="/wiki/ALGOL" title="ALGOL">ALGOL</a>-like languages, such as <a href="/wiki/Pascal_(programming_language)" title="Pascal (programming language)">Pascal</a>, <a href="/wiki/Modula-2" title="Modula-2">Modula-2</a>, <a href="/wiki/Ada_(programming_language)" title="Ada (programming language)">Ada (programming language)</a> and <a href="/wiki/Delphi_(programming_language)" title="Delphi (programming language)">Delphi</a>, <i>conversion</i> and <i>casting</i> are distinctly different concepts. In these languages, <i>conversion</i> refers to either implicitly or explicitly changing a value from one data type storage format to another, e.g. a 16-bit integer to a 32-bit integer. The storage needs may change as a result of the conversion, including a possible loss of precision or truncation. The word <i>cast</i>, on the other hand, refers to explicitly changing the <i>interpretation</i> of the <i>bit pattern</i> representing a value from one type to another. For example, 32 contiguous bits may be treated as an array of 32 booleans, a 4-byte string, an unsigned 32-bit integer or an IEEE single precision floating point value. Because the stored bits are never changed, the programmer must know low level details such as representation format, byte order, and alignment needs, to meaningfully cast.</p>/n<p>In the C family of languages and <a href="/wiki/ALGOL_68" title="ALGOL 68">ALGOL 68</a>, the word <i>cast</i> typically refers to an <i>explicit</i> type conversion (as opposed to an implicit conversion), causing some ambiguity about whether this is a re-interpretation of a bit-pattern or a real data representation conversion. More important is the multitude of ways and rules that apply to what data type (or class) is located by a pointer and how a pointer may be adjusted by the compiler in cases like object (class) inheritance.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toctitle">/n<h2>Contents</h2>/n</div>/n<ul>/n<li class="toclevel-1 tocsection-1"><a href="#Language_comparison"><span class="mw-editsection-bracket">]</span></span></h3>/n<h4><span class="mw-headline" id="Implicit_type_conversion">Implicit type conversion</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_conversion&amp;action=edit&amp;section=3" title="Edit section: Implicit type conversion">edit</a><span class="mw-editsection-bracket">]</span></span></h4>/n<p>Implicit type conversion, also known as <i>coercion</i>, is an automatic type conversion by the <a href="/wiki/Compiler" title="Compiler">compiler</a>. Some <a href="/wiki/Programming_language" title="Programming language">programming languages</a> allow compilers to provide coercion; others require it.</p>/n<p>In a mixed-type expression, data of one or more <a href="/wiki/Subtyping" title="Subtyping">subtypes</a> can be <a href="/w/index.php?title=Type_conversion_in_C&amp;action=edit&amp;redlink=1" class="new" title="Type conversion in C (page does not exist)">converted</a> to a supertype as needed at <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">runtime</a> so that the program will run correctly. For example, the following is legal <a href="/wiki/C_(programming_language)" title="C (programming language)">C language</a> code:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="kt">double</span>  <span class="n">d</span><span class="p">;</span>/n<span class="kt">long</span>    <span class="n">l</span><span class="p">;</span>/n<span class="kt">int</span>     <span class="n">i</span><span class="p">;</span>/n/n<span class="k">if</span> <span class="p">(</span><span class="n">d</span> <span class="o">&gt;</span> <span class="n">i</span><span class="p">)</span>   <span class="n">d</span> <span class="o">=</span> <span class="n">i</span><span class="p">;</span>/n<span class="k">if</span> <span class="p">(</span><span class="n">i</span> <span class="o">&gt;</span> <span class="n">l</span><span class="p">)</span>   <span class="n">l</span> <span class="o">=</span> <span class="n">i</span><span class="p">;</span>/n<span class="k">if</span> <span class="p">(</span><span class="n">d</span> <span class="o">==</span> <span class="n">l</span><span class="p">)</span>  <span class="n">d</span> <span class="o">*=</span> <span class="mi">2</span><span class="p">;</span>/n</pre></div>/n<p>Although <code class="mw-highlight" dir="ltr"><span class="n">d</span></code>, <code class="mw-highlight" dir="ltr"><span class="n">l</span></code> and <code class="mw-highlight" dir="ltr"><span class="n">i</span></code> belong to different data types, they will be automatically converted to equal data types each time a comparison or assignment is executed. This behavior should be used with caution, as unintended consequences can arise. Data can be lost when converting representations from floating-point to integer, as the fractional components of the floating-point values will be truncated (rounded toward zero). Conversely, precision can be lost when converting representations from integer to floating-point, since a floating-point type may be unable to exactly represent an integer type. For example, <code class="mw-highlight" dir="ltr"><span class="kt">float</span></code> might be an <a href="/wiki/IEEE_754" title="IEEE 754">IEEE 754</a> single precision type, which cannot represent the integer 16777217 exactly, while a 32-bit integer type can. This can lead to unintuitive behavior, as demonstrated by the following code:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="cp">#include</span> <span class="cpf">&lt;stdio.h&gt;</span>/n/n<span class="kt">int</span> <span class="nf">main</span><span class="p">(</span><span class="kt">void</span><span class="p">)</span>/n<span class="p">{</span>/n    <span class="kt">int</span> <span class="n">i_value</span>   <span class="o">=</span> <span class="mi">16777217</span><span class="p">;</span>/n    <span class="kt">float</span> <span class="n">f_value</span> <span class="o">=</span> <span class="mf">16777216.0</span><span class="p">;</span>/n    <span class="n">printf</span><span class="p">(</span><span class="s">"The integer is: %d</span><span class="se">\n</span><span class="s">"</span><span class="p">,</span> <span class="n">i_value</span><span class="p">);</span>/n    <span class="n">printf</span><span class="p">(</span><span class="s">"The float is:   %f</span><span class="se">\n</span><span class="s">"</span><span class="p">,</span> <span class="n">f_value</span><span class="p">);</span>/n    <span class="n">printf</span><span class="p">(</span><span class="s">"Their equality: %d</span><span class="se">\n</span><span class="s">"</span><span class="p">,</span> <span class="n">i_value</span> <span class="o">==</span> <span class="n">f_value</span><span class="p">);</span>/n<span class="p">}</span>/n</pre></div>/n<p>On compilers that implement floats as IEEE single precision, and ints as at least 32 bits, this code will give this peculiar print-out:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n    <span class="n">The</span> <span class="n">integer</span> <span class="nl">is</span><span class="p">:</span> <span class="mi">16777217</span>/n    <span class="n">The</span> <span class="kt">float</span> <span class="nl">is</span><span class="p">:</span> <span class="mf">16777216.000000</span>/n    <span class="n">Their</span> <span class="nl">equality</span><span class="p">:</span> <span class="mi">1</span>/n</pre></div>/n<p>Note that 1 represents equality in the last line above. This odd behavior is caused by an implicit conversion of <code class="mw-highlight" dir="ltr"><span class="n">i_value</span></code> to float when it is compared with <code class="mw-highlight" dir="ltr"><span class="n">f_value</span></code>. The conversion causes loss of precision, which makes the values equal before the comparison.</p>/n<p>Important takeaways:</p>/n<ol>/n<li><code class="mw-highlight" dir="ltr"><span class="kt">float</span></code> to <code class="mw-highlight" dir="ltr"><span class="kt">int</span></code> causes <a href="/wiki/Truncation" title="Truncation">truncation</a>, i.e., removal of the fractional part.</li>/n<li><code class="mw-highlight" dir="ltr"><span class="kt">double</span></code> to <code class="mw-highlight" dir="ltr"><span class="kt">float</span></code> causes rounding of digit.</li>/n<li><code class="mw-highlight" dir="ltr"><span class="kt">long</span></code> to <code class="mw-highlight" dir="ltr"><span class="kt">int</span></code> causes dropping of excess higher order bits.</li>/n</ol>/n<h5><span class="mw-headline" id="Type_promotion">Type promotion</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_conversion&amp;action=edit&amp;section=4" title="Edit section: Type promotion">edit</a><span class="mw-editsection-bracket">]</span></span></h5>/n<p>One special case of implicit type conversion is type promotion, where the compiler automatically expands the binary representation of objects of integer or floating-point types. Promotions are commonly used with types smaller than the native type of the target platform's <a href="/wiki/Arithmetic_logic_unit" title="Arithmetic logic unit">arithmetic logic unit</a> (ALU), before arithmetic and logical operations, to make such operations possible, or more efficient if the ALU can work with more than one type. C and C++ perform such promotion for objects of boolean, character, wide character, enumeration, and short integer types which are promoted to int, and for objects of type float, which are promoted to double. Unlike some other type conversions, promotions never lose precision or modify the value stored in the object.</p>/n<p>In <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="kt">int</span> <span class="n">x</span> <span class="o">=</span> <span class="mi">3</span><span class="o">;</span>/n<span class="kt">double</span> <span class="n">y</span> <span class="o">=</span> <span class="mf">3.5</span><span class="o">;</span>/n<span class="n">System</span><span class="o">.</span><span class="na">out</span><span class="o">.</span><span class="na">println</span><span class="o">(</span><span class="n">x</span><span class="o">+</span><span class="n">y</span><span class="o">);</span> <span class="c1">//The output will be 6.5</span>/n</pre></div>/n<h4><span class="mw-headline" id="Explicit_type_conversion">Explicit type conversion</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_conversion&amp;action=edit&amp;section=5" title="Edit section: Explicit type conversion">edit</a><span class="mw-editsection-bracket">]</span></span></h4>/n<p>Explicit type conversion is a type conversion which is explicitly defined within a program (instead of being done by a compiler for implicit type conversion). It is defined by the user in the program.</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="kt">double</span> <span class="n">da</span> <span class="o">=</span> <span class="mf">3.3</span><span class="p">;</span>/n<span class="kt">double</span> <span class="n">db</span> <span class="o">=</span> <span class="mf">3.3</span><span class="p">;</span>/n<span class="kt">double</span> <span class="n">dc</span> <span class="o">=</span> <span class="mf">3.4</span><span class="p">;</span>/n<span class="kt">int</span> <span class="n">result</span> <span class="o">=</span> <span class="p">(</span><span class="kt">int</span><span class="p">)</span><span class="n">da</span> <span class="o">+</span> <span class="p">(</span><span class="kt">int</span><span class="p">)</span><span class="n">db</span> <span class="o">+</span> <span class="p">(</span><span class="kt">int</span><span class="p">)</span><span class="n">dc</span><span class="p">;</span> <span class="c1">//result == 9</span>/n<span class="c1">//if implicit conversion would be used (as with "result = da + db + dc"), result would be equal to 10</span>/n</pre></div>/n<p>There are several kinds of explicit conversion.</p>/n<dl>/n<dt>checked</dt>/n<dd>Before the conversion is performed, a runtime check is done to see if the destination type can hold the source value. If not, an error condition is raised.</dd>/n<dt>unchecked</dt>/n<dd>No check is performed. If the destination type cannot hold the source value, the result is undefined.</dd>/n<dt>bit pattern</dt>/n<dd>The raw bit representation of the source is copied verbatim, and it is re-interpreted according to the destination type. This can also be achieved via <a href="/wiki/Aliasing_(computing)" title="Aliasing (computing)">aliasing</a>.</dd>/n</dl>/n<p>In <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> languages, objects can also be <a href="/wiki/Downcasting" title="Downcasting">downcast</a>&#160;: a reference of a base class is cast to one of its derived classes.</p>/n<h3><span id="C#_and_C++"></span><span class="mw-headline" id="C.23_and_C.2B.2B">C# and C++</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_conversion&amp;action=edit&amp;section=7" title="Edit section: Eiffel">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>In <a href="/wiki/Eiffel_(programming_language)" title="Eiffel (programming language)">Eiffel</a> the notion of type conversion is integrated into the rules of the type system. The Assignment Rule says that an assignment, such as:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n <span class="n">x</span> <span class="o">:=</span> <span class="n">y</span>/n</pre></div>/n<p>is valid if and only if the type of its source expression, <code lang="eiffel" xml:lang="eiffel">y</code> in this case, is <i>compatible with</i> the type of its target entity, <code lang="eiffel" xml:lang="eiffel">x</code> in this case. In this rule, <i>compatible with</i> means that the type of the source expression either <i>conforms to</i> or <i>converts to</i> that of the target. Conformance of types is defined by the familiar rules for <a href="/wiki/Polymorphism_in_object-oriented_programming" class="mw-redirect" title="Polymorphism in object-oriented programming">polymorphism in object-oriented programming</a>. For example, in the assignment above, the type of <code lang="eiffel" xml:lang="eiffel">y</code> conforms to the type of <code lang="eiffel" xml:lang="eiffel">x</code> if the class upon which <code lang="eiffel" xml:lang="eiffel">y</code> is based is a descendant of that upon which <code lang="eiffel" xml:lang="eiffel">x</code> is based.</p>/n<h4><span class="mw-headline" id="Definition_of_type_conversion_in_Eiffel">Definition of type conversion in Eiffel</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_conversion&amp;action=edit&amp;section=8" title="Edit section: Definition of type conversion in Eiffel">edit</a><span class="mw-editsection-bracket">]</span></span></h4>/n<p>The actions of type conversion in Eiffel, specifically <i>converts to</i> and <i>converts from</i> are defined as:</p>/n<blockquote>/n<p>A type based on a class CU <i>converts to</i> a type T based on a class CT (and T <i>converts from</i> U) if either</p>/n<dl>/n<dd>CT has a <i>conversion procedure</i> using U as a conversion type, or</dd>/n<dd>CU has a <i>conversion query</i> listing T as a conversion type</dd>/n</dl>/n</blockquote>/n<h4><span class="mw-headline" id="Example">Example</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_conversion&amp;action=edit&amp;section=9" title="Edit section: Example">edit</a><span class="mw-editsection-bracket">]</span></span></h4>/n<p>Eiffel is a fully compliant <a href="/wiki/List_of_CLI_languages" title="List of CLI languages">language</a> for Microsoft <a href="/wiki/.NET_Framework" title=".NET Framework">.NET Framework</a>. Before development of .NET, Eiffel already had extensive class libraries. Using the .NET type libraries, particularly with commonly used types such as strings, poses a conversion problem. Existing Eiffel software uses the string classes (such as <code lang="eiffel" xml:lang="eiffel">STRING_8</code>) from the Eiffel libraries, but Eiffel software written for .NET must use the .NET string class (<code lang="eiffel" xml:lang="eiffel">System.String</code>) in many cases, for example when calling .NET methods which expect items of the .NET type to be passed as arguments. So, the conversion of these types back and forth needs to be as seamless as possible.</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n    <span class="n">my_string</span><span class="p">:</span> <span class="nc">STRING_8</span>                 <span class="c1">-- Native Eiffel string</span>/n    <span class="n">my_system_string</span><span class="p">:</span> <span class="nc">SYSTEM_STRING</span>     <span class="c1">-- Native .NET string</span>/n/n        <span class="p">...</span>/n/n            <span class="n">my_string</span> <span class="o">:=</span> <span class="n">my_system_string</span>/n</pre></div>/n<p>In the code above, two strings are declared, one of each different type (<code lang="eiffel" xml:lang="eiffel">SYSTEM_STRING</code> is the Eiffel compliant alias for System.String). Because <code lang="eiffel" xml:lang="eiffel">System.String</code> does not conform to <code lang="eiffel" xml:lang="eiffel">STRING_8</code>, then the assignment above is valid only if <code lang="eiffel" xml:lang="eiffel">System.String</code> converts to <code lang="eiffel" xml:lang="eiffel">STRING_8</code>.</p>/n<p>The Eiffel class <code lang="eiffel" xml:lang="eiffel">STRING_8</code> has a conversion procedure <code lang="eiffel" xml:lang="eiffel">make_from_cil</code> for objects of type <code lang="eiffel" xml:lang="eiffel">System.String</code>. Conversion procedures are also always designated as creation procedures (similar to constructors). The following is an excerpt from the <code lang="eiffel" xml:lang="eiffel">STRING_8</code> class:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n    <span class="kr">class</span> <span class="nc">STRING_8</span>/n        <span class="p">...</span>/n    <span class="kr">create</span>/n        <span class="n">make_from_cil</span>/n        <span class="p">...</span>/n    <span class="kr">convert</span>/n        <span class="n">make_from_cil</span> <span class="p">({</span><span class="nc">SYSTEM_STRING</span><span class="p">})</span>/n        <span class="p">...</span>/n</pre></div>/n<p>The presence of the conversion procedure makes the assignment:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n            <span class="n">my_string</span> <span class="o">:=</span> <span class="n">my_system_string</span>/n</pre></div>/n<p>semantically equivalent to:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n            <span class="kr">create</span> <span class="n">my_string</span><span class="p">.</span><span class="n">make_from_cil</span> <span class="p">(</span><span class="n">my_system_string</span><span class="p">)</span>/n</pre></div>/n<p>in which <code lang="eiffel" xml:lang="eiffel">my_string</code> is constructed as a new object of type <code lang="eiffel" xml:lang="eiffel">STRING_8</code> with content equivalent to that of <code lang="eiffel" xml:lang="eiffel">my_system_string</code>.</p>/n<p>To handle an assignment with original source and target reversed:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n            <span class="n">my_system_string</span> <span class="o">:=</span> <span class="n">my_string</span>/n</pre></div>/n<p>the class <code lang="eiffel" xml:lang="eiffel">STRING_8</code> also contains a conversion query <code lang="eiffel" xml:lang="eiffel">to_cil</code> which will produce a <code lang="eiffel" xml:lang="eiffel">System.String</code> from an instance of <code lang="eiffel" xml:lang="eiffel">STRING_8</code>.</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n    <span class="kr">class</span> <span class="nc">STRING_8</span>/n        <span class="p">...</span>/n    <span class="kr">create</span>/n        <span class="n">make_from_cil</span>/n        <span class="p">...</span>/n    <span class="kr">convert</span>/n        <span class="n">make_from_cil</span> <span class="p">({</span><span class="nc">SYSTEM_STRING</span><span class="p">})</span>/n        <span class="n">to_cil</span><span class="p">:</span> <span class="p">{</span><span class="nc">SYSTEM_STRING</span><span class="n">to_cil</span>/n</pre></div>/n<p>In Eiffel, the setup for type conversion is included in the class code, but then appears to happen as automatically as <a The includes not just assignments but other types of attachments as well, such as argument (parameter) substitution.</p>/n<h2><span class="mw-headline" id="Security_issues">Security issues</span><span
