class="mw-parser-output"><p>In <a href="/wiki/Computability_theory" title="Computability theory">computability theory</a>, <b>Rice's theorem</b> states that all non-trivial, <a href="/wiki/Semantics_(computer_science)" title="Semantics (computer science)">semantic</a> properties of programs are <a href="/wiki/Decision_problem" title="Decision problem">undecidable</a>. A semantic property is one about the program's behavior (for instance, does the program <a href="/wiki/Halting_problem" title="Halting problem">terminate</a> for all inputs), unlike a syntactic property (for instance, does the program contain an <a href="/wiki/If-then-else" class="mw-redirect" title="If-then-else">if-then-else</a> statement). A property is <i>non-trivial</i> if it is neither true for every program, nor for no program.</p>/n<p>Rice's theorem can also be put in terms of functions: for any non-trivial property of <a href="/wiki/Partial_functions" class="mw-redirect" title="Partial functions">partial functions</a>, no general and effective method can decide whether an <a href="/wiki/Algorithm" title="Algorithm">algorithm</a> computes a partial function with that property. Here, a property of partial functions is called <i>trivial</i> if it holds for all <a href="/wiki/Partial_computable_function" class="mw-redirect" title="Partial computable function">partial computable functions</a> or for none, and an effective decision method is called <i>general</i> if it decides correctly for every algorithm. The theorem is named after <a href="/wiki/Henry_Gordon_Rice" title="Henry Gordon Rice">Henry Gordon Rice</a>, and is also known as the <b>Rice–Myhill–Shapiro theorem</b> after Rice, <a href="/wiki/John_Myhill" title="John Myhill">John Myhill</a>, and <a href="/wiki/Norman_Shapiro" title="Norman Shapiro">Norman Shapiro</a>.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Rice%27s_theorem&amp;action=edit&amp;section=1" title="Edit section: Introduction">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Another way of stating Rice's theorem that is more useful in <a href="/wiki/Computability_theory_(computer_science)" class="mw-redirect" title="Computability theory (computer science)">computability theory</a> follows.</p>/n<p>Let <i>S</i> be a set of <a href="/wiki/Formal_language" title="Formal language">languages</a> that is nontrivial, meaning</p>/n<ol>/n<li>there exists a Turing machine that recognizes a language in <i>S</i></li>/n<li>there exists a Turing machine that recognizes a language not in <i>S</i></li>/n</ol>/n<p>Then, it is <a href="/wiki/Decision_problem" title="Decision problem">undecidable</a> to determine whether the language recognized by an arbitrary <a href="/wiki/Turing_machine" title="Turing machine">Turing machine</a> lies in <i>S</i>.</p>/n<p>In practice, this means that there is no machine that can always decide whether the language of a given Turing machine has a particular nontrivial property. Special cases include the undecidability of whether a Turing machine accepts a particular string, whether a Turing machine recognizes a particular recognizable language, and whether the language recognized by a Turing machine could be recognized by a nontrivial simpler machine, such as a <a href="/wiki/Finite_state_machine" class="mw-redirect" title="Finite state machine">finite automaton</a>.</p>/n<p>It is important to note that Rice's theorem does not say anything about those properties of machines or programs that are not also properties of functions and languages. For example, whether a machine runs for more than 100 steps on some input is a decidable property, even though it is non-trivial. Implementing exactly the same language, two different machines might require a different number of steps to recognize the same input. Similarly, whether a machine has more than 5 states is a decidable property of the machine, as the number of states can simply be counted. Where a property is of the kind that either of the two machines may or may not have it, while still implementing exactly the same language, the property is of the machines and not of the language, and Rice's Theorem does not apply.</p>/n<p>Using <a href="/wiki/Hartley_Rogers,_Jr" class="mw-redirect" title="Hartley Rogers, Jr">Rogers</a>' characterization of <a href="/wiki/Roger%27s_equivalence_theorem" class="mw-redirect" title="Roger's equivalence theorem">acceptable programming systems</a>, Rice's Theorem may essentially be generalized from Turing machines to most computer <a href="/wiki/Programming_language" title="Programming language">programming languages</a>: there exists no automatic method that decides with generality non-trivial questions on the behavior of computer programs.</p>/n<p>As an example, consider the following variant of the <a href="/wiki/Halting_problem" title="Halting problem">halting problem</a>. Let <i>P</i> be the following property of partial functions <b>F</b> of one argument: <i>P</i>(<b>F</b>) means that <b>F</b> is defined for the argument '1'. It is obviously non-trivial, since there are partial functions that are defined at 1, and others that are undefined at 1. The <i>1-halting problem</i> is the problem of deciding of any algorithm whether it defines a function with this property, i.e., whether the algorithm halts on input 1. By Rice's theorem, the 1-halting problem is undecidable. Similarly the question of whether a Turing machine <i>T</i> terminates on an initially empty tape (rather than with an initial word <i>w</i> given as second argument in addition to a description of <i>T</i>, as in the full halting problem) is still undecidable.</p>/n<h2><span class="mw-headline" id="Formal_statement">Formal statement</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Rice%27s_theorem&amp;action=edit&amp;section=2" title="Edit section: Formal statement">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Let <span class="mwe-math-element"><span class="mwe-math-mathml-inline mwe-math-mathml-a11y" style="display: none;"><math xmlns="http://www.w3.org/1998/Math/MathML" >/n  <semantics>/n    <mrow class="MJX-TeXAtom-ORD">/n      <mstyle displaystyle="true" scriptlevel="0">/n        <mi>&#x03D5;<!-- ϕ --></mi>/n        <mo>&#x003A;<!-- : --></mo>/n        <mrow class="MJX-TeXAtom-ORD">/n          <mi mathvariant="double-struck">N</mi>/n        </mrow>/n        <mo stretchy="false">&#x2192;<!-- → --></mo>/n        <msup>/n          <mrow class="MJX-TeXAtom-ORD">/n            <mi mathvariant="bold">P</mi>/n          </mrow>/n          <mrow class="MJX-TeXAtom-ORD">/n            <mo stretchy="false">(</mo>/n            <mn>1</mn>/n            <mo stretchy="false">)</mo>/n          </mrow>/n        </msup>/n      </mstyle>/n    </mrow>/n    <annotation encoding="application/x-tex">{\displaystyle \phi \colon \mathbb {N} \to \mathbf {P} ^{(1)}}</annotation>/n  </semantics>/n</math></span><img src="https://wikimedia.org/api/rest_v1/media/math/render/svg/45c2980238b410ffd0a47beab0478af56c5ec44d" class="mwe-math-fallback-image-inline" aria-hidden="true" style="vertical-align: -0.671ex; The proof works just as well if we have an algorithm for deciding any other nontrivial property of programs, and is given in general below.</p>/n<p>The claim is that we can convert our algorithm for identifying squaring programs into one that identifies functions that halt. We will describe an algorithm that takes inputs <i>a</i> and <i>i</i> and determines whether program <i>a</i> halts when given input <i>i</i>.</p>/n<p>The algorithm for deciding this is conceptually simple: it constructs (the description of) a new program <i>t</i> taking an argument <i>n</i>, which (1) first executes program <i>a</i> on input <i>i</i> (both <i>a</i> and <i>i</i> being hard-coded into the definition of <i>t</i>), and (2) then returns the square of <i>n</i>. If <i>a</i>(<i>i</i>) runs forever, then <i>t</i> never gets to step (2), regardless of <i>n</i>. Then clearly, <i>t</i> is a function for computing squares if and only if step (1) terminates. Since we've assumed that we can infallibly identify programs for computing squares, we can determine whether <i>t</i>, which depends on <i>a</i> and <i>i</i>, is such a program, and that for every <i>a</i> and <i>i</i>; thus we have obtained a program that decides whether program <i>a</i> halts on input <i>i</i>. Note that our halting-decision algorithm never executes <i>t</i>, but only passes its description to the squaring-identification program, which by assumption always terminates; since the construction of the description of <i>t</i> can also be done in a way that always terminates, the halting-decision cannot fail to halt either.</p>/n<pre>/n halts (a,i) {/n   define t(n) {/n     a(i)/n     <b>return</b> n×n/n   }/n   <b>return</b> is_a_squaring_function(t)/n }/n</pre>/n<p>This method doesn't depend specifically on being able to recognize functions that compute squares; as long as <i>some</i> program can do what we're trying to recognize, we can add a call to <i>a</i> to obtain our <i>t</i>. We could have had a method for recognizing programs for computing square roots, or programs for computing the monthly payroll, or programs that halt when given the input <tt>"Abraxas"</tt>; in each case, we would be able to solve the halting problem similarly.</p>/n<h3><span class="mw-headline" id="Formal_proof">Formal proof</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Rice%27s_theorem&amp;action=edit&amp;section=7" title="Edit section: Formal proof">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div class="thumb tright">/n<div class="thumbinner" style="width:222px;"><a href="/wiki/File:Rice_reduction.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/3/3a/Rice_reduction.svg/220px-Rice_reduction.svg.png" width="220" height="264" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/3/3a/Rice_reduction.svg/330px-Rice_reduction.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/3/3a/Rice_reduction.svg/440px-Rice_reduction.svg.png 2x" data-file-width="500" data-file-height="600" /></a>/n<div class="thumbcaption">/n<div class="magnify"><a href="/wiki/File:Rice_reduction.svg" class="internal" title="Enlarge"></a></div>/nIf we have an algorithm that decides a non-trivial property, we can construct a Turing machine that decides the halting problem.</div>/n</div>/n</div>/n<p>For the formal proof, algorithms are presumed to define partial functions over <a href="/wiki/String_(computer_science)" title="String (computer science)">strings</a> and are themselves represented by strings. The partial function computed by the algorithm represented by a string <i>a</i> is denoted <b>F</b><sub><i>a</i></sub>. This proof proceeds by <a href="/wiki/Reductio_ad_absurdum" title="Reductio ad absurdum">reductio ad absurdum</a>: we assume that there is a non-trivial property that is decided by an algorithm, and then show that it follows that we can decide the <a href="/wiki/Halting_problem" title="Halting problem">halting problem</a>, which is not possible, and therefore a contradiction.</p>/n<p>Let us now assume that <i>P</i>(<i>a</i>) is an algorithm that decides some non-trivial property of <b>F</b><sub><i>a</i></sub>. Without loss of generality we may assume that <i>P</i>(<i>no-halt</i>) = "no", with <i>no-halt</i> being the representation of an algorithm that never halts. If this is not true, then this holds for the negation of the property. Since <i>P</i> decides a non-trivial property, it follows that there is a string <i>b</i> that represents an algorithm and <i>P</i>(<i>b</i>) = "yes". We can then define an algorithm <i>H</i>(<i>a</i>, <i>i</i>) as follows:</p>/n<dl>/n<dd>1. construct a string <i>t</i> that represents an algorithm <i>T</i>(<i>j</i>) such that/n<ul>/n<li><i>T</i> first simulates the computation of <b>F</b><sub><i>a</i></sub>(<i>i</i>)</li>/n<li>then <i>T</i> simulates the computation of <b>F</b><sub><i>b</i></sub>(<i>j</i>) and returns its result.</li>/n</ul>/n</dd>/n<dd>2. return <i>P</i>(<i>t</i>).</dd>/n</dl>/n<p>We can now show that <i>H</i> decides the halting problem:</p>/n<ul>/n<li>Assume that the algorithm represented by <i>a</i> halts on input <i>i</i>. In this case <b>F</b><sub><i>t</i></sub> = <b>F</b><sub><i>b</i></sub> and, because <i>P</i>(<i>b</i>) = "yes" and the output of <i>P</i>(<i>x</i>) depends only on <b>F</b><sub><i>x</i></sub>, it follows that <i>P</i>(<i>t</i>) = "yes" and, therefore <i>H</i>(<i>a</i>, <i>i</i>) = "yes".</li>/n<li>Assume that the algorithm represented by <i>a</i> does not halt on input <i>i</i>.
