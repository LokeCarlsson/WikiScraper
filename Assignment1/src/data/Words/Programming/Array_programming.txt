as <a href="/wiki/Vector_(computing)" class="mw-redirect" title="Vector (computing)">vector</a> or <b>multidimensional</b> languages) generalize operations on <a href="/wiki/Scalar_(computing)" class="mw-redirect" title="Scalar (computing)">scalars</a> to apply transparently to <a href="/wiki/Vector_(geometric)" class="mw-redirect" title="Vector (geometric)">vectors</a>, <a href="/wiki/Matrix_(mathematics)" title="Matrix (mathematics)">matrices</a>, and higher-dimensional arrays.</p>/n<p>Array programming primitives concisely express broad ideas about data manipulation. The level of concision can be dramatic in certain cases: it is not uncommon to find array programming language <a href="/wiki/One-liner_program" title="One-liner program">one-liners</a> that require more than a couple of pages of Java code.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup></p>/n<p>Modern programming languages that support array programming are commonly used in <a href="/wiki/Computational_science" title="Computational science">scientific</a> and engineering settings; these include <a href="/wiki/Fortran_90" class="mw-redirect" title="Fortran 90">Fortran 90</a>, Mata, <a href="/wiki/MATLAB" title="MATLAB">MATLAB</a>, <a href="/wiki/Analytica_(software)" title="Analytica (software)">Analytica</a>, <a href="/wiki/TK_Solver" title="TK Solver">TK Solver</a> (as lists), <a href="/wiki/GNU_Octave" title="GNU Octave">Octave</a>, <a href="/wiki/R_(programming_language)" title="R (programming language)">R</a>, <a href="/wiki/Cilk_Plus" class="mw-redirect" title="Cilk Plus">Cilk Plus</a>, <a href="/wiki/Julia_(programming_language)" title="Julia (programming language)">Julia</a>, <a href="/wiki/Perl_Data_Language" title="Perl Data Language">Perl Data Language (PDL)</a> and the <a href="/wiki/NumPy" title="NumPy">NumPy</a> extension to <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>. In these languages, an operation that operates on entire arrays can be called a <b>vectorized</b> operation,<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> regardless of whether it is executed on a <a href="/wiki/Vector_processor" title="Vector processor">vector processor</a> or not.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Array_programming&amp;action=edit&amp;section=1" title="Edit section: Concepts">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The fundamental idea behind array programming is that operations apply at once to an entire set of values. This makes it a <a href="/wiki/High-level_programming_language" title="High-level programming language">high-level programming</a> model as it allows the programmer to think and operate on whole aggregates of data, without having to resort to explicit loops of individual scalar operations.</p>/n<p>Iverson described the rationale behind array programming (actually referring to APL) as follows:<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup></p>/n<blockquote class="templatequote">/n<p>most programming languages are decidedly inferior to mathematical notation and are little used as tools of thought in ways that would be considered significant by, say, an applied mathematician. [...]</p>/n<p>The thesis [...] is that the advantages of executability and universality found in programming languages can be effectively combined, in a single coherent language, with the advantages offered by mathematical notation. [...] it is important to distinguish the difficulty of describing and of learning a piece of notation from the difficulty of mastering its implications. For example, learning the rules for computing a matrix product is easy, but a mastery of its implications (such as its associativity, its distributivity over addition, and its ability to represent linear functions and geometric operations) is a different and much more difficult matter.</p>/n<p>Indeed, the very suggestiveness of a notation may make it seem harder to learn because of the many properties it suggests for explorations.</p>/n<p>[...] Users of computers and programming languages are often concerned primarily with the efficiency of execution of algorithms, and might, therefore, summarily dismiss many of the algorithms presented here. Such dismissal would be short-sighted, since a clear statement of an algorithm can usually be used as a basis from which one may easily derive more efficient algorithm.</p>/n</blockquote>/n<p>The basis behind array programming and thinking is to find and exploit the properties of data where individual elements are similar or adjacent. Unlike object orientation which implicitly breaks down data to its constituent parts (or <a href="/wiki/Scalar_(computing)" class="mw-redirect" title="Scalar (computing)">scalar</a> quantities), array orientation looks to group data and apply a uniform handling.</p>/n<p><a href="/w/index.php?title=Function_rank&amp;action=edit&amp;redlink=1" class="new" title="Function rank (page does not exist)">Function rank</a> is an important concept to array programming languages in general, by analogy to <a href="/wiki/Tensor" title="Tensor">tensor</a> rank in mathematics: functions that operate on data may be classified by the number of dimensions they act on. Ordinary multiplication, for example, is a scalar ranked function because it operates on zero-dimensional data (individual numbers). The <a href="/wiki/Cross_product" title="Cross product">cross product</a> operation is an example of a vector rank function because it operates on vectors, not scalars. <a href="/wiki/Matrix_multiplication" title="Matrix multiplication">Matrix multiplication</a> is an example of a 2-rank function, because it operates on 2-dimensional objects (matrices). <a href="/wiki/Reduce_(higher-order_function)" class="mw-redirect" title="Reduce (higher-order function)">Collapse operators</a> reduce the dimensionality of an input data array by one or more dimensions. For example, summing over elements collapses the input array by 1 dimension.</p>/n<h2><span class="mw-headline" id="Uses">Uses</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Array_programming&amp;action=edit&amp;section=2" title="Edit section: Uses">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Array programming is very well suited to implicit parallelization; a topic of much research nowadays. class="m">30</span>/n</pre></div>/n<h2><span class="mw-headline" id="Mathematical_reasoning_and_language_notation">Mathematical reasoning and language notation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Array_programming&amp;action=edit&amp;section=14" title="Edit section: Mathematical reasoning and language notation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The matrix left-division operator concisely expresses some semantic properties of matrices. As in the scalar equivalent, if the (<a href="/wiki/Determinant" title="Determinant">determinant</a> of the) coefficient (matrix) <code>A</code> is not null then it is possible to solve the (vectorial) equation <code>A * x = b</code> by left-multiplying both sides by the <a href="/wiki/Matrix_inversion" class="mw-redirect" title="Matrix inversion">inverse</a> of <code>A</code>: <code>A<sup>−1</sup></code> (in both MATLAB and GNU Octave languages: <code>A^-1</code>). The following mathematical statements hold when <code>A</code> is a <a href="/wiki/Matrix_rank#Properties" class="mw-redirect" title="Matrix rank">full rank</a> <a href="/wiki/Square_matrix#Square_matrices" title="Square matrix">square matrix</a>:</p>/n<dl>/n<dd><code>A^-1 *(A * x)==A^-1 * (b)</code></dd>/n<dd><code>(A^-1 * A)* x ==A^-1 * b</code> &#160; &#160; &#160; (matrix-multiplication <a href="/wiki/Associativity" class="mw-redirect" title="Associativity">associativity</a>)</dd>/n<dd><code>x = A^-1 * b</code></dd>/n</dl>/n<p>where <code>==</code> is the equivalence <a href="/wiki/Relational_operator" title="Relational operator">relational operator</a>. The previous statements are also valid MATLAB expressions if the third one is executed before the others (numerical comparisons may be false because of round-off errors).</p>/n<p>If the system is overdetermined - so that <code>A</code> has more rows than columns - the pseudoinverse <code>A<sup>+</sup></code> (in MATLAB and GNU Octave languages: <code>pinv(A)</code>) can replace the inverse <code>A<sup>−1</sup></code>, as follows:</p>/n<dl>/n<dd><code>pinv(A) *(A * x)==pinv(A) * (b)</code></dd>/n<dd><code>(pinv(A) * A)* x ==pinv(A) * b</code> &#160; &#160; &#160; (matrix-multiplication <a href="/wiki/Associativity" class="mw-redirect" title="Associativity">associativity</a>)</dd>/n<dd><code>x = pinv(A) * b</code></dd>/n</dl>/n<p>However, these solutions are neither the most concise ones (e.g. still remains the need to notationally differentiate overdetermined systems) nor the most computationally efficient. The latter point is easy to understand when considering again the scalar equivalent <code>a * x = b</code>, for which the solution <code>x = a^-1 * b</code> would require two operations instead of the more efficient <code>x = b / a</code>. The problem is that generally matrix multiplications are not <a href="/wiki/Commutativity" class="mw-redirect" title="Commutativity">commutative</a> as the extension of the scalar solution to the matrix case would require:</p>/n<dl>/n<dd><code>(a * x)/ a ==b / a</code></dd>/n<dd><code>(x * a)/ a ==b / a</code> &#160; &#160; &#160; (commutativity does not hold for matrices!)</dd>/n<dd><code>x * (a / a)==b / a</code> &#160; &#160; &#160; (associativity also holds for matrices)</dd>/n<dd><code>x = b / a</code></dd>/n</dl>/n<p>The MATLAB language introduces the left-division operator <code>\</code> to maintain the essential part of the analogy with the scalar case, therefore simplifying the mathematical reasoning and preserving the conciseness:</p>/n<dl>/n<dd><code>A \ (A * x)==A \ b</code></dd>/n<dd><code>(A \ A)* x ==A \ b</code> &#160; &#160; &#160; (associativity also holds for matrices, commutativity is no more required)</dd>/n<dd><code>x = A \ b</code></dd>/n</dl>/n<p>This is not only an example of terse array programming from the coding point of view but also from the computational efficiency perspective, which in several array programming languages benefits from quite efficient linear algebra libraries such as <a href="/wiki/Automatically_Tuned_Linear_Algebra_Software" title="Automatically Tuned Linear Algebra Software">ATLAS</a> or <a href="/wiki/LAPACK" title="LAPACK">LAPACK</a>.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup><sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup></p>/n<p>Returning to the previous quotation of Iverson, the rationale behind it should now be evident:</p>/n<blockquote class="templatequote">/n<p>it is important to distinguish the difficulty of describing and of learning a piece of notation from the difficulty of mastering its implications. For example, learning the rules for computing a matrix product is easy, but a mastery of its implications (such as its associativity, its distributivity over addition, and its ability to represent linear functions and geometric operations) is a different and much more difficult matter.</p>/n<p>Indeed, the very suggestiveness of a notation may make it seem harder to learn because of the many properties it suggests for explorations.</p>/n</blockquote>/n<h2><span class="mw-headline" id="Third-party_libraries">Third-party libraries</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Array_programming&amp;action=edit&amp;section=15" title="Edit section: Third-party libraries">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The use of specialized and efficient libraries to provide more terse abstractions is also common in other programming languages. In <a href="/wiki/C%2B%2B" title="C++">C++</a> several linear algebra libraries exploit the language ability to <a href="/wiki/Operator_overloading" title="Operator overloading">overload operators</a>. In some cases a very terse abstraction in those languages is explicitly influenced by the array programming paradigm, as the <a href="/wiki/Armadillo_(C%2B%2B_library)" title="Armadillo (C++ library)">Armadillo</a> and <a href="/wiki/Blitz%2B%2B" title="Blitz++">Blitz++</a> libraries do.<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[12]</a></sup><sup id="cite_ref-13" class="reference"><a href="#cite_note-13">[13]</a></sup></p>/n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Array_programming&amp;action=edit&amp;section=16" title="Edit section: See also">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<ul>/n<li><a href="/wiki/Array_slicing" title="Array slicing">Array slicing</a></li>/n<li><a href="/wiki/List_of_programming_languages_by_type#Array_languages" title="List of programming languages by type">List of array programming languages</a></li>/n</ul>/n<h2><span class="mw-headline" id="References">References</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Array_programming&amp;action=edit&amp;section=17" title="Edit section: References">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div class="reflist columns references-column-width" style="-moz-column-width: 30em; -webkit-column-width: 30em; column-width: 30em; list-style-type: decimal;">/n<ol class="references">/n<li id="cite_note-1"><span class="mw-cite-backlink"><b><a href="#cite_ref-1">^</a></b></span> <span class="reference-text"><cite class="citation web">Michael Schidlowsky. <a rel="nofollow" class="external text" href="http://www.cs.nyu.edu/~michaels/screencasts/Java_vs_K/Java_vs_K.html">"Java and K"</a><span class="reference-accessdate">.
