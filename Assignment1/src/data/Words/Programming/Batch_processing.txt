processing</b> is the execution of a series of jobs in a <a href="/wiki/Computer_program" title="Computer program">program</a> on a <a href="/wiki/Computer" title="Computer">computer</a> without manual intervention (<a href="/wiki/Non-interactive" class="mw-redirect" title="Non-interactive">non-interactive</a>). Strictly speaking, it is a <a href="/wiki/Processing_mode" title="Processing mode">processing mode</a>: the execution of a series of programs each on a set or "batch" of inputs, rather than a <i>single</i> input (which would instead be a custom <i><a href="/wiki/Job_(computing)" title="Job (computing)">job</a></i>). However, this distinction has largely been lost, and the series of steps in a batch process are often called a "job" or "batch job".</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=1" title="Edit section: Benefits">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Batch processing has these benefits:</p>/n<ul>/n<li>It can shift the time of job processing to when the computing resources are less busy.</li>/n<li>It avoids idling the computing resources with minute-by-minute manual intervention and supervision.</li>/n<li>By keeping high overall rate of utilization, it <a href="/wiki/Amortization" title="Amortization">amortizes</a> the computer, especially an expensive one.</li>/n<li>It allows the system to use different <a href="/wiki/Scheduling_(computing)" title="Scheduling (computing)">priorities</a> for interactive and non-interactive work.</li>/n<li>Rather than running one program multiple times to process one transaction each time, batch processes will run the program only once for many transactions, reducing system overhead.</li>/n</ul>/n<p>It also has multiple disadvantages, for instance users are unable to terminate a process during execution, and have to wait until execution completes.</p>/n<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The term "batch processing" originates in the traditional classification of <a href="/wiki/Methods_of_production" title="Methods of production">methods of production</a> as <a href="/wiki/Job_production" title="Job production">job production</a> (one-off production), <a href="/wiki/Batch_production" title="Batch production">batch production</a> (production of a "batch" of multiple items at once, one stage at a time), and <a href="/wiki/Flow_production" class="mw-redirect" title="Flow production">flow production</a> (mass production, all stages in process at once).</p>/n<h3><span id="Early_history_(19th_century_through_1960s)"></span><span class="mw-headline" id="Early_history_.2819th_century_through_1960s.29">Early history (19th century through 1960s)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=3" title="Edit section: Early history (19th century through 1960s)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div class="thumb tright">/n<div class="thumbinner" style="width:222px;"><a href="/wiki/File:HollerithMachine.CHM.jpg" processing of punch cards (in stack on each machine) with human operators at U.S. Social Security Administration</div>/n</div>/n</div>/n<p>Batch processing dates to the late 19th century, in the processing of data stored on decks of <a href="/wiki/Punch_card" class="mw-redirect" title="Punch card">punch card</a> by <a href="/wiki/Unit_record_equipment" title="Unit record equipment">unit record equipment</a>, specifically the <a href="/wiki/Tabulating_machine" title="Tabulating machine">tabulating machine</a> by <a href="/wiki/Herman_Hollerith" title="Herman Hollerith">Herman Hollerith</a>, used for the <a href="/wiki/1890_United_States_Census" title="1890 United States Census">1890 United States Census</a>. This was the earliest use of a machine-readable medium for data, rather than for control (as in <a href="/wiki/Jacquard_loom" title="Jacquard loom">Jacquard looms</a>; today <i>control</i> corresponds to <i>code</i>), and thus the earliest processing of machine-read data was batch processing. Each card stored a separate <a href="/wiki/Record_(computer_science)" title="Record (computer science)">record</a> of data with different fields: cards were processed by the machine one by one, all in the same way, as a batch. Batch processing continued to be the dominant processing mode on <a href="/wiki/Mainframe_computer" title="Mainframe computer">mainframe computers</a> from the earliest days of electronic computing in the 1950s.</p>/n<p>Originally machines only tabulated data, counting records with certain properties, like "male" or "female". In later use, separate stages or "cycles" of processing could be done, analogous to the stages in batch production. In modern data processing terms, one can think of each stage as an <a href="/wiki/SQL" title="SQL">SQL</a> clause, such as SELECT (filter columns), then WHERE (filter cards, or "rows"), etc. The earliest machines were built (hard-wired) for a single function, while from 1906 they could be rewired via <a href="/wiki/Plugboard" title="Plugboard">plugboards</a>, and electronic computers could be reprogrammed without being rewired. Thus early multi-stage processing required separate machines for each stage, or rewiring a single machine after each stage. Early electronic computers were not capable of having multiple programs loaded into main memory (<a href="/wiki/Multiprogramming" class="mw-redirect" title="Multiprogramming">multiprogramming</a>), and thus while they could process multiple stages on a single machine without rewiring, the program for each stage had to be loaded into memory, run over the entire batch, and then the program for the next loaded and run.</p>/n<p>There were a variety of reasons why batch processing dominated early computing. One reason is that the most urgent business problems for reasons of profitability and competitiveness were primarily accounting problems, such as billing or payroll; this priority of accounting in early use of information technology is ancient: see <a href="/wiki/History_of_writing" title="History of writing">history of writing</a> and <a href="/wiki/History_of_accounting" title="History of accounting">history of accounting</a>. Billing may conveniently be performed as a batch-oriented business process, and practically every business must bill, reliably and on-time. While accounting is time-sensitive, it can be done daily (particularly overnight, after close of business), and does not require interaction. Also, every computing resource was expensive, so sequential submission of batches on <a href="/wiki/Punched_cards" class="mw-redirect" title="Punched cards">punched cards</a> matched the resource constraints and technology evolution at the time.</p>/n<p>Batch data processing took advantage of the economies of scale in <a href="/wiki/Sorting" title="Sorting">sorting</a> and processing sequential data storage media, such as punch cards and, later, magnetic tape. Typically transactions for a recording period, such as a day or a week, would be entered onto cards from paper forms using a <a href="/wiki/Keypunch" title="Keypunch">keypunch</a> machine. At the close of the period, the data would be sorted using a <a href="/wiki/IBM_card_sorter" title="IBM card sorter">card sorting machine</a>, or, later a computer. The sorted data would then be used to update a master file, such as an accounting ledger or inventory file, that was kept sorted by the same key. Only one pass through the sequential files would be needed for the updates. Reports and other outputs, such as bills and payment checks, would then be generated from the master file.</p>/n<p>In addition to batches of data, early electronic computers could also run one-off computations, notably compilation (see <a href="/wiki/History_of_compiler_construction" title="History of compiler construction">history of compiler construction</a>). These were accordingly referred to as <a href="/wiki/Job_(computing)" title="Job (computing)">jobs</a>, as they were one-off processing, and were controlled by languages such as <a href="/wiki/Job_Control_Language" title="Job Control Language">Job Control Language</a> (JCL). However, this distinction between jobs and batches later became blurred with the advent of interactive computing.</p>/n<h3><span id="Later_history_(1960s_onwards)"></span><span class="mw-headline" id="Later_history_.281960s_onwards.29">Later history (1960s onwards)</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=4" title="Edit section: Later history (1960s onwards)">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>From the late 1960s onwards, <a href="/wiki/Interactive_computing" title="Interactive computing">interactive computing</a> such as via text-based <a href="/wiki/Computer_terminal" title="Computer terminal">computer terminal</a> interfaces (as in <a href="/wiki/Unix_shell" title="Unix shell">Unix shells</a> or <a href="/wiki/Read-eval-print_loop" class="mw-redirect" title="Read-eval-print loop">read-eval-print loops</a>), and later <a href="/wiki/Graphical_user_interface" title="Graphical user interface">graphical user interfaces</a> became common. Non-interactive computation, both one-off jobs such as compilation, and processing of multiple items in batches, became retrospectively referred to as <i>batch processing</i>, and the oxymoronic term <i><a href="/wiki/Batch_job" class="mw-redirect" title="Batch job">batch job</a></i> (in early use often "batch <i>of</i> jobs") became common. Early use is particularly found at the <a href="/wiki/University_of_Michigan" title="University of Michigan">University of Michigan</a>, around the <a href="/wiki/Michigan_Terminal_System" title="Michigan Terminal System">Michigan Terminal System</a> (MTS); examples from 1968 and 1969:</p>/n<blockquote class="templatequote">/n<p>Only the compilation and execution of a FORTRAN program as a batch "job" will be described in this section. The term "batch processing" refers to the processing of many jobs (a "batch") in sequence from card input. Each job in the batch is completely processed before the next is begun.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup></p>/n</blockquote>/n<blockquote class="templatequote">/n<p>BATCH MODE, BATCH JOB — A process or task prepared and presented in its entirety, as opposed to an interaction at a remote terminal by a user who issues commands often based on the computer's response to previous commands. This latter mode of interaction is called conversational. Batch-mode jobs are submitted as decks of punched cards which are read into the computer in groups (batches).<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup></p>/n</blockquote>/n<p>Non-interactive computation remains pervasive in computing, both for general data processing and for system "housekeeping" tasks (using <a href="/wiki/System_software" title="System software">system software</a>). A high-level program (executing multiple programs, with some additional "glue" logic) is today most often called a <i>script</i>, and written in <a href="/wiki/Scripting_language" title="Scripting language">scripting languages</a>, particularly <a href="/wiki/Shell_script" title="Shell script">shell scripts</a> for system tasks; however, in DOS this is instead known as a <a href="/wiki/Batch_file" title="Batch file">batch file</a>. That includes <a href="/wiki/UNIX" class="mw-redirect" title="UNIX">UNIX</a>-based computers, <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a>, <a href="/wiki/MacOS" title="MacOS">macOS</a> (whose foundation is the <a href="/wiki/BSD" class="mw-redirect" title="BSD">BSD</a> Unix kernel), and even <a href="/wiki/Smartphones" class="mw-redirect" title="Smartphones">smartphones</a>. A running script, particularly one executed from an interactive <a href="/wiki/Login_session" title="Login session">login session</a>, is often known as a <a href="/wiki/Job_(Unix)" class="mw-redirect" title="Job (Unix)">job</a>, but that term is used very ambiguously.</p>/n<p>Batch processing narrowly speaking (processing multiple records through stage, one stage at a time) is still pervasive in <a href="/wiki/Mainframe_computing" class="mw-redirect" title="Mainframe computing">mainframe computing</a>, but is less common in interactive online networked systems, particularly in <a href="/wiki/Client%E2%80%93server" class="mw-redirect" title="Client–server">client–server</a> systems such as the <a href="/wiki/Request%E2%80%93response" title="Request–response">request–response</a> messages of <a href="/wiki/Web_server" title="Web server">web servers</a>. These systems instead function as <i>flow</i> processing, where for each task messages are passed between <a href="/wiki/Server_(computing)" title="Server (computing)">servers</a>, all servers working at once on different stages of different tasks. Even in non-networked settings, flow processing is common, specifically as <a href="/wiki/Pipeline_(software)" title="Pipeline (software)">pipelines</a> of connected processes, <a href="/wiki/Concurrency_(computer_science)" title="Concurrency (computer science)">concurrently</a> processing like an assembly line.</p>/n<p>Where batch processing remains in use, the outputs of separate stages (and input for the subsequent stage) are typically stored as <a href="/wiki/Computer_file" title="Computer file">files</a>. This is often used for ease of development and debugging, as it allows intermediate data to be reused or inspected. For example, to process data using two program <code>step1</code> and <code>step2</code>, one might get initial data from a file <code>input</code>, and store the ultimate result in a file <code>output</code>. Via batch processing, one can use an intermediate file, <code>intermediate</code>, and run the steps in sequence (Unix syntax):</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/nstep1 &lt; input &gt; intermediate/nstep2 &lt; intermediate &gt; output/n</pre></div>/n<p>This batch processing can be replaced with a flow: the intermediary file can be elided with a <a href="/wiki/Pipe_(Unix)" class="mw-redirect" title="Pipe (Unix)">pipe</a>, feeding output from one step to the next as it becomes available:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/nstep1 &lt; input <span class="p">|</span> step2 &gt; output/n</pre></div>/n<h2><span class="mw-headline" id="Modern_systems">Modern systems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=5" title="Edit section: Modern systems">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Batch applications are still critical in most organizations in large part because many common business processes are amenable to batch processing. While online systems can also function when manual intervention is not desired, they are not typically optimized to perform high-volume, repetitive tasks. Therefore, even new systems usually contain one or more batch applications for updating information at the end of the day, generating reports, printing documents, and other non-interactive tasks that must complete reliably within certain business deadlines.</p>/n<p>Some applications are amenable to flow processing, namely those that only need data from a single input at once (not totals, for instance): start the next step for each input as it completes the previous step. In this case flow processing lowers <a href="/wiki/Latency_(engineering)" title="Latency (engineering)">latency</a> for individual inputs, allowing them to be completed without waiting for the entire batch to finish. However, many applications require data from all records, notably computations such as totals. In this case the entire batch must be completed before one has a usable result: partial results are not usable.</p>/n<p>Modern batch applications make use of modern batch frameworks such as <a href="/wiki/Jem_The_Bee" title="Jem The Bee">Jem The Bee</a>, <a href="/wiki/Spring_Batch" title="Spring Batch">Spring Batch</a> or implementations of <a href="/wiki/Java_Specification_Request" class="mw-redirect" title="Java Specification Request">JSR</a> 352<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup> written for <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, and other frameworks for other programming languages, to provide the <a href="/wiki/Fault_tolerance" title="Fault tolerance">fault tolerance</a> and <a href="/wiki/Scalability" title="Scalability">scalability</a> required for high-volume processing. In order to ensure high-speed processing, batch applications are often integrated with <a href="/wiki/Grid_computing" title="Grid computing">grid computing</a> solutions to <a href="/wiki/Partition_of_a_set" title="Partition of a set">partition</a> a batch job over a large number of processors, although there are significant programming challenges in doing so. High volume batch processing places particularly heavy demands on system and application architectures as well. Architectures that feature strong <a href="/wiki/Input/output" title="Input/output">input/output</a> performance and vertical <a href="/wiki/Scalability" title="Scalability">scalability</a>, including modern <a href="/wiki/Mainframe_computers" class="mw-redirect" title="Mainframe computers">mainframe computers</a>, tend to provide better batch performance than alternatives.</p>/n<p><a href="/wiki/Scripting_languages" class="mw-redirect" title="Scripting languages">Scripting languages</a> became popular as they evolved along with batch processing.</p>/n<h2><span class="mw-headline" id="Batch_window">Batch window</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=6" title="Edit section: Batch window">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>A <i>batch window</i> is "a period of less-intensive online activity",<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> when the computer system is able to run batch jobs without interference from online systems.</p>/n<p>Many early computer systems offered only batch processing, so jobs could be run any time within a 24-hour day. With the advent of <a href="/wiki/Transaction_processing" title="Transaction processing">transaction processing</a> the online applications might only be required from 9:00&#160;a.m. to 5:00&#160;p.m., leaving two shifts available for batch work, in this case the batch window would be sixteen hours. The problem is not usually that the computer system is incapable of supporting concurrent online and batch work, but that the batch systems usually require access to data in a consistent state, free from online updates until the batch processing is complete.</p>/n<p>In a bank, for example, so-called <i>end-of-day (EOD)</i> jobs include interest calculation, generation of reports and data sets to other systems, printing statements, and payment processing. This coincides with the concept of Cutover, where transaction and data are cut off for a particular day's batch activity and any further data is contributed to the next following day's batch activity (this is the reason for messages like "deposits after 3 PM will be processed the next day").</p>/n<p>The batch window is further complicated by the actual run-time of a particular batch activity. Some batches in banking can take between 5-9 hours of run time, coupled with global constraints some batch activity is broken up or even stalled to allow periodic use of databases mid batch (usually in read-only) to support automated testing scripts that may run in the evening hours or outsourced\contract testing and development resources abroad. More complex problems arise when institutions both have batch activities that may be dependent meaning both batches have to complete in the same batch window.</p>/n<p>As requirements for online systems uptime expanded to support <a href="/wiki/Globalization" title="Globalization">globalization</a>, the <a href="/wiki/Internet" title="Internet">Internet</a>, and other business requirements the batch window shrank and increasing emphasis was placed on techniques that would require online data to be available for a maximum amount of time.</p>/n<h2><span class="mw-headline" id="Common_batch_processing_usage">Common batch processing usage</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=7" title="Edit section: Common batch processing usage">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<h3><span class="mw-headline" id="Databases">Databases</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=8" title="Edit section: Databases">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Batch processing is also used for efficient bulk database updates and automated <a href="/wiki/Transaction_processing" title="Transaction processing">transaction processing</a>, as contrasted to interactive <a href="/wiki/Online_transaction_processing" title="Online transaction processing">online transaction processing</a> (OLTP) applications. The <a href="/wiki/Extract,_transform,_load" title="Extract, transform, load">extract, transform, load</a> (ETL) step in populating <a href="/wiki/Data_warehouses" class="mw-redirect" title="Data warehouses">data warehouses</a> is inherently a batch process in most implementations.</p>/n<h3><span class="mw-headline" id="Images">Images</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=9" title="Edit section: Images">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Batch processing is often used to perform various operations with <a href="/wiki/Digital_image" title="Digital image">digital images</a> such as resize, convert, watermark, or otherwise edit image files.</p>/n<h3><span class="mw-headline" id="Conversions">Conversions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=10" title="Edit section: Conversions">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Batch processing may also be used for converting computer files from one format to another. For example, a batch job may convert proprietary and legacy files to common standard formats for end-user queries and display.</p>/n<h2><span class="mw-headline" id="Notable_batch_scheduling_and_execution_environments">Notable batch scheduling and execution environments</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=11" title="Edit section: Notable batch scheduling and execution environments">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The Unix programs <code><a href="/wiki/Cron_(Unix)" class="mw-redirect" title="Cron (Unix)">cron</a></code>, <code><a href="/wiki/At_(Unix)" title="At (Unix)">at</a></code>, and <code><a href="/wiki/Batch_(Unix)" class="mw-redirect" title="Batch (Unix)">batch</a></code> (today <code>batch</code> is a variant of <code>at</code>) allow for complex scheduling of jobs. Windows has a <a href="/wiki/Job_scheduler" title="Job scheduler">job scheduler</a>. Most <a href="/wiki/High-performance_computing" class="mw-redirect" title="High-performance computing">high-performance computing</a> <a href="/wiki/Cluster_(computing)" class="mw-redirect" title="Cluster (computing)">clusters</a> use batch processing to maximize cluster usage.</p>/n<p>The <a href="/wiki/IBM_mainframe" title="IBM mainframe">IBM mainframe</a> <a href="/wiki/Z/OS" title="Z/OS">z/OS</a> <a href="/wiki/Operating_system" title="Operating system">operating system</a> or platform has arguably the most highly refined and evolved set of batch processing facilities owing to its origins, long history, and continuing evolution. Today such systems commonly support hundreds or even thousands of concurrent online and batch tasks within a single <a href="/wiki/Operating_system" title="Operating system">operating system</a> image. Technologies that aid concurrent batch and online processing include <a href="/wiki/Job_Control_Language" title="Job Control Language">Job Control Language</a> (JCL), scripting languages such as <a href="/wiki/REXX" class="mw-redirect" title="REXX">REXX</a>, Job Entry Subsystem (<a href="/wiki/JES2" class="mw-redirect" title="JES2">JES2</a> and <a href="/wiki/JES3" class="mw-redirect" title="JES3">JES3</a>), <a href="/wiki/Workload_Manager" title="Workload Manager">Workload Manager</a> (WLM), Automatic Restart Manager (ARM), Resource Recovery Services (RRS), <a href="/wiki/IBM_DB2" class="mw-redirect" title="IBM DB2">DB2</a> data sharing, <a href="/wiki/Parallel_Sysplex" class="mw-redirect" title="Parallel Sysplex">Parallel Sysplex</a>, unique performance optimizations such as <a href="/wiki/HiperDispatch" title="HiperDispatch">HiperDispatch</a>, <a href="/wiki/Channel_I/O" title="Channel I/O">I/O channel architecture</a>, and several others.</p>/n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Batch_processing&amp;action=edit&amp;section=12" title="Edit section: See also">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<ul>/n<li><a href="/wiki/Background_process" title="Background process">Background process</a></li>/n<li><a href="/wiki/Batch_file" title="Batch file">Batch file</a></li>/n<li><a href="/wiki/Batch_queue" class="mw-redirect" title="Batch queue">Batch queue</a> -
