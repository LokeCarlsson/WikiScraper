links</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Haskell_(programming_language)&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Following the release of <a href="/wiki/Miranda_(programming_language)" title="Miranda (programming language)">Miranda</a> by Research Software Ltd, in 1985, interest in <a href="/wiki/Lazy_evaluation" title="Lazy evaluation">lazy functional languages</a> grew. By 1987, more than a dozen <a href="/wiki/Non-strict" class="mw-redirect" title="Non-strict">non-strict</a>, purely functional programming languages existed. Miranda was the most widely used, but it was <a href="/wiki/Proprietary_software" title="Proprietary software">proprietary software</a>. At the conference on <a href="/wiki/Functional_Programming_Languages_and_Computer_Architecture" class="mw-redirect" title="Functional Programming Languages and Computer Architecture">Functional Programming Languages and Computer Architecture</a> (FPCA '87) in <a href="/wiki/Portland,_Oregon" title="Portland, Oregon">Portland, Oregon</a>, there was a strong consensus that a committee be formed to define an <a href="/wiki/Open_standard" title="Open standard">open standard</a> for such languages. The committee's purpose was to consolidate existing <a href="/wiki/Functional_languages" class="mw-redirect" title="Functional languages">functional languages</a> into a common one to serve as a basis for future research in functional-language design.<sup id="cite_ref-FOOTNOTEPeyton_Jones2003Preface_37-0" class="reference"><a href="#cite_note-FOOTNOTEPeyton_Jones2003Preface-37">[37]</a></sup></p>/n<h3><span class="mw-headline" id="Haskell_1.0_to_1.4">Haskell 1.0 to 1.4</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Haskell_(programming_language)&amp;action=edit&amp;section=2" title="Edit section: Haskell 1.0 to 1.4">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>The first version of Haskell ("Haskell 1.0") was defined in 1990.<sup id="cite_ref-FOOTNOTEHudakHughesPeyton_JonesWadler2007_1-2" class="reference"><a href="#cite_note-FOOTNOTEHudakHughesPeyton_JonesWadler2007-1">[1]</a></sup> The committee's efforts resulted in a series of language definitions (1.0, 1.1, 1.2, 1.3, 1.4).</p>/n<h3><span class="mw-headline" id="Haskell_98">Haskell 98</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Haskell_(programming_language)&amp;action=edit&amp;section=3" title="Edit section: Haskell 98">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>In late 1997, the series culminated in <i>Haskell 98</i>, intended to specify a stable, minimal, portable version of the language and an accompanying standard <a href="/wiki/Library_(computer_science)" class="mw-redirect" title="Library (computer science)">library</a> for teaching, and as a base for future extensions. The committee expressly welcomed creating extensions and variants of Haskell 98 via adding and incorporating experimental features.<sup id="cite_ref-FOOTNOTEPeyton_Jones2003Preface_37-1" class="reference"><a href="#cite_note-FOOTNOTEPeyton_Jones2003Preface-37">[37]</a></sup></p>/n<p>In February 1999, the Haskell 98 language standard was originally published as <i>The Haskell 98 Report</i>.<sup id="cite_ref-FOOTNOTEPeyton_Jones2003Preface_37-2" class="reference"><a href="#cite_note-FOOTNOTEPeyton_Jones2003Preface-37">[37]</a></sup> In January 2003, a revised version was published as <i>Haskell 98 Language and Libraries: The Revised Report</i>.<sup id="cite_ref-FOOTNOTEPeyton_Jones2003_28-1" class="reference"><a href="#cite_note-FOOTNOTEPeyton_Jones2003-28">[28]</a></sup> The language continues to evolve rapidly, with the <a href="/wiki/Glasgow_Haskell_Compiler" title="Glasgow Haskell Compiler">Glasgow Haskell Compiler</a> (GHC) implementation representing the current <i>de facto</i> standard.<sup id="cite_ref-HaskellWikiImpl_38-0" class="reference"><a href="#cite_note-HaskellWikiImpl-38">[38]</a></sup></p>/n<h3><span class="mw-headline" id="Haskell_2010">Haskell 2010</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Haskell_(programming_language)&amp;action=edit&amp;section=4" title="Edit section: Haskell 2010">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>In early 2006, the process of defining a successor to the Haskell 98 standard, informally named <i>Haskell Prime</i>, began.<sup id="cite_ref-39" class="reference"><a href="#cite_note-39">[39]</a></sup> This was intended to be an ongoing incremental process to revise the language definition, producing a new revision up to once per year. The first revision, named <b>Haskell 2010</b>, was announced in November 2009<sup id="cite_ref-2010ann_2-1" class="reference"><a href="#cite_note-2010ann-2">[2]</a></sup> and published in July 2010.</p>/n<p>Haskell 2010 is an incremental update to the language, mostly incorporating several well-used and uncontroversial features previously enabled via compiler-specific flags.</p>/n<ul>/n<li>Hierarchical module names. Module names are allowed to consist of dot-separated sequences of capitalised identifiers, rather than only one such identifier. This lets modules be named in a hierarchical manner (e.g., <code>Data.List</code> instead of <code>List</code>), although technically modules are still in a single monolithic namespace. This extension was specified in an addendum to Haskell 98 and was in practice universally used.</li>/n<li>The <a href="/wiki/Foreign_function_interface" title="Foreign function interface">foreign function interface</a> (FFI) allows bindings to other programming languages. Only bindings to <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> are specified in the Report, but the design allows for other language bindings. To support this, data type declarations were permitted to contain no constructors, enabling robust nonce types for foreign data that could not be constructed in Haskell. This extension was also previously specified in an Addendum to the Haskell 98 Report and widely used.</li>/n<li>So-called <i>n</i>+<i>k</i> patterns (definitions of the form <code>fact (n+1) = (n+1) * fact n</code>) were no longer allowed. This <a href="/wiki/Syntactic_sugar" title="Syntactic sugar">syntactic sugar</a> had misleading semantics, in which the code looked like it used the <code>(+)</code> operator, but in fact desugared to code using <code>(-)</code> and <code>(&gt;=)</code>.</li>/n<li>The rules of <a href="/wiki/Type_inference" title="Type inference">type inference</a> were relaxed to allow more programs to type check.</li>/n<li>Some <a href="/wiki/Syntax_(programming_languages)" title="Syntax (programming languages)">syntax</a> issues (changes in the formal grammar) were fixed: pattern guards were added, allowing pattern matching within guards; resolution of <a href="/wiki/Order_of_operations" title="Order of operations">operator fixity</a> was specified in a simpler way that reflected actual practice; an edge case in the interaction of the language's <a href="/wiki/Lexical_syntax" class="mw-redirect" title="Lexical syntax">lexical syntax</a> of operators and comments was addressed; and the interaction of do-notation and if-then-else was tweaked to eliminate unexpected syntax errors.</li>/n<li>The <code>LANGUAGE</code> <a href="/wiki/Directive_(programming)" title="Directive (programming)">pragma</a> was specified. By 2010 dozens of extensions to the language were in wide use, and GHC (among other compilers) provided the <code>LANGUAGE</code> pragma to specify individual extensions with a list of identifiers. Haskell 2010 compilers are required to support the <code>Haskell2010</code> extension, and encouraged to support several others that correspond to extensions added in Haskell 2010.</li>/n</ul>/n<h2><span class="mw-headline" id="Features">Features</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Haskell_(programming_language)&amp;action=edit&amp;section=5" title="Edit section: Features">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Haskell_features" title="Haskell features">Haskell features</a></div>/n<div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/Glasgow_Haskell_Compiler#Extensions_to_Haskell" title="Glasgow Haskell Compiler">Glasgow Haskell Compiler §&#160;Extensions to Haskell</a></div>/n<p>Haskell features <a href="/wiki/Lazy_evaluation" title="Lazy evaluation">lazy evaluation</a>, <a href="/wiki/Pattern_matching" title="Pattern matching">pattern matching</a>, <a href="/wiki/List_comprehension" title="List comprehension">list comprehension</a>, <a href="/wiki/Type_class" title="Type class">type classes</a> and <a href="/wiki/Type_polymorphism" class="mw-redirect" title="Type polymorphism">type polymorphism</a>. It is a <a href="/wiki/Purely_functional_language" class="mw-redirect" title="Purely functional language">purely functional language</a>, which means that functions generally have no <a href="/wiki/Side_effect_(computer_science)" title="Side effect (computer science)">side effects</a>. A distinct construct exists to represent side effects, <a href="/wiki/Orthogonal#Computer_science" class="mw-redirect" title="Orthogonal">orthogonal</a> to the type of functions. A pure function can return a side effect that is subsequently executed, modeling the <a href="/wiki/Pure_function#Impure_functions" title="Pure function">impure functions</a> of other languages.</p>/n<p>Haskell has a <a href="/wiki/Strongly_typed_programming_language" class="mw-redirect" title="Strongly typed programming language">strong</a>, <a href="/wiki/Static_type#Static_typing" class="mw-redirect" title="Static type">static</a> type system based on <a href="/wiki/Hindley%E2%80%93Milner_type_inference" class="mw-redirect" title="Hindley–Milner type inference">Hindley–Milner type inference</a>. Its principal innovation in this area is type classes, originally conceived as a principled way to add <a href="/wiki/Polymorphism_(computer_science)" title="Polymorphism (computer science)">overloading</a> to the language,<sup id="cite_ref-wadler89_40-0" class="reference"><a href="#cite_note-wadler89-40">[40]</a></sup> but since finding many more uses.<sup id="cite_ref-hallgren01_41-0" class="reference"><a href="#cite_note-hallgren01-41">[41]</a></sup></p>/n<p>The construct that represents side effects is an example of a <a href="/wiki/Monad_(functional_programming)" title="Monad (functional programming)">monad</a>. Monads are a general framework that can model different kinds of computation, including error handling, <a href="/wiki/Nondeterministic_algorithm" title="Nondeterministic algorithm">nondeterminism</a>, <a href="/wiki/Parsing" title="Parsing">parsing</a> and <a href="/wiki/Software_transactional_memory" title="Software transactional memory">software transactional memory</a>. Monads are defined as ordinary datatypes, but Haskell provides some <a href="/wiki/Syntactic_sugar" title="Syntactic sugar">syntactic sugar</a> for their use.</p>/n<p>Haskell has an open, published specification,<sup id="cite_ref-FOOTNOTEPeyton_Jones2003_28-2" class="reference"><a href="#cite_note-FOOTNOTEPeyton_Jones2003-28">[28]</a></sup> and <a href="#Implementations">multiple implementations exist</a>. Its main implementation, the <a href="/wiki/Glasgow_Haskell_Compiler" title="Glasgow Haskell Compiler">Glasgow Haskell Compiler</a> (GHC), is both an <a href="/wiki/Interpreter_(computing)" title="Interpreter (computing)">interpreter</a> and <a href="/wiki/Machine_code" title="Machine code">native-code</a> <a href="/wiki/Compiler" title="Compiler">compiler</a> that runs on most platforms. GHC is noted for its rich type system incorporating recent innovations such as <a href="/wiki/Generalized_algebraic_data_type" title="Generalized algebraic data type">generalized algebraic data types</a> and type families. <a href="/wiki/The_Computer_Language_Benchmarks_Game" title="The Computer Language Benchmarks Game">The Computer Language Benchmarks Game</a> also highlights its high-performance implementation of <a href="/wiki/Concurrency_(computer_science)" title="Concurrency (computer science)">concurrency</a> and <a href="/wiki/Parallel_computing" title="Parallel computing">parallelism</a>.<sup id="cite_ref-shootout_42-0" class="reference"><a href="#cite_note-shootout-42">[42]</a></sup></p>/n<p>An active, growing community exists around the language, and more than 5,400 third-party open-source libraries and tools are available in the online package repository <i>Hackage</i>.<sup id="cite_ref-hackage-stats_43-0" class="reference"><a you may find correspondences with the algebra world that underlies the standard typeclasses of the Haskell basic library.</p>/n<p>The <i>Num</i> class has the operation signatures required for a <a href="/wiki/Ring_(mathematics)" title="Ring (mathematics)">Ring</a>, except for the (+) and (*) neutral elements, which are predefined as literals.<sup id="cite_ref-79" class="reference"><a instances.</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="o">$</span> <span class="n">ghci</span>/n<span class="kt">Prelude</span><span class="o">&gt;</span> <span <span class="kt">:</span><span class="kr">type</span> <span class="mi">1</span>/n<span class="mi">1</span> <span class="ow">::</span> <span class="kt">Num</span> <span class="n">a</span> <span title="Two's complement">two's complement</a> <a href="/wiki/Arithmetic" title="Arithmetic">arithmetic</a> that does not match the <a href="/wiki/Decimal_arithmetic" class="mw-redirect" title="Decimal <i>Fractional</i> type class adds to Num the <a href="/wiki/Multiplicative_inverse" title="Multiplicative inverse">multiplicative inverse</a> in the <i>recip</i> function (for "reciprocal") and, as a
