programming language</h1>/n									<div id="bodyContent" class="mw-body-content">/n									<div id="siteSub" class="noprint">From Wikipedia, the free encyclopedia</div>/n								<div id="contentSub"></div>/n												<div id="jump-to-nav" class="mw-jump">/n					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>/n				</div>/n				<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="mw-parser-output"><p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, a <b>high-level programming language</b> is a <a href="/wiki/Programming_language" title="Programming language">programming language</a> with strong <a href="/wiki/Abstraction_(computer_science)" class="mw-redirect" title="Abstraction (computer science)">abstraction</a> from the details of the <a href="/wiki/Computer" title="Computer">computer</a>. In comparison to <a href="/wiki/Low-level_programming_language" title="Low-level programming language">low-level programming languages</a>, it may use <a href="/wiki/Natural_language" title="Natural language">natural language</a> elements, be easier to use, or may automate (or even hide entirely) significant areas of computing systems (e.g. <a href="/wiki/Memory_management" title="Memory management">memory management</a>), making the process of developing a program simpler and more understandable relative to a lower-level language. The amount of abstraction provided defines how "high-level" a programming language is.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup></p>/n<p>In the 1960s, high-level programming languages using a <a href="/wiki/Compiler" title="Compiler">compiler</a> were commonly called <b><a href="/wiki/Autocode" title="Autocode">autocodes</a></b>.<sup id="cite_ref-kleith_2-0" class="reference"><a href="#cite_note-kleith-2">[2]</a></sup> Examples of autocodes are <a href="/wiki/COBOL" title="COBOL">COBOL</a> and <a href="/wiki/Fortran" title="Fortran">Fortran</a>.<sup id="cite_ref-kleith2_3-0" class="reference"><a href="#cite_note-kleith2-3">[3]</a></sup></p>/n<p>The first high-level programming language designed for computers was <a href="/wiki/Plankalk%C3%BCl" title="Plankalkül">Plankalkül</a>, created by <a href="/wiki/Konrad_Zuse" title="Konrad Zuse">Konrad Zuse</a>.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> However, it was not implemented in his time, and his original contributions were (due to <a href="/wiki/World_War_II" title="World War II">World War II</a>) largely isolated from other developments, although it influenced <a href="/wiki/Heinz_Rutishauser" title="Heinz Rutishauser">Heinz Rutishauser</a>'s language "Superplan" (and to some degree also Algol). The first really widespread high-level language was <a href="/wiki/Fortran" title="Fortran">Fortran</a>, a machine independent development of IBM's earlier <a href="/wiki/Autocode" title="Autocode">Autocode</a> systems. <a href="/wiki/ALGOL" title="ALGOL">Algol</a>, defined in 1958 and 1960, by committees of European and American computer scientists, introduced <a href="/wiki/Recursion" title="Recursion">recursion</a> as well as <a href="/wiki/Nested_functions" class="mw-redirect" title="Nested functions">nested functions</a> under <a href="/wiki/Lexical_scope" class="mw-redirect" title="Lexical scope">lexical scope</a>. It was also the first language with a clear distinction between <a href="/wiki/Call_by_value" class="mw-redirect" title="Call by value">value</a> and <a href="/wiki/Call_by_name" class="mw-redirect" title="Call by name">name-parameters</a> and their corresponding <a href="/wiki/Semantics" title="Semantics">semantics</a>.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> Algol also introduced several <a href="/wiki/Structured_programming" title="Structured programming">structured programming</a> concepts, such as the <b>while-do</b> and <b>if-then-else</b> constructs and its <a href="/wiki/Syntax" title="Syntax">syntax</a> was the first to be described by a formal method, <a href="/wiki/Backus%E2%80%93Naur_form" title="Backus–Naur form">Backus–Naur form</a> (BNF). During roughly the same period <a href="/wiki/Cobol" class="mw-redirect" title="Cobol">Cobol</a> introduced <a href="/wiki/Record_(computer_science)" title="Record (computer science)">records</a> (also called structs) and <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">Lisp</a> introduced a fully general <a href="/wiki/Lambda_abstraction" class="mw-redirect" title="Lambda abstraction">lambda abstraction</a> in a programming language for the first time.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toctitle">/n<h2>Contents</h2>/n</div>/n<ul>/n<li class="toclevel-1 tocsection-8"><a href="#External_links"><span class="tocnumber">7</span> <span class="toctext">External links</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="Features">Features</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=High-level_programming_language&amp;action=edit&amp;section=1" title="Edit section: Features">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>"High-level language" refers to the higher level of abstraction from <a href="/wiki/Machine_language" class="mw-redirect" title="Machine language">machine language</a>. Rather than dealing with registers, memory addresses and call stacks, high-level languages deal with variables, arrays, <a href="/wiki/Object_(computer_science)" title="Object (computer science)">objects</a>, complex arithmetic or boolean expressions, subroutines and functions, loops, <a href="/wiki/Thread_(computer_science)" class="mw-redirect" title="Thread (computer science)">threads</a>, locks, and other abstract computer science concepts, with a focus on <a href="/wiki/Usability" title="Usability">usability</a> over optimal program efficiency. Unlike low-level <a href="/wiki/Assembly_language" title="Assembly language">assembly languages</a>, high-level languages have few, if any, language elements that translate directly into a machine's native <a href="/wiki/Opcode" title="Opcode">opcodes</a>. Other features, such as string handling routines, object-oriented language features, and file input/output, may also be present.</p>/n<h2><span class="mw-headline" id="Abstraction_penalty">Abstraction penalty</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=High-level_programming_language&amp;action=edit&amp;section=2" title="Edit section: Abstraction penalty">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>High-level languages intend to provide features which standardize common tasks, permit rich debugging, and maintain architectural agnosticism; while low-level languages often produce more efficient code through optimization for a specific system architecture. <i>Abstraction penalty</i> is the border that prevents high-level programming techniques from being applied in situations where computational limitations, standards conformance or physical constraints require access to low-level architectural resources (fi, response time(s), hardware integration). High-level programming exhibits features like more generic data structures/operations, run-time interpretation, and intermediate code files; which often result in execution of far more operations than necessary, higher memory consumption, and larger binary program size.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup><sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup><sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> For this reason, code which needs to run particularly quickly and efficiently may require the use of a lower-level language, even if a higher-level language would make the coding easier. In many cases, critical portions of a program mostly in a high-level language can be hand-coded in <a href="/wiki/Assembly_language" title="Assembly language">assembly language</a>, leading to a much faster, more efficient, or simply reliably functioning <a href="/wiki/Program_optimisation" class="mw-redirect" title="Program optimisation">optimised program</a>.</p>/n<p>However, with the growing complexity of modern <a href="/wiki/Microprocessor" title="Microprocessor">microprocessor</a> architectures, well-designed compilers for high-level languages frequently produce code comparable in efficiency to what most low-level programmers can produce by hand, and the higher abstraction may allow for more powerful techniques providing better overall results than their low-level counterparts in particular settings.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> High-level languages are designed independent of a specific computing system architecture. This facilitates executing a program written in such a language on any computing system with compatible support for the Interpreted or <a href="/wiki/Just-in-time_compilation" title="Just-in-time compilation">JIT</a> program. High-level languages can be improved as their designers develop improvements. In other cases, new high-level languages evolve from one or more others with the goal of aggregating the most popular constructs with new or improved features. An example of this is <a href="/wiki/Scala_(programming_language)" title="Scala (programming language)">Scala</a> which maintains backward compatibility with <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> which means that programs and libraries written in Java will continue to be usable even if a programming shop switches to Scala; this makes the transition easier and the lifespan of such high-level coding indefinite. In contrast, low-level programs rarely survive beyond the system architecture which they were written for without major revision. This is the engineering 'trade-off' for the 'Abstraction Penalty'.</p>/n<h2><span class="mw-headline" id="Relative_meaning">Relative meaning</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=High-level_programming_language&amp;action=edit&amp;section=3" title="Edit section: Relative meaning">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Examples of high-level programming languages in active use today include <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, <a href="/wiki/Visual_Basic" title="Visual Basic">Visual Basic</a>, <a href="/wiki/Delphi_(programming_language)" title="Delphi (programming language)">Delphi</a>, <a href="/wiki/Perl" title="Perl">Perl</a>, <a href="/wiki/PHP" title="PHP">PHP</a>, <a href="/wiki/ECMAScript" title="ECMAScript">ECMAScript</a>, <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a> and many others.</p>/n<p>The terms <i>high-level</i> and <i>low-level</i> are inherently relative. Some decades ago, the <a href="/wiki/C_(programming_language)" title="C (programming language)">C language</a>, and similar languages, were most often considered "high-level", as it supported concepts such as expression evaluation, <a href="/wiki/Parameter" title="Parameter">parameterised</a> recursive functions, and data types and structures, while <a href="/wiki/Assembly_language" title="Assembly language">assembly language</a> was considered "low-level". Today, many programmers might refer to C as low-level, as it lacks a large <a href="/wiki/Run_time_system" class="mw-redirect" title="Run time system">runtime</a>-system (no garbage collection, etc.), basically supports only scalar operations, and provides direct memory addressing. It, therefore, readily blends with assembly language and the machine level of <a href="/wiki/CPU" class="mw-redirect" title="CPU">CPUs</a> and <a href="/wiki/Microcontroller" title="Microcontroller">microcontrollers</a>.</p>/n<p>Assembly language may itself be regarded as a higher level (but often still one-to-one if used without <a href="/wiki/Macro_(computer_science)" title="Macro (computer science)">macros</a>) representation of <a href="/wiki/Machine_code" title="Machine code">machine code</a>, as it supports concepts such as constants and (limited) expressions, sometimes even variables, procedures, and <a href="/wiki/Data_structure" title="Data structure">data structures</a>. <a href="/wiki/Machine_code" title="Machine code">Machine code</a>, in its turn, is inherently at a slightly higher level than the <a href="/wiki/Microcode" title="Microcode">microcode</a> or <a href="/wiki/Micro-operation" title="Micro-operation">micro-operations</a> used internally in many processors.</p>/n<h2><span class="mw-headline" id="Execution_modes">Execution modes</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=High-level_programming_language&amp;action=edit&amp;section=4" title="Edit section: Execution modes">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>There are three general modes of execution for modern high-level languages:</p>/n<dl>/n<dt>Interpreted&#160;</dt>/n<dd>When code written in a language is <a href="/wiki/Interpreted_language" title="Interpreted language">interpreted</a>, its syntax is read and then executed directly, with no compilation stage. A program called an <i>interpreter</i> reads each program statement, following the program flow, then decides what to do, and does it. A hybrid of an interpreter and a compiler will compile the statement into machine code and execute that; the machine code is then discarded, to be interpreted anew if the line is executed again. Interpreters are commonly the simplest implementations of the behavior of a language, compared to the other two variants listed here.</dd>/n<dt>Compiled&#160;</dt>/n<dd>When code written in a language is <a href="/wiki/Compiled_language" title="Compiled language">compiled</a>, its syntax is transformed into an executable form before running. There are two types of compilation:/n<dl>/n<dt>Machine code generation&#160;</dt>/n<dd>Some compilers compile source code directly into <a href="/wiki/Machine_code" title="Machine code">machine code</a>. This is the original mode of compilation, and languages that are directly and completely transformed to machine-native code in this way may be called "truly compiled" languages. See <a href="/wiki/Assembly_language" title="Assembly language">assembly language</a>.</dd>/n<dt>Intermediate representations&#160;</dt>/n<dd>When code written in a language is compiled to an intermediate representation, that representation can be optimized or saved for later execution without the need to re-read the source file. When the intermediate representation is saved, it may be in a form such as <a href="/wiki/Byte_code" class="mw-redirect" title="Byte code">byte code</a>. The intermediate representation must then be interpreted or further compiled to execute it. <a href="/wiki/Virtual_machine" title="Virtual machine">Virtual machines</a> that execute byte code directly or transform it further into machine code have blurred the once clear distinction between intermediate representations and truly compiled languages.</dd>/n</dl>/n</dd>/n</dl>/n<dl>/n<dt>Source-to-Source Translated or Trans-compiled</dt>/n<dd>Code written in a language may be translated into terms of a lower-level programming language for which native code compilers are already widely available. <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a> and the <a href="/wiki/C_(programming_language)" title="C (programming language)">C programming language</a> are common targets for such translators. See <a href="/wiki/CoffeeScript" title="CoffeeScript">CoffeeScript</a>, <a href="/wiki/Chicken_Scheme" class="mw-redirect" title="Chicken Scheme">Chicken Scheme</a>, and <a href="/wiki/Eiffel_(programming_language)" title="Eiffel (programming language)">Eiffel</a> as examples. Specifically, the generated C and C++ code can be seen (as generated from the Eiffel programming language when using the <a href="/wiki/EiffelStudio" title="EiffelStudio">EiffelStudio</a> IDE) in the EIFGENs directory of any compiled Eiffel project. In Eiffel, the "Translated" process is referred to as Trans-compiling or Trans-compiled, and the Eiffel compiler as a <a href="/wiki/Source-to-source_compiler" title="Source-to-source compiler">Transcompiler</a>.</dd>/n</dl>/n<p>Note that languages are not strictly "interpreted" languages or "compiled" languages. Rather, implementations of language behavior use interpretation or compilation. For example, <a href="/wiki/Algol_60" class="mw-redirect" title="Algol 60">Algol 60</a> and <a href="/wiki/Fortran" title="Fortran">Fortran</a> have both been interpreted (even though they were more typically compiled). Similarly, Java shows the difficulty of trying to apply these labels to languages, rather than to implementations; Java is compiled to bytecode and the bytecode is subsequently executed by either interpretation (in a <a href="/wiki/Java_Virtual_Machine" class="mw-redirect" title="Java Virtual Machine">JVM</a>) or compilation (typically with a just-in-time compiler such as <a href="/wiki/HotSpot" title="HotSpot">HotSpot</a>, again in a JVM). Moreover, compilation, trans-compiling, and interpretation are not strictly limited to just a description of the compiler artifact (binary executable or IL assembly).</p>/n<h3><span class="mw-headline" id="High-level_language_computer_architecture">High-level language computer architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=High-level_programming_language&amp;action=edit&amp;section=5" title="Edit section: High-level language computer architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Alternatively, it is possible for a high-level language to be directly implemented by a computer – the computer directly executes the HLL code. This is known as a <i><a href="/wiki/High-level_language_computer_architecture" title="High-level language computer architecture">high-level language computer architecture</a></i> – the <a href="/wiki/Computer_architecture" title="Computer architecture">computer architecture</a> itself is designed to be targeted by a specific high-level language.</p>/n<h2><span class="mw-headline"
