href="/wiki/Help:Maintenance_template_removal" title="Help:Maintenance template removal">Learn how and when to remove this template message</a>)</i></small></div>/n</td>/n</tr>/n</table>/n<p>In <a href="/wiki/Computer_programming" title="Computer programming">computer programming</a>, <b>undefined behavior</b> (<b>UB</b>) is the result of executing computer code whose <a href="/wiki/Behavior" title="Behavior">behavior</a> is not prescribed by the <a href="/wiki/Language_specification" class="mw-redirect" title="Language specification">language specification</a> to which the code adheres, for the current state of the program (e.g. memory). This happens when the <a href="/wiki/Translator_(computing)" class="mw-redirect" title="Translator (computing)">translator</a> of the source code makes certain assumptions, but these assumptions are not satisfied during execution.</p>/n<p>The behavior of some programming languages—most famously <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> and <a href="/wiki/C%2B%2B" title="C++">C++</a>—is undefined in some cases.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> In the standards for these languages, the semantics of certain operations is described as <i>undefined</i>. An implementation is allowed to assume that such operations never occur in standard-conforming program code; the implementation will be considered correct whatever it does in such cases, analogous to <a href="/wiki/Don%27t-care_term" title="Don't-care term">don't-care terms</a> in digital logic. This assumption can make various <a href="/wiki/Program_transformations" class="mw-redirect" title="Program transformations">program transformations</a> valid or simplify their proof of correctness, giving flexibility to the implementation. As a result, the <a href="/wiki/Compiler" title="Compiler">compiler</a> can often make more optimizations. It is the responsibility of the programmer to write code that never invokes undefined behavior, although compiler implementations are allowed to issue diagnostics when this happens.</p>/n<p>Undefined behavior is often unpredictable and a frequent cause of <a href="/wiki/Software_bugs" class="mw-redirect" title="Software bugs">software bugs</a>. In the C community, undefined behavior may be humorously referred to as "<b>nasal demons</b>", after a <a href="/wiki/Usenet" title="Usenet">comp.std.c</a> post that explained undefined behavior as allowing the compiler to do anything it chooses, even "to make demons fly out of your nose".<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> Under some circumstances there can be specific restrictions on undefined behavior. For example, the <a href="/wiki/Instruction_set" class="mw-redirect" title="Instruction set">instruction set</a> specifications of a <a href="/wiki/Central_processing_unit" title="Central processing unit">CPU</a> might leave the behavior of some forms of an instruction undefined, but if the CPU supports <a href="/wiki/Memory_protection" title="Memory protection">memory protection</a> then the specification will probably include a blanket rule stating that no user-accessible instruction may cause a hole in the <a href="/wiki/Operating_system" title="Operating system">operating system</a>'s security; so an actual CPU would be permitted to corrupt user registers in response to such an instruction, but would not be allowed to, for example, switch into <a href="/wiki/Supervisor_mode" class="mw-redirect" title="Supervisor mode">supervisor mode</a>.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toctitle">/n<h2>Contents</h2>/n</div>/n<ul>/n<li class="toclevel-1 tocsection-1"><a href="#Benefits"><span class="tocnumber">1</span> <span class="toctext">Benefits</span></a></li>/n<li class="toclevel-1 tocsection-2"><a href="#Risks"><span class="tocnumber">2</span> <span class="toctext">Risks</span></a></li>/n<li class="toclevel-1 tocsection-3"><a href="#Examples_in_C_and_C.2B.2B"><span class="tocnumber">3</span> <span class="toctext">Examples in C and C++</span></a></li>/n<li class="toclevel-1 tocsection-4"><a href="#See_also"><span class="tocnumber">4</span> <span class="toctext">See also</span></a></li>/n<li class="toclevel-1 tocsection-5"><a href="#References"><span class="tocnumber">5</span> <span class="toctext">References</span></a></li>/n<li class="toclevel-1 tocsection-6"><a href="#Further_reading"><span class="tocnumber">6</span> <span class="toctext">Further reading</span></a></li>/n<li class="toclevel-1 tocsection-7"><a href="#External_links"><span class="tocnumber">7</span> <span class="toctext">External links</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="Benefits">Benefits</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Undefined_behavior&amp;action=edit&amp;section=1" title="Edit section: Benefits">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Documenting an operation as undefined behavior allows compilers to assume that this operation will never happen in a conforming program. This gives the compiler more information about the code and this information can lead to more optimization opportunities.</p>/n<p>An example for the C language:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="kt">int</span> <span class="nf">foo</span><span class="p">(</span><span class="kt">unsigned</span> <span class="kt">char</span> <span class="n">x</span><span class="p">)</span>/n<span class="p">{</span>/n     <span class="kt">int</span> <span class="n">value</span> <span class="o">=</span> <span class="mi">2147483600</span><span class="p">;</span> <span class="cm">/* assuming 32 bit int */</span>/n     <span class="n">value</span> <span class="o">+=</span> <span class="n">x</span><span class="p">;</span>/n     <span class="k">if</span> <span class="p">(</span><span class="n">value</span> <span class="o">&lt;</span> <span class="mi">2147483600</span><span class="p">)</span>/n        <span class="n">bar</span><span class="p">();</span>/n     <span class="k">return</span> <span class="n">value</span><span class="p">;</span>/n<span class="p">}</span>/n</pre></div>/n<p>The value of <code>x</code> cannot be negative and, given that signed <a href="/wiki/Integer_overflow" title="Integer overflow">integer overflow</a> is undefined behavior in C, the compiler can assume that at the line of the if check <code>value &gt;= 2147483600</code>. Thus the <code>if</code> and the call to the function <code>bar</code> can be ignored by the compiler since the <code>if</code> has no side effects and its condition will never be satisfied. The code above is therefore semantically equivalent to:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="kt">int</span> <span class="nf">foo</span><span class="p">(</span><span class="kt">unsigned</span> <span class="kt">char</span> <span class="n">x</span><span class="p">)</span>/n<span class="p">{</span>/n     <span class="kt">int</span> <span class="n">value</span> <span class="o">=</span> <span class="mi">2147483600</span><span class="p">;</span>/n     <span class="n">value</span> <span class="o">+=</span> <span class="n">x</span><span class="p">;</span>/n     <span class="k">return</span> <span class="n">value</span><span class="p">;</span>/n<span class="p">}</span>/n</pre></div>/n<p>Had the compiler been forced to assume that signed integer overflow has <i>wraparound</i> behavior, then the transformation above would not have been legal.</p>/n<p>Such optimizations become hard to spot by humans when the code is more complex and other optimizations, like <a href="/wiki/Inlining" class="mw-redirect" title="Inlining">inlining</a>, take place.</p>/n<p>Another benefit from allowing signed integer overflow to be undefined is that it makes it possible to store and manipulate a variable's value in a <a href="/wiki/Processor_register" title="Processor register">processor register</a> that is larger than the size of the variable in the source code. For example, if the type of a variable as specified in the source code is narrower than the native register width (such as "<a href="/wiki/C_data_types#Basic_types" title="C data types">int</a>" on a <a href="/wiki/64-bit" class="mw-redirect" title="64-bit">64-bit</a> machine, a common scenario), then the compiler can safely use a signed 64-bit integer for the variable in the <a href="/wiki/Machine_code" title="Machine code">machine code</a> it produces, without changing the defined behavior of the code. If the behavior of a 32-bit integer under overflow conditions was depended upon by the program, then a compiler would have to insert additional logic when compiling for a 64-bit machine, because the overflow behavior of most machine code instructions depends on the register width.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup></p>/n<h2><span class="mw-headline" id="Risks">Risks</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Undefined_behavior&amp;action=edit&amp;section=2" title="Edit section: Risks">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>C and C++ standards are littered with undefined behavior throughout, which offers increased liberty in compiler implementations at the expense of limiting what users of the language can do. In particular, there is an entire appendix section dedicated to a non-exhaustive listing of common sources of undefined behavior in C.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> Moreoever, compilers are not required to diagnose code that relies on undefined behavior. Hence, it is common for programmers, even experienced ones, to unintentionally rely on undefined behavior either by mistake, or simply because they are not well-versed in the rules of the language that can span over hundreds of pages. This can result in bugs that occur when optimizations are enabled on the compiler, or when a compiler of a different vendor or version is used.</p>/n<p>In scenarios where security is critical, undefined behavior can lead to <a href="/wiki/Computer_security" title="Computer security">security</a> vulnerabilities in software. When <a href="/wiki/GNU_C_Compiler" class="mw-redirect" title="GNU C Compiler">GCC</a>'s developers changed their compiler in 2008 such that it omitted certain overflow checks that relied on undefined behavior, <a href="/wiki/CERT_Coordination_Center" title="CERT Coordination Center">CERT</a> issued a warning against the newer versions of the compiler.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> <a href="/wiki/Linux_Weekly_News" class="mw-redirect" title="Linux Weekly News">Linux Weekly News</a> pointed out that the same behavior was observed in <a href="/wiki/PathScale" title="PathScale">PathScale C</a>, <a href="/wiki/Visual_C%2B%2B" class="mw-redirect" title="Visual C++">Microsoft Visual C++ 2005</a> and several other compilers;<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> the warning was later amended to warn about various compilers.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup></p>/n<p><a href="/wiki/HTML" title="HTML">HTML</a> versions 4 and earlier left error handling undefined. Over time pages started relying on unspecified error-recovery implemented in popular browsers. This caused difficulties for vendors of less-popular browsers who were forced to reverse-engineer and implement <a href="/wiki/Bug_compatibility" title="Bug compatibility">bug compatible</a> error recovery. This has led to <a href="/wiki/De_facto" title="De facto">de facto</a> standard (i.e. <a href="/wiki/HTML5" title="HTML5">HTML5</a>) that was much more complicated than it could have been if this behavior was specified from the start.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (June 2014)">citation needed</span></a></i>]</sup></p>/n<h2><span id="Examples_in_C_and_C++"></span><span class="mw-headline" id="Examples_in_C_and_C.2B.2B">Examples in C and C++</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Undefined_behavior&amp;action=edit&amp;section=3" title="Edit section: Examples in C and C++">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>In C the use of any <a href="/wiki/Automatic_variable" title="Automatic variable">automatic variable</a> before it has been initialized yields undefined behavior, as does integer <a href="/wiki/Division_by_zero" title="Division by zero">division by zero</a> or indexing an array outside of its defined bounds (see <a href="/wiki/Buffer_overflow" title="Buffer overflow">buffer overflow</a>). In general, any instance of undefined behavior leaves the abstract execution machine in an unknown state, and any subsequent behavior is also undefined. If it is not required that the compiler diagnose undefined behavior, programs invoking undefined behavior may compile and run producing correct results, incorrect results, or have any other behavior. Because of this, undefined behavior can create errors that are difficult to detect.</p>/n<p>Attempting to modify a <a href="/wiki/String_literal" title="String literal">string literal</a> causes undefined behavior:<sup id="cite_ref-C.2B.2B03_2.13.4.2F2_8-0" class="reference"><a href="#cite_note-C.2B.2B03_2.13.4.2F2-8">[8]</a></sup></p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="kt">char</span> <span class="o">*</span><span class="n">p</span> <span class="o">=</span> <span class="s">"wikipedia"</span><span class="p">;</span> <span class="c1">// valid C, ill-formed C++11, deprecated C++98/C++03</span>/n<span class="n">p</span><span class="p">[</span><span class="mi">0</span><span class="p">]</span> <span class="o">=</span> <span class="sc">'W'</span><span class="p">;</span> <span class="c1">// undefined behavior</span>/n</pre></div>/n<p>Integer <a href="/wiki/Division_by_zero" title="Division by zero">division by zero</a> results in undefined behavior:<sup id="cite_ref-C.2B.2B03_5.6.2F4_9-0" class="reference"><a href="#cite_note-C.2B.2B03_5.6.2F4-9">[9]</a></sup></p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="kt">int</span> <span class="n">x</span> <span class="o">=</span> <span class="mi">1</span><span class="p">;</span>/n<span class="k">return</span> <span class="n">x</span> <span class="o">/</span> <span class="mi">0</span><span class="p">;</span> <span class="c1">// undefined behavior</span>/n</pre></div>/n<p>Certain pointer operations may result in undefined behavior:<sup id="cite_ref-C.2B.2B03_5.6.2F5_10-0" class="reference"><a href="#cite_note-C.2B.2B03_5.6.2F5-10">[10]</a></sup></p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="kt">int</span> <span class="n">arr</span><span class="p">[</span><span class="mi">4</span><span class="p">]</span> <span class="o">=</span> <span class="mi">5</span><span class="p">;</span>  <span class="c1">// undefined behavior</span>/n</pre></div>/n<p>Reaching the end of a value-returning function (other than main()) without a return statement results in undefined behavior if the value of the function call is used by the caller:<sup id="cite_ref-C99_6.9.1.2F12_11-0" class="reference"><a href="#cite_note-C99_6.9.1.2F12-11">[11]</a></sup></p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="kt">int</span> <span class="nf">f</span><span class="p">()</span>/n<span class="p">{</span>/n<span class="p">}</span>  <span class="cm">/* undefined behavior if the value of the function call is used*/</span>/n</pre></div>/n<p>Modifying an object
